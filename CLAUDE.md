# Agent Instructions

## Language

- 回答は原則として日本語で行う。
- コード、ファイル名、識別子は英語を優先する。
- Markdown 設計書は自然な日本語で、曖昧な表現を避ける。

## Project Policy

- 変更前に関連する設計書を確認する。
- `docs/` 配下の Markdown では、frontmatter の `id`・`based_on`・`status` を尊重する。
- 既存の命名規則・ディレクトリ規則を優先する。
- 大きな変更は、まず設計書に反映してからコードを変更する。

## SpecDojo ワークフロー

実装を開始する前に以下を実施すること:

1. このプロンプトで提供される exec plan を読む。
2. plan からタスクの owner ロールを特定し、そのロール視点で作業する。
3. claim したタスクのみ実行する。
4. claim したタスクが明示的に要求しない限り、関係のない成果物を編集しない。
5. plan の「実施手順」セクションに記載された result ファイルのパスを確認し、そのファイルを記入する。
6. 完了後に `specdojo exec validate` を実行する。成功なら終了コード 0、ブロックなら 1 で終了する。
7. 終了コード 1 で終了する場合は、ブロック理由を stderr に書き出す。

## 安全規則

- `.env`、`.env.*`、`secrets/`、認証情報、秘密鍵は読み込まない。
- 破壊的変更を行う前に git diff を確認する。
- 実際に実装していないタスクを complete にしない。
- 明示的に指示されない限り、1プロセスで複数タスクを claim しない。
- Git に予期しない変更がある場合は中断して報告する。
