# プロジェクト管理計画

本書は、SpecDojo プロジェクトの計画化、進捗確認、課題・リスク管理、変更管理の運用方針を、PM が実行管理に使える粒度で定義する。

## 1. 概要

本プロジェクトは、人と AI Agent の協働を前提とした個人・小規模運用から開始する。組織構成の最終判断は人間の `PO` に集約し、PM は計画、進捗、課題、リスク、変更要求の管理プロセスを起案・運用する。

採用する Role code の語彙は [[prj-0001:pm-organization|組織定義]] と [[prj-0001:pm-roles|ロール定義]] を正とする。小規模運用では `PM` の責務が専任 member に固定されない場合があるため、実行主体の割り当ては [[prj-0001:pm-members|メンバー定義]] で管理する。

AI Agent は文書作成、差分整理、整合確認、構成確認、機械的更新、改善提案を支援してよい。ただし、最終承認、公開可否、スコープ変更の採否、説明責任は人間の `PO` が担う。

## 2. 管理方針（スコープ / スケジュール / コスト）

### 2.1. 基本方針

| 管理領域     | 管理単位                                     | PM の運用方針                                                                    | 最終判断          | 逸脱時対応                                                                       |
| ------------ | -------------------------------------------- | -------------------------------------------------------------------------------- | ----------------- | -------------------------------------------------------------------------------- |
| スコープ     | 成果物カタログ、変更要求、未決事項           | 成果物の追加・削除・統合、対象外事項、未決論点を作業単位へ分解し、影響を整理する | `PO`              | 目的、公開範囲、主要成果物へ影響する場合は変更要求として扱う                     |
| スケジュール | Schedule、マイルストーン、依存関係、実行結果 | 成果物の依存、順序、所要時間見積もり、期限超過を確認し、再計画案を作る           | `PM`、必要時 `PO` | 重要タスクの遅延、依存未解決、マイルストーン影響を課題またはリスクとして記録する |
| コスト       | 作業負荷、外部サービス利用、公開・運用負荷   | 金額予算ではなく、個人・小規模運用で継続可能な負荷かを管理する                   | `PO`              | 有償化、外部委託、継続費用、運用負荷増加が発生した場合は判断事項として起票する   |
| 品質         | rulebook、既存構成、lint、レビュー結果       | 成果物が後続作業に使える粒度か、構造と参照が矛盾しないかを確認する               | `QE`、必要時 `PO` | 重大な不整合、公開不適切情報、再利用困難な粒度は是正対象として扱う               |
| 変更         | 変更要求、決定記録、影響範囲                 | 変更内容、理由、影響先、採否案を分離して整理する                                 | `PO`              | 変更要求ログまたは決定記録へ転記し、必要な成果物と Schedule を更新する           |

### 2.2. スコープ管理

- PM は、成果物の追加、削除、統合、分割、優先順位変更を、作業単位と依存関係へ分解して管理する。
- 目的、公開範囲、主要成果物、採用 Role code に影響する変更は、軽微な文書修正ではなく変更要求として扱う。
- 成果物本文に判断しきれない事項を埋め込まず、未決事項、課題、リスク、変更要求として分離する。
- 下流成果物へ渡すべき入力は、対象、決定事項、未決事項、前提、依存、制約を区別して残す。

### 2.3. スケジュール管理

- PM は、成果物の依存関係、作成順序、所要時間見積もり、進捗状態、期限超過を Schedule と実行結果で管理する。
- Schedule の `owner` には [[prj-0001:pm-roles|ロール定義]] の Role code を使用し、member nickname、agent 名、個人名を置かない。
- 実行主体の割り当ては [[prj-0001:pm-members|メンバー定義]] で管理し、Schedule の `owner` と混同しない。
- 依存未解決、入力不足、lint/test 未解消、レビュー待ちが後続タスクへ影響する場合は、課題またはリスクとして切り出す。
- 再計画時は、作業順序、ブロック理由、影響を受ける成果物、判断が必要な Role code を明示する。

### 2.4. コスト管理

本プロジェクトでは、初期段階の金額予算を設定しない。PM は、個人・小規模運用で継続可能かを判断するため、次の負荷をコスト相当の制約として管理する。

| 観点       | 管理内容                                         | 判断条件                                            | 記録先                     |
| ---------- | ------------------------------------------------ | --------------------------------------------------- | -------------------------- |
| 作業負荷   | 文書作成、レビュー、再生成、整合確認に必要な時間 | 継続更新が困難な粒度、量、手順になった              | 課題、リスク               |
| 外部依存   | 生成 AI、GitHub、公開基盤、補助ツールへの依存    | 有償化、利用制限、代替困難性が発生した              | リスク、変更要求、決定記録 |
| 運用負荷   | 公開後の問い合わせ、更新頻度、障害時対応         | 個人運用で対応困難な量または頻度になった            | 課題、変更要求             |
| 再作業負荷 | 章構成、命名、参照関係、生成物の作り直し         | rulebook や正本の変更で広範囲に再編集が必要になった | 課題、Schedule             |

## 3. 品質・リスク・課題管理方針

### 3.1. 品質管理

PM は品質管理の詳細を個別計画または `QE` のレビューに委ねつつ、本計画では計画運用上の最低基準を管理する。

| 観点         | 基準                                                            | 確認方法                                             | 主な責任    |
| ------------ | --------------------------------------------------------------- | ---------------------------------------------------- | ----------- |
| 構造         | frontmatter、見出し、必須表、関連リンクが rulebook に従う       | rulebook、既存構成、lint を確認する                  | `PM`, `QE`  |
| 一貫性       | 採用 Role code、`owner`、責任分担、管理台帳への接続が矛盾しない | 組織定義、ロール定義、RACI、Schedule を横断確認する  | `PM`, `ARC` |
| 計画化可能性 | 内容がタスク化、順序付け、所要時間見積もり、進捗確認に使える    | 入力、出力、依存、未決事項が分離されているか確認する | `PM`        |
| 公開適性     | 個人情報、非公開情報、認証情報、秘密情報を含めない              | 公開前レビューと差分確認を行う                       | `PO`, `QE`  |

### 3.2. リスク管理

- PM は、将来発生する可能性があり、スコープ、スケジュール、品質、公開判断、継続運用へ影響する事項をリスクとして扱う。
- リスクには、内容、影響、発生可能性、対応方針、責任 Role code、期限、状態を記録できるようにする。
- 主要なリスクは、プロジェクト登録簿またはリスク登録簿へ転記できる単位で分離する。
- リスクが顕在化して作業を止める状態になった場合は、課題へ切り替えて次アクションと期限を管理する。

| 記録トリガー                         | 例                                    | 初期対応                                       |
| ------------------------------------ | ------------------------------------- | ---------------------------------------------- |
| 依存成果物が未完成または矛盾している | 組織定義と Role code の不整合         | 影響先を特定し、ブロックまたは前提付きで進める |
| 後続タスクの順序や期限へ影響する     | rulebook 未整備により成果物作成が滞る | Schedule 影響を整理し、対応順序を見直す        |
| 公開判断へ影響する                   | 非公開情報混入、ライセンス判断未了    | 公開判断前の確認事項として隔離する             |
| 継続運用へ影響する                   | 生成物更新が手作業に依存しすぎる      | 自動化、手順化、対象削減を検討する             |

### 3.3. 課題管理

- PM は、すでに発生しており、対応しなければ作業が進まない事項を課題として扱う。
- 課題には、状態、優先度、対応 Role code、期限、ブロック対象、次アクションを記録する。
- 課題が変更要求、リスク、決定記録に派生する場合は、同じ本文に混在させず、対応する記録へ分離する。
- 影響が将来発生する可能性にとどまる事項は、課題ではなくリスクとして扱う。

| 課題区分 | 判断基準                                             | PM の初期対応                                |
| -------- | ---------------------------------------------------- | -------------------------------------------- |
| 入力不足 | depends_on 成果物の決定事項が不足している            | 不足箇所、必要な判断、暫定対応を記録する     |
| 整合不備 | Role code、責任分担、章構成、リンクが矛盾している    | 正本を特定し、修正対象を切り分ける           |
| 検証失敗 | lint、schema、build、catalog validate が失敗している | 失敗内容、対象ファイル、再実行条件を記録する |
| 判断待ち | `PO` の採否、公開可否、変更承認が必要                | 判断材料、選択肢、影響範囲を整理する         |

### 3.4. 変更管理

- 目的、スコープ、主要成果物、Role code、責任分担、公開方針、Schedule へ影響する変更は変更要求として扱う。
- PM は変更要求について、変更内容、理由、影響先、代替案、推奨案、採否判断者を整理する。
- 軽微な表記修正、リンク修正、誤字修正は通常更新として扱う。ただし、意味、責務、判断条件が変わる場合は変更要求に含める。
- 採否、判断理由、適用対象、Schedule への反映要否は、変更要求または決定記録へ転記できる単位で残す。

## 4. 体制と意思決定

### 4.1. 採用ロール

採用する Role code の語彙は [[prj-0001:pm-organization|組織定義]] と [[prj-0001:pm-roles|ロール定義]] を正とする。本書では、管理計画に関わる責務の使い分けだけを記述し、具体的な member や兼務割り当ては複製しない。

| Role code | 管理計画での主な責務                                               |
| --------- | ------------------------------------------------------------------ |
| `PO`      | 目的、優先順位、公開方針、変更要求の採否、最終承認、説明責任を担う |
| `PM`      | 計画化、進捗確認、課題・リスク管理、変更要求の影響整理を担う       |
| `BA`      | 利用者価値、要件、受入条件、関係者影響の観点を提供する             |
| `ARC`     | 文書体系、構成、命名、依存関係、生成物との整合を確認する           |
| `DEV`     | 実装・設定・技術的変更作業の実現可能性を確認する                   |
| `QE`      | 品質基準、レビュー、検証、公開適性の観点を確認する                 |
| `UX`      | 利用者導線、説明、文書体験の観点を提供する                         |
| `OPS`     | 公開、配布、運用、変更適用、リリース影響の観点を確認する           |

### 4.2. 意思決定

| 判断対象                            | 起案・整理  | 最終判断                | 相談先                      | 証跡                                 |
| ----------------------------------- | ----------- | ----------------------- | --------------------------- | ------------------------------------ |
| 作業順序、進捗確認、ブロック解消    | `PM`        | `PM`                    | `ARC`, `QE`, 関連 Role code | Schedule、実行結果、課題             |
| 目的、スコープ、優先順位の変更      | `PM`        | `PO`                    | `BA`, `ARC`, `QE`           | 変更要求、決定記録                   |
| Role code、責任分担、実行主体の変更 | `PM`        | `PO`                    | `ARC`, `QE`, `OPS`          | 組織定義、ロール定義、RACI、決定記録 |
| 品質基準、検証方針、公開前確認      | `QE`、`PM`  | `PO`                    | `ARC`, `OPS`                | 品質管理計画、レビュー結果、決定記録 |
| AI Agent の出力採否                 | `PM`        | `PO` または成果物の `A` | 関連 Role code              | 差分、レビュー結果、決定記録         |
| 公開可否、ライセンス、外部提供      | `OPS`、`PM` | `PO`                    | `ARC`, `QE`                 | 公開前チェック、決定記録             |

## 5. レポーティングと見直し

### 5.1. レポーティング

PM は、進捗、課題、リスク、変更要求、決定事項を、後続の管理台帳または報告に転記できる単位で分離する。

| 管理領域 | 指標                                           | 報告タイミング            | 主な報告先           | 閾値超過時対応                                 |
| -------- | ---------------------------------------------- | ------------------------- | -------------------- | ---------------------------------------------- |
| 進捗     | 完了、着手中、ブロック、期限超過、依存未解決   | 作業節目、Schedule 更新時 | `PO`, 関連 Role code | ブロック理由と再計画案を整理する               |
| 課題     | open 件数、高優先度、期限超過、判断待ち        | 課題発生時、作業節目      | `PO`, `PM`           | 対応 Role code、期限、次アクションを明確にする |
| リスク   | 高影響リスク、顕在化兆候、対応期限             | リスク識別時、見直し時    | `PO`, `PM`, `QE`     | 回避、軽減、受容、移管の方針を整理する         |
| 変更要求 | スコープ、責任分担、Schedule、公開方針への影響 | 変更要求起票時            | `PO`, `ARC`, `OPS`   | 影響範囲と採否案を決定記録へ接続する           |
| 品質     | lint、schema、build、レビュー指摘、不整合      | 主要成果物更新時          | `PM`, `QE`, `ARC`    | 正本を修正し、必要な生成物を再生成する         |

### 5.2. 見直し条件

| 更新トリガー                                   | 見直し内容                                                           | 主な責任          |
| ---------------------------------------------- | -------------------------------------------------------------------- | ----------------- |
| 組織定義またはロール定義を変更した             | 判断者、Schedule `owner`、RACI、報告先との整合を確認する             | `PM`, `ARC`       |
| 目的、スコープ、優先順位、公開方針を変更した   | 管理単位、変更要求、見直し条件、GO / Not GO 判断への影響を確認する   | `PM`, `PO`        |
| 成果物カタログまたは Schedule を大きく変更した | 進捗管理単位、依存関係、報告タイミング、再計画手順を確認する         | `PM`              |
| 課題滞留、リスク顕在化、変更要求増加が継続した | エスカレーション先、判断期限、PM 責務の専任化要否を確認する          | `PM`, `PO`        |
| 初回公開または外部参加者の受け入れを開始した   | 公開前確認、問い合わせ、運用負荷、個人情報・機密情報の扱いを確認する | `PO`, `OPS`, `QE` |

### 5.3. GO / Not GO 判断

PM は、本格実行または公開判断の前に、`PO` が GO / Not GO を判断できる材料を整理する。

- 目的、スコープ、優先順位、成功基準が、現行の成果物と矛盾していないこと。
- 採用 Role code、Schedule `owner`、RACI、実行主体の割り当てが整合していること。
- 主要な課題、リスク、変更要求、未決事項が識別され、対応方針または判断待ちとして分離されていること。
- lint、schema、build、catalog validate など必要な検証の結果が確認されていること。
- 公開対象に個人情報、非公開情報、認証情報、秘密情報が含まれていないこと。

## 7. 継続確認事項

| 論点                 | 現状                                                                        | 対応方針                                                                 | 判断者 |
| -------------------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------ | ------ |
| PM 責務の専任化      | 小規模運用では専任化せず、実行主体の割り当ては `pm-members.yaml` で管理する | 進捗遅延、課題滞留、リスク顕在化が継続した場合に専任化または分担を見直す | `PO`   |
| 管理台帳への転記単位 | 課題、リスク、変更要求、決定記録を本文に混在させず分離する                  | PM が初期分類し、必要に応じて該当台帳へ転記する                          | `PM`   |
| 初回公開判断         | _UNDECIDED_: 公開可否、公開範囲、公開前確認の最終判断は未了                 | GO / Not GO 判断時に、公開対象、検証結果、公開不適切情報の有無を確認する | `PO`   |
