# コミュニケーション計画

## 1. 概要

本書は、SpecDojo プロジェクトにおける報告、連絡、会議体、情報共有、エスカレーションの方針を定義する。

本計画は [pm-organization.md](../020-organization/pm-organization.md) の採用ロールを前提とする。現時点でコミュニケーション上の責任主体として扱う Role code は `PO`、`BA`、`ARC`、`QE` に限定する。

本プロジェクトは個人・小規模運用を前提とするため、定例会議や詳細な連絡網を過剰に設けず、プロジェクト登録簿、Pull Request、Issue、決定記録、README 等の証跡を中心にコミュニケーションを管理する。AI Agent は作業支援やレビュー支援を行ってよいが、最終判断や承認は `PO` が行う。

公開リポジトリに配置する文書であるため、個人名、個人連絡先、非公開の組織情報、機密性の高い利害関係は記載しない。

## 2. ロール別コミュニケーション分類

採用ロールと未採用ロールは [pm-organization.md](../020-organization/pm-organization.md) を正とする。

| Role code | コミュニケーション上の位置づけ | 主な関心事項 | 受信情報 | 発信情報 | 頻度 / タイミング |
| --------- | ------------------------------ | ------------ | -------- | -------- | ----------------- |
| `PO` | 最終判断者、管理責務の兼務者 | 目的、スコープ、優先順位、公開方針、変更判断、成果物品質 | 主要判断事項、未決事項、リスク、課題、変更要求、レビュー結果 | 判断結果、優先順位、承認可否、公開方針 | 作業節目ごと / 重要判断時 |
| `BA` | 利用者視点、要件、受入条件の整理役 | 利用しやすさ、導入手順、サンプル、受入条件、外部フィードバック | 目的、対象範囲、利用者向け文書、Issue / PR の内容 | 利用者視点の指摘、受入条件、改善提案 | 利用者向け成果物更新時 / 外部入力発生時 |
| `ARC` | 文書体系、構成、技術制約の整理役 | 文書構成、ID、命名、配置、成果物カタログ、生成物との整合 | 成果物変更、構成変更案、Schedule 変更 | 構成判断、命名・配置方針、技術制約 | 構成・配置変更時 / 成果物追加時 |
| `QE` | 品質基準、レビュー観点、整合性確認役 | ルール適合、文書間矛盾、レビュー結果、公開適性 | レビュー対象成果物、lint 結果、差分、生成物 | レビュー指摘、品質判断、公開前確認結果 | 主要成果物更新時 / 公開前 |

## 3. 会議体計画

現時点では個人・小規模運用であるため、会議体は必要最小限とする。会議を開催しない場合も、判断内容と結果はプロジェクト登録簿、決定記録、Pull Request 等に残す。

| 会議名 | 目的 | 参加ロール | 頻度 | 成果物 / 証跡 |
| ------ | ---- | ---------- | ---- | ------------- |
| 作業計画レビュー | 次に作成・更新する成果物、優先順位、未決事項を確認する | `PO`、必要に応じて `BA`, `ARC`, `QE` | 作業節目ごと | プロジェクト登録簿、Schedule、更新対象文書 |
| 成果物レビュー | 主要成果物の内容、構造、リンク、ルール適合を確認する | `PO`, `QE`、観点に応じて `BA`, `ARC` | 主要成果物更新時 | Pull Request、レビューコメント、修正差分 |
| 構成レビュー | 成果物の追加、配置、ID、命名、派生関係を確認する | `PO`, `ARC`、必要に応じて `QE` | 成果物追加時 / 構成変更時 | 成果物カタログ、Schedule、決定記録 |
| リスク・課題確認 | 高優先度のリスク、課題、変更要求の対応方針を確認する | `PO`、必要に応じて関係ロール | 作業節目ごと / 重大事項発生時 | プロジェクト登録簿、リスク登録簿、課題ログ、変更要求ログ |
| 公開前確認 | 公開範囲、README、LICENSE、個人情報・機密情報の有無を確認する | `PO`, `ARC`, `QE` | 初回公開前 / 主要リリース前 | 公開前チェック結果、決定記録、README、LICENSE |

## 4. 報告計画

### 4.1. 報告種別

| 報告種別 | 受信ロール | 目的 | 期限 / タイミング | 形式 | 責任ロール |
| -------- | ---------- | ---- | ----------------- | ---- | ---------- |
| 進捗報告 | `PO` | 作業状況、完了成果物、次の作業、遅延を確認する | 作業節目ごと | Markdown、プロジェクト登録簿、進捗報告 | `PO` |
| 課題・リスク報告 | `PO` | 対応が必要な課題、リスク、変更要求を確認する | 重大事項発生時 / 作業節目ごと | プロジェクト登録簿、生成ビュー | `PO` |
| 成果物レビュー結果 | `PO`、関係ロール | レビュー指摘と修正結果を確認する | 主要成果物更新時 | Pull Request、レビューコメント、差分 | `QE` |
| 構成変更報告 | `PO`, `ARC`, `QE` | 成果物追加、配置変更、ID / 命名変更の影響を確認する | 構成変更時 | 成果物カタログ、Schedule、差分 | `ARC` |
| 利用者視点レビュー | `PO`, `BA`, `QE` | README、導入手順、サンプル、受入条件の分かりやすさを確認する | 利用者向け成果物更新時 | レビューコメント、改善提案 | `BA` |
| 公開準備報告 | `PO`, `ARC`, `QE` | 公開可否判断に必要な状態を確認する | 初回公開前 / 主要リリース前 | README、LICENSE、決定記録 | `PO` |

### 4.2. 報告に含める内容

進捗報告または作業節目の共有では、必要に応じて次を含める。

- 完了した成果物または変更内容。
- 未完了の成果物、未決事項、次の作業。
- スコープ、スケジュール、品質、公開判断への影響。
- 主要な課題、リスク、変更要求。
- 判断が必要な事項と判断者。

## 5. 連絡チャネルとエスカレーション

### 5.1. 連絡チャネル

| チャネル | 用途 | 使用条件 | 証跡 |
| -------- | ---- | -------- | ---- |
| プロジェクト登録簿 | 未決事項、課題、リスク、変更要求の管理 | 対応方針、優先度、期限を追跡したい場合 | [pjr-index.md](../controls/project-register/pjr-index.md) |
| Pull Request | 成果物変更のレビュー、差分確認、承認 | 複数ファイル変更、公開前レビュー、外部貢献の受け入れ時 | PR、レビューコメント |
| Issue | 外部からの問い合わせ、改善要望、課題受付 | 外部参加受付後、利用者・貢献者からの入力を受ける場合 | Issue、コメント |
| 決定記録 | 重要判断と根拠の保存 | 目的、スコープ、公開方針、ライセンス、GO / Not GO などを判断する場合 | 決定記録、生成ビュー |
| README / ドキュメント | 利用者・貢献者への公開情報提供 | 導入手順、利用方法、貢献方法を知らせる場合 | README、ガイド、サンプル |
| チャット / 作業指示 | AI Agent への作業依頼、草案作成、レビュー依頼 | 一時的な作業指示や対話が必要な場合 | 指示内容、生成結果、採否判断 |

### 5.2. エスカレーション

| 区分 | 一次対応 | エスカレーション条件 | 最終判断 | 証跡 |
| ---- | -------- | -------------------- | -------- | ---- |
| スコープ変更 | `PO` | 主要成果物の追加・削除、目的や対象範囲の変更が必要な場合 | `PO` | プロジェクト登録簿、変更要求ログ、関連文書 |
| スケジュール影響 | `PO` | マイルストーン、依存関係、GO / Not GO 判断に影響する場合 | `PO` | Schedule、進捗報告 |
| 品質問題 | `QE` | ルール不適合、文書間矛盾、公開不適切情報がある場合 | `PO` | レビュー記録、修正差分 |
| 構成・命名問題 | `ARC` | 文書配置、ID、成果物カタログ、派生関係に影響する場合 | `PO` | 成果物カタログ、レビュー記録 |
| 利用者視点の不備 | `BA` | 導入手順、サンプル、受入条件、説明文の見直しが必要な場合 | `PO` | レビュー記録、Issue、Pull Request |
| AI Agent 出力 | `PO` | 出力内容に不整合、根拠不明、過剰補完、責任境界の誤認がある場合 | `PO` | 指示内容、生成結果、レビュー結果 |

## 6. 情報管理と公開方針

### 6.1. 証跡管理

- 重要判断は、プロジェクト登録簿、決定記録、Pull Request、Issue のいずれかに証跡を残す。
- 成果物更新時は、関連文書、根拠文書、派生する生成物への影響を確認する。
- 生成物と正本が矛盾する場合は、正本を更新してから生成物を再作成する。
- RACI 上の `A` は原則として `PO` に集約し、Agent には最終承認を委ねない。

### 6.2. 公開情報の扱い

- 公開リポジトリには、個人名、個人連絡先、非公開の組織情報、機密性の高い利害関係を記載しない。
- AI Agent への入力にも、公開できない情報を含めない。
- 外部向け説明は、README、導入ガイド、サンプル、Issue / PR ルールに集約する。

## 7. 見直し条件

次のいずれかに該当する場合、本書を見直す。

| 更新トリガー | 見直し内容 | 責任ロール |
| ------------ | ---------- | ---------- |
| 採用ロールまたは未採用ロールを変更した | ロール別分類、報告責任、相談先、最終判断者を更新する | `PO` |
| RACI を変更した | 会議体、報告計画、エスカレーション条件を更新する | `PO`, `QE` |
| 複数人での継続運用を開始した | `PM` の独立採用、会議体、報告頻度、責任分担を見直す | `PO` |
| 外部からの Issue / Pull Request 受付を開始した | 外部向けチャネル、応答方針、レビュー導線を追加する | `BA`, `QE` |
| 初回公開または主要リリースを行う | 公開前確認、Release notes、利用者向け案内を更新する | `PO`, `ARC`, `QE` |
| コミュニケーション不備による手戻りが発生した | 報告頻度、証跡、エスカレーション条件を見直す | `PO` |
