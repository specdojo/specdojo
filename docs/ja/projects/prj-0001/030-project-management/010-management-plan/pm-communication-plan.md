# コミュニケーション計画

本書は、SpecDojo プロジェクトにおける進捗、課題、リスク、変更要求、判断事項の報告・連絡・会議体・エスカレーションを、PM が計画化と進捗確認に使える粒度で定義する。

## 1. 概要

本計画は、[[prj-0001:pm-plan|プロジェクト管理計画]] が定める管理方針を前提に、報告経路、情報要求、会議体、証跡、エスカレーション条件を具体化する。PM は進捗、課題、リスク、変更要求、決定事項を管理・報告に接続できる単位へ分離し、PO が判断できる材料を整える。

小規模運用では PM 責務を専任 member に固定しない場合があるため、本文では責任主体を Role code で記述する。実行主体の割り当ては別文書を正本とし、本書には個人名、agent 名、個人連絡先、非公開情報を記載しない。

AI Agent は草案作成、差分整理、整合確認、レビュー支援を行ってよい。ただし、最終承認、公開可否、変更要求の採否、GO / Not GO 判断は人間の判断に残す。

## 2. ロール別コミュニケーション分類

本書では、コミュニケーション上の情報要求を、[[prj-0001:pm-plan|プロジェクト管理計画]] に示された管理責務に対応する Role code 単位で整理する。

| Role code | コミュニケーション上の位置づけ             | 主な関心事項                                                  | 受信情報                                                             | 発信情報                                             | 頻度 / タイミング                      |
| --------- | ------------------------------------------ | ------------------------------------------------------------- | -------------------------------------------------------------------- | ---------------------------------------------------- | -------------------------------------- |
| `PO`      | 最終判断者                                 | 目的、優先順位、公開方針、変更要求の採否、GO / Not GO         | 主要判断事項、未決事項、重大な課題・リスク、変更要求、公開前確認結果 | 採否判断、優先順位、公開可否、承認結果               | 重要判断時、作業節目、公開前           |
| `PM`      | 計画・進捗・課題・リスク・変更管理の運用者 | 作業順序、依存、期限超過、課題化、リスク化、再計画            | 進捗、ブロック理由、検証結果、レビュー結果、変更影響                 | 進捗報告、再計画案、課題・リスク・変更要求の整理結果 | 作業節目、Schedule 更新時、阻害発生時  |
| `BA`      | 利用者価値・受入条件の確認者               | 利用者向け説明、業務影響、受入条件、外部入力                  | 目的、対象範囲、利用者向け成果物、変更候補                           | 業務観点の懸念、受入条件、改善提案                   | 利用者向け成果物更新時、変更要求検討時 |
| `ARC`     | 文書体系・構成・依存の確認者               | 文書構成、ID、命名、依存関係、生成物との整合                  | 成果物追加・変更、Schedule 変更、構成変更案                          | 構成判断、命名・配置方針、依存影響                   | 構成変更時、成果物追加時               |
| `QE`      | 品質基準・検証・公開適性の確認者           | rulebook 適合、文書間矛盾、lint、レビュー指摘、公開不適切情報 | レビュー対象、差分、検証結果、生成物                                 | レビュー指摘、品質判断、公開前確認結果               | 主要成果物更新時、公開前               |
| `OPS`     | 公開・配布・運用影響の確認者               | 公開手順、外部提供、問い合わせ、継続運用負荷                  | 公開候補、変更要求、運用負荷、利用者向け案内                         | 公開準備状況、運用上の懸念、リリース影響             | 公開準備時、運用変更時                 |

## 3. 会議体計画

小規模運用では常設会議を増やさず、作業節目または判断が必要なときに同期・非同期の確認を行う。会議を開催しない場合も、判断内容と結果はプロジェクト登録簿、変更要求、決定記録、Pull Request、実行結果のいずれかに残す。

| 会議名           | 目的                                                             | 参加ロール                                        | 頻度                       | 成果物 / 証跡                                                      |
| ---------------- | ---------------------------------------------------------------- | ------------------------------------------------- | -------------------------- | ------------------------------------------------------------------ |
| 作業計画レビュー | 次に扱う成果物、順序、依存、ブロック理由を確認する               | `PM`、必要に応じて `PO`, `ARC`, `QE`              | 作業節目、Schedule 更新時  | Schedule、実行結果、課題                                           |
| 成果物レビュー   | 成果物の構造、内容、リンク、rulebook 適合を確認する              | `PM`, `QE`、観点に応じて `BA`, `ARC`              | 主要成果物更新時           | Pull Request、レビューコメント、修正差分                           |
| 課題・リスク確認 | 高優先度の課題、顕在化しそうなリスク、期限超過を確認する         | `PM`、必要に応じて `PO`, `QE`, 関連 Role code     | 阻害発生時、作業節目       | [[prj-0001:pjr-index\|プロジェクト登録簿]]、課題ログ、リスク登録簿 |
| 変更要求確認     | スコープ、責任分担、Schedule、公開方針への影響と採否案を確認する | `PM`, `PO`、必要に応じて `BA`, `ARC`, `OPS`, `QE` | 変更要求起票時             | 変更要求ログ、決定記録、関連成果物                                 |
| 公開前確認       | 公開範囲、検証結果、公開不適切情報、運用負荷を確認する           | `PO`, `PM`, `OPS`, `QE`、必要に応じて `ARC`       | 初回公開前、主要リリース前 | 公開前チェック結果、決定記録、README、LICENSE                      |

## 4. 報告計画

### 4.1. 報告種別

| 報告種別       | 受信ロール               | 目的                                                         | 期限 / タイミング          | 形式                                 | 責任ロール         |
| -------------- | ------------------------ | ------------------------------------------------------------ | -------------------------- | ------------------------------------ | ------------------ |
| 進捗報告       | `PO`, 関連 Role code     | 完了、着手中、ブロック、期限超過、次アクションを共有する     | 作業節目、Schedule 更新時  | 実行結果、進捗報告、Schedule 差分    | `PM`               |
| 課題報告       | `PO`, `PM`               | 発生済みの阻害要因、対応 Role code、期限、判断待ちを共有する | 課題発生時、期限超過時     | プロジェクト登録簿、課題ログ         | `PM`               |
| リスク報告     | `PO`, `PM`, `QE`         | 高影響リスク、顕在化兆候、対応期限を共有する                 | リスク識別時、見直し時     | プロジェクト登録簿、リスク登録簿     | `PM`               |
| 変更要求報告   | `PO`, `ARC`, `OPS`, `QE` | 変更内容、理由、影響先、代替案、採否案を共有する             | 変更要求起票時             | 変更要求ログ、決定記録、影響差分     | `PM`               |
| 品質・検証報告 | `PM`, `QE`, `ARC`        | lint、schema、build、レビュー指摘、不整合を共有する          | 主要成果物更新時、公開前   | 検証ログ、レビューコメント、修正差分 | `QE`               |
| 公開準備報告   | `PO`, `PM`, `OPS`, `QE`  | 公開可否判断に必要な検証結果、公開範囲、運用負荷を共有する   | 初回公開前、主要リリース前 | 公開前チェック結果、README、決定記録 | `OPS`、必要時 `PM` |

### 4.2. 報告に含める内容

進捗報告または作業節目の共有では、必要に応じて次を含める。

- 完了した成果物、変更内容、検証結果。
- 未完了の成果物、ブロック理由、次アクション、判断者。
- スコープ、Schedule、品質、公開判断への影響。
- 課題、リスク、変更要求、決定記録へ転記すべき事項。
- 期限超過または依存未解決がある場合の再計画案。

## 6. 情報管理と公開方針

### 6.1. 証跡管理

- 重要判断は、プロジェクト登録簿、決定記録、変更要求、Pull Request、Issue、実行結果のいずれかに残す。
- 課題、リスク、変更要求、決定事項は本文に混在させず、追跡できる単位へ分離する。
- 生成ビューと正本が矛盾する場合は、正本を更新してから生成物を再作成する。
- 報告の責任主体は Role code で記述し、実行主体や兼務割り当ては本書へ複製しない。

### 6.2. 公開情報の扱い

- 公開リポジトリには、個人名、個人連絡先、非公開の組織情報、認証情報、秘密情報を記載しない。
- AI Agent への入力にも、公開できない情報を含めない。
- 外部向け説明は、README、導入ガイド、サンプル、Issue / PR ルールへ集約する。
- 公開前確認では、成果物本文、生成物、差分、添付ファイルに公開不適切情報が含まれていないことを確認する。

| 更新トリガー                                                    | 見直し内容                                                   | 責任ロール        |
| --------------------------------------------------------------- | ------------------------------------------------------------ | ----------------- |

| 採用 Role code または責任分担を変更した                         | ロール別分類、報告責任、相談先、最終判断者を更新する         | `PM`, `ARC`       |
| 成果物カタログまたは Schedule を大きく変更した                  | 作業計画レビュー、進捗報告、変更要求報告の対象を更新する     | `PM`              |
| 課題滞留、リスク顕在化、変更要求増加が継続した                  | 報告頻度、エスカレーション条件、PO 判断のタイミングを見直す  | `PM`, `PO`        |
| 外部参加者の Issue / Pull Request 受付を開始した                | 外部向けチャネル、応答方針、レビュー導線を追加する           | `BA`, `QE`, `OPS` |
| 初回公開または主要リリースを行う                                | 公開前確認、公開準備報告、運用影響の連絡先を更新する         | `PO`, `OPS`, `QE` |
