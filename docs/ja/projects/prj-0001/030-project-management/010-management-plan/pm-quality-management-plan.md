# 品質管理計画

## 1. 概要

本書は、SpecDojo プロジェクトにおける品質目標、レビュー方針、品質メトリクス、検査基準、是正プロセスを定義する。

本プロジェクトの品質管理は、文書体系、ルールブック、サンプル、AI 向け指示テンプレート、補助ツール、生成物が、人間と AI Agent の双方にとって参照・再利用・更新しやすい状態を保つことを目的とする。

品質確認の実作業は主に `QE` が担い、最終判断は `PO` が行う。採用ロールと責務境界は [pm-organization.md](../020-organization/pm-organization.md) および [pm-raci.md](../020-organization/pm-raci.md) を正とする。

## 2. 品質目標

| 項目           | 目標値                                                                   | 期限 / タイミング                     | 判定方法                                         | 責任ロール  |
| -------------- | ------------------------------------------------------------------------ | ------------------------------------- | ------------------------------------------------ | ----------- |
| 文書構造適合   | 対象成果物の frontmatter、見出し、関連リンクがルールブックに適合している | 主要成果物レビュー時                  | ルールブック、サンプル、Markdown lint の確認     | `QE`        |
| 文書間整合     | 成果物カタログ、WBS、Schedule、RACI、管理計画の記述が矛盾していない      | 主要成果物更新時                      | 関連文書の横断確認                               | `QE`, `ARC` |
| AI 利用可能性  | AI Agent が参照して次の作業に使える粒度、ID、リンク、制約を持つ          | ルールブック / 指示テンプレート更新時 | 指示テンプレート、サンプル、生成結果の確認       | `QE`, `BA`  |
| 公開適性       | 個人情報、非公開情報、機密性の高い情報を含まない                         | 初回公開前 / 主要リリース前           | 公開前確認、差分確認                             | `QE`, `PO`  |
| レビュー指摘残 | 公開判断または完了判断時点で重大指摘 0 件                                | 公開前 / 完了判断前                   | レビュー記録、課題ログ、プロジェクト登録簿の確認 | `QE`        |
| 生成物整合     | generated 成果物が正本となる work / control 成果物と矛盾していない       | 生成物更新時                          | 生成元、生成結果、差分の確認                     | `ARC`, `QE` |

## 3. レビュープロセス

### 3.1. レビュー種別

| 種別            | 主な対象                                      | 実施ロール        | 頻度 / タイミング             | 出口条件                                               |
| --------------- | --------------------------------------------- | ----------------- | ----------------------------- | ------------------------------------------------------ |
| 構造レビュー    | frontmatter、ID、見出し、リンク、配置         | `QE`, `ARC`       | 新規成果物作成時 / 主要更新時 | ルールブックと成果物カタログに対する不整合がない       |
| 内容レビュー    | 目的、範囲、方針、基準、未決事項              | `PO`, `BA`, `QE`  | 主要成果物更新時              | 意味の矛盾、過不足、曖昧な判断条件が解消されている     |
| 整合性レビュー  | 成果物カタログ、WBS、Schedule、RACI、管理計画 | `ARC`, `QE`       | 関連文書をまたぐ更新時        | 参照先、責任分担、生成元、派生関係が矛盾していない     |
| 公開前レビュー  | README、LICENSE、公開対象文書、サンプル       | `PO`, `ARC`, `QE` | 初回公開前 / 主要リリース前   | 個人情報、非公開情報、機密情報、未判断の公開事項がない |
| AI 利用レビュー | ルールブック、指示テンプレート、サンプル      | `BA`, `QE`        | AI 向け成果物更新時           | AI Agent が作業可能な入力、制約、出力条件が明確である  |

### 3.2. レビュー手順

1. レビュー対象、根拠文書、関連文書を確認する。
2. ルールブック、サンプル、成果物カタログ、RACI との整合を確認する。
3. 指摘事項を Pull Request、レビューコメント、プロジェクト登録簿、課題ログのいずれかに記録する。
4. 指摘対応後、必要に応じて再レビューを行う。
5. 重大指摘が残っていないことを `QE` が確認し、最終判断を `PO` が行う。

## 4. 品質メトリクス

| 指標               | 算出方法                                                                                                                     | 閾値       | 計測頻度                | 報告先            |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------- | ---------- | ----------------------- | ----------------- |
| Markdown lint 結果 | `npm run -s lint:md` の失敗件数                                                                                              | 0 件       | 主要成果物更新時        | `PO`, `QE`        |
| YAML schema 検証   | `npm run validate:schema:file` のエラー件数（`pm-roles.yaml`, `pm-members.yaml` 等の YAML 成果物が schema に適合しているか） | 0 件       | YAML 成果物更新時       | `ARC`, `QE`       |
| ビルド検証         | `npm run docs:build` の失敗件数                                                                                              | 0 件       | 公開前 / 主要リリース前 | `PO`, `QE`        |
| 重大レビュー指摘残 | 公開・完了判断を妨げる未解決指摘件数                                                                                         | 0 件       | 公開前 / 完了判断前     | `PO`, `QE`        |
| 関連リンク不整合   | 存在しない参照、誤った相対リンク、古い成果物名の件数                                                                         | 0 件       | 主要成果物更新時        | `ARC`, `QE`       |
| ルールブック不適合 | 対象成果物が該当ルールブックの必須構成を満たさない件数                                                                       | 0 件       | 新規成果物作成時        | `PO`, `QE`        |
| 文書間矛盾         | 成果物カタログ、RACI、WBS、Schedule、管理計画間の矛盾件数                                                                    | 0 件       | 関連文書更新時          | `PO`, `ARC`, `QE` |
| 公開不適切情報     | 個人情報、非公開情報、機密情報の検出件数                                                                                     | 0 件       | 公開前 / 主要リリース前 | `PO`, `QE`        |
| 未決事項の放置     | 期限または判断タイミングを過ぎた `_UNDECIDED_` 件数                                                                          | 0 件を目標 | 作業節目ごと            | `PO`              |

## 5. 検査基準と是正

### 5.1. 検査基準

成果物を完了または公開候補として扱うには、次の条件を満たすこと。

| 観点    | 受入基準                                                                      |
| ------- | ----------------------------------------------------------------------------- |
| 構造    | frontmatter、見出し、必須セクション、関連リンクが該当ルールブックに沿っている |
| 内容    | 目的、対象範囲、判断条件、責任ロール、未決事項が読み取れる                    |
| 整合    | 成果物カタログ、組織定義、RACI、管理計画、生成物と矛盾していない              |
| 追跡    | 根拠文書、派生文書、関連する登録簿または生成物への導線がある                  |
| 公開    | 個人情報、非公開情報、機密情報を含まず、公開前に必要な未決事項が残っていない  |
| AI 利用 | AI Agent が参照すべき入力、制約、出力形式が明確である                         |

### 5.2. 是正プロセス

| 不適合区分 | 例                                                     | 是正方法                                   | 再判定                            |
| ---------- | ------------------------------------------------------ | ------------------------------------------ | --------------------------------- |
| 軽微       | 誤字、表記ゆれ、リンク表示の軽微な修正                 | 通常の文書更新として修正する               | `QE` が必要に応じて確認する       |
| 構造不適合 | 必須見出し不足、frontmatter 不備、ID 不整合            | 該当ルールブックに合わせて修正する         | `QE` と `ARC` が確認する          |
| 内容不整合 | 関連文書間の矛盾、責任ロールの食い違い                 | 正となる文書を確認し、影響範囲を修正する   | `PO` が最終判断する               |
| 公開不適合 | 個人情報、非公開情報、機密情報、未判断の公開事項       | 公開対象から除外または内容を修正する       | `PO`, `QE` が公開可否を再判定する |
| 重大       | 目的、スコープ、公開方針、ライセンス方針に影響する不備 | プロジェクト登録簿または変更要求として扱う | `PO` が対応方針を決定する         |

### 5.3. 生成物の扱い

- `generated` 配下の成果物は、正本となる work / control 成果物から派生する補助成果物として扱う。
- 正本と生成物が矛盾する場合は、正本を修正したうえで生成物を再作成する。
- 生成物を手修正した場合は、再生成時に失われる可能性があるため、必要な変更は生成元へ反映する。

## 6. 品質上の役割分担

| Role code | 品質管理上の責務                                                         |
| --------- | ------------------------------------------------------------------------ |
| `PO`      | 品質基準、公開可否、重大不適合、変更要求の最終判断を行う                 |
| `BA`      | 利用者視点、導入しやすさ、受入条件、サンプルの分かりやすさを確認する     |
| `ARC`     | 文書体系、ID、命名、配置、成果物カタログ、生成物との整合を確認する       |
| `QE`      | レビュー観点、ルール適合、文書間矛盾、公開適性、品質メトリクスを確認する |

## 8. 見直し条件

| 更新トリガー                                             | 見直し内容                                     | 責任ロール  |
| -------------------------------------------------------- | ---------------------------------------------- | ----------- |
| 成果物カタログまたは文書構成を変更した                   | 品質目標、レビュー観点、整合性確認対象を見直す | `ARC`, `QE` |
| ルールブック、サンプル、指示テンプレートを追加・変更した | ルール適合基準と AI 利用レビュー観点を見直す   | `BA`, `QE`  |
| 公開方針または公開対象を変更した                         | 公開前レビュー、公開不適合基準を見直す         | `PO`, `QE`  |
| 採用ロールまたは RACI を変更した                         | 品質上の役割分担、承認責任、報告先を見直す     | `PO`, `QE`  |
| 重大な不整合または手戻りが発生した                       | メトリクス、閾値、是正プロセスを見直す         | `PO`, `QE`  |

| 論点                       | 現状        | 対応方針                                                        | 判断者      |
| -------------------------- | ----------- | --------------------------------------------------------------- | ----------- |
| 公開前チェックリストの形式 | _UNDECIDED_ | 初回公開前に Markdown または Issue テンプレートとして定義する   | `PO`, `QE`  |
| リンクチェックの自動化     | _UNDECIDED_ | 必要に応じて lint または補助ツールに追加する                    | `ARC`, `QE` |
| 生成物の再生成手順         | _UNDECIDED_ | `specdojo` コマンド運用が固まり次第、手順を関連ガイドへ反映する | `ARC`       |
