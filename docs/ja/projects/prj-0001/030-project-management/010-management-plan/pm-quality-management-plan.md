# 品質管理計画

本書は、SpecDojo プロジェクトにおける品質目標、レビュー方針、品質メトリクス、検査基準、是正プロセスを、PM が計画・進捗・課題・リスク管理へ接続できる粒度で定義する。

## 1. 概要

本計画は、[[prj-0001:pm-plan|プロジェクト管理計画]] が定める品質・リスク・課題管理方針を前提に、成果物と参考資料の品質確認、レビュー、検証、是正の運用を具体化する。PM は品質上の確認事項を作業計画、進捗、課題、リスク、変更要求へ接続できる単位に分離し、QE と PO が判断できる材料を整える。

本プロジェクトの品質管理は、文書体系、参考資料、補助ツール、生成物が、人間と AI Agent の双方にとって参照・再利用・更新しやすい状態を保つことを目的とする。採用する Role code の語彙は [[prj-0001:pm-organization|組織定義]] と [[prj-0001:pm-roles|ロール定義]] を正とし、成果物別の責任分担は [[prj-0001:pm-raci|RACI]] を確認元とする。

AI Agent は文書草案、差分整理、構成確認、レビュー支援を行ってよい。ただし、最終承認、公開可否、重大不適合の採否、GO / Not GO 判断は人間の判断に残す。

## 2. 品質目標

| 項目 | 目標値 | 期限 / タイミング | 判定方法 | 責任ロール |
| --- | --- | --- | --- | --- |
| 文書構造適合 | 対象成果物の frontmatter、見出し、必須表、関連リンクが rulebook に適合している | 主要成果物レビュー時 | rulebook、sample、Markdown lint の確認 | `PM`, `QE` |
| 文書間整合 | 成果物カタログ、Schedule、RACI、管理計画の記述が矛盾していない | 主要成果物更新時 | 関連文書の横断確認 | `PM`, `ARC`, `QE` |
| AI 利用可能性 | AI Agent が参照して次の作業に使える粒度、ID、リンク、制約を持つ | 参考資料更新時 | rulebook、recipe、sample、template、実行結果の確認 | `PM`, `BA`, `QE` |
| 公開適性 | 個人情報、非公開情報、認証情報、秘密情報を含まない | 初回公開前 / 主要リリース前 | 公開前確認、差分確認 | `PO`, `QE` |
| レビュー指摘残 | 公開判断または完了判断時点で重大指摘 0 件 | 公開前 / 完了判断前 | レビュー記録、課題ログ、プロジェクト登録簿の確認 | `QE` |
| 生成物整合 | generated 成果物が正本となる work / control 成果物と矛盾していない | 生成物更新時 | 生成元、生成結果、差分の確認 | `ARC`, `QE` |

## 3. レビュープロセス

### 3.1. レビュー種別

| 種別 | 主な対象 | 実施ロール | 頻度 / タイミング | 出口条件 |
| --- | --- | --- | --- | --- |
| 構造レビュー | frontmatter、ID、見出し、リンク、配置 | `PM`, `QE`, `ARC` | 新規成果物作成時 / 主要更新時 | rulebook と成果物カタログに対する不整合がない |
| 内容レビュー | 目的、範囲、方針、基準、未決事項 | `PO`, `BA`, `QE` | 主要成果物更新時 | 意味の矛盾、過不足、曖昧な判断条件が解消されている |
| 整合性レビュー | 成果物カタログ、Schedule、RACI、管理計画 | `PM`, `ARC`, `QE` | 関連文書をまたぐ更新時 | 参照先、責任分担、生成元、派生関係が矛盾していない |
| 公開前レビュー | README、LICENSE、公開対象文書、サンプル、生成物 | `PO`, `ARC`, `QE`, `OPS` | 初回公開前 / 主要リリース前 | 個人情報、非公開情報、認証情報、秘密情報、未判断の公開事項がない |
| AI 利用レビュー | rulebook、recipe、sample、template、exec plan / result | `PM`, `BA`, `QE` | 参考資料更新時 | AI Agent が作業可能な入力、制約、出力条件、記録先が明確である |

### 3.2. レビュー手順

1. PM はレビュー対象、根拠文書、関連文書、必要な検証コマンドを特定する。
2. 実施ロールは rulebook、sample、成果物カタログ、RACI、Schedule との整合を確認する。
3. 指摘事項は Pull Request、レビューコメント、プロジェクト登録簿、課題ログのいずれかに記録し、重大度と対応 Role code を分ける。
4. 指摘対応後、PM は課題、リスク、変更要求、決定記録へ転記すべき事項を分離する。
5. QE は重大指摘が残っていないことを確認し、公開可否または重大不適合の最終判断を PO へ上げる。

| --- | --- | --- | --- | --- |

| 重大レビュー指摘残 | 公開・完了判断を妨げる未解決指摘件数 | 0 件 | 公開前 / 完了判断前 | `PO`, `QE` |
| 関連リンク不整合 | 存在しない参照、誤った相対リンク、古い成果物名の件数 | 0 件 | 主要成果物更新時 | `ARC`, `QE` |
| rulebook 不適合 | 対象成果物が該当 rulebook の必須構成を満たさない件数 | 0 件 | 新規成果物作成時 | `PM`, `QE` |
| 文書間矛盾 | 成果物カタログ、RACI、Schedule、管理計画間の矛盾件数 | 0 件 | 関連文書更新時 | `PM`, `ARC`, `QE` |
| 公開不適切情報 | 個人情報、非公開情報、認証情報、秘密情報の検出件数 | 0 件 | 公開前 / 主要リリース前 | `PO`, `QE` |
| 未決事項の放置 | 期限または判断タイミングを過ぎた `_UNDECIDED_` 件数 | 0 件を目標 | 作業節目ごと | `PM`, `PO` |

## 5. 検査基準と是正

### 5.1. 検査基準

成果物を完了または公開候補として扱うには、次の条件を満たすこと。

| 観点 | 受入基準 |
| --- | --- |
| 構造 | frontmatter、見出し、必須セクション、関連リンクが該当 rulebook に沿っている |
| 内容 | 目的、対象範囲、判断条件、責任ロール、未決事項が読み取れる |
| 整合 | 成果物カタログ、組織定義、RACI、管理計画、生成物と矛盾していない |
| 追跡 | 根拠文書、派生文書、関連する登録簿または生成物への導線がある |
| 公開 | 個人情報、非公開情報、認証情報、秘密情報を含まず、公開前に必要な未決事項が残っていない |
| AI 利用 | AI Agent が参照すべき入力、制約、出力形式、記録先が明確である |

### 5.2. 是正プロセス

| 不適合区分 | 例 | 是正方法 | 再判定 |
| --- | --- | --- | --- |
| 軽微 | 誤字、表記ゆれ、リンク表示の軽微な修正 | 通常の文書更新として修正する | `PM` または `QE` が必要に応じて確認する |
| 構造不適合 | 必須見出し不足、frontmatter 不備、ID 不整合 | 該当 rulebook に合わせて修正する | `QE` と `ARC` が確認する |
| 内容不整合 | 関連文書間の矛盾、責任ロールの食い違い | 正となる文書を確認し、影響範囲を修正する | `PO` が最終判断する |
| 公開不適合 | 個人情報、非公開情報、認証情報、秘密情報、未判断の公開事項 | 公開対象から除外または内容を修正する | `PO`, `QE` が公開可否を再判定する |
| 重大 | 目的、スコープ、公開方針、ライセンス方針に影響する不備 | プロジェクト登録簿または変更要求として扱う | `PO` が対応方針を決定する |

### 5.3. 生成物の扱い

- `generated` 配下の成果物は、正本となる work / control 成果物から派生する補助成果物として扱う。
- 正本と生成物が矛盾する場合は、正本を修正したうえで生成物を再作成する。
- 生成物を手修正した場合は、再生成時に失われる可能性があるため、必要な変更は生成元へ反映する。
- 生成物の不整合が Schedule、PJR、公開判断へ影響する場合は、PM が課題またはリスクとして分離する。

## 6. 品質上の役割分担

| Role code | 品質管理上の責務 |
| --- | --- |
| `PO` | 品質基準、公開可否、重大不適合、変更要求の最終判断を行う |
| `PM` | 品質確認を作業計画、進捗、課題、リスク、変更要求へ接続し、判断材料を整理する |
| `BA` | 利用者視点、導入しやすさ、受入条件、サンプルの分かりやすさを確認する |
| `ARC` | 文書体系、ID、命名、配置、成果物カタログ、生成物との整合を確認する |
| `DEV` | 技術的変更、検証コマンド、実装・設定作業の実現可能性を確認する |
| `QE` | レビュー観点、rulebook 適合、文書間矛盾、公開適性、品質メトリクスを確認する |
| `UX` | 利用者導線、説明、文書体験、読みやすさの観点を確認する |
| `OPS` | 公開、配布、リリース、運用負荷、外部提供への影響を確認する |
