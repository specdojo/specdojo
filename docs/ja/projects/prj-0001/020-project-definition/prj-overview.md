# プロジェクト概要: SpecDojo

## 1. 背景

生成AIおよびAI Agentの発展により、システムの企画、要件定義、設計、実装、テスト、運用・保守まで、AIが支援・実行できる領域が急速に拡大している。

その一方で、AI Agentが適切に作業するためには、単に「何を作るか」が記述されているだけでは不十分である。なぜそのシステムや機能が必要なのか、誰のどの課題を解決するのか、どのような成果を目指すのか、どの判断に基づいて仕様が決まったのかが、構造的に記録されている必要がある。

従来の開発文書は、主に人間同士の理解、合意形成、契約、納品を目的として作成されてきた。しかし今後は、人間に加えてAI Agentも読み手・実行主体となる。そのため、ドキュメントには、人が理解しやすいことに加え、AI Agentが解釈し、参照し、整合性を確認し、タスク実行に活用できることが求められる。

特に、シビックテック、地域活動、教育、福祉、行政、NPO、OSS、企業内の業務改善など、限られた人材・時間・予算で社会的な課題解決に取り組むプロジェクトでは、プロジェクトの目的、判断根拠、成果物、変更履歴、運用知識を継続的に引き継げる形で残すことが重要である。

本プロジェクトでは、このような課題に対応するため、人間とAI Agentが共有できる成果物体系を中心に、プロジェクトを構築・改修・運用し、継続的に改善していく開発・運営スタイルを **成果物駆動プロジェクト運営** と位置付ける。

SpecDojo は、Waterfall や PMBOK で重視されてきた成果物、責任分担、変更管理、品質管理、トレーサビリティの考え方を参考にする。ただし、Waterfall のように工程順序を固定するものではない。要件定義から始めても、プロトタイプ開発から始めても、運用課題から始めてもよい。どこから始めても、成果物間の関係を整理し、必要に応じて上流工程に戻り、判断根拠と変更履歴を残せることを重視する。

SpecDojo の基本思想は、**Why before What** である。
何を作るかを決める前に、なぜ作るのか、誰のために作るのか、どの成果を目指すのかを明らかにする。ただし、Why を最初から完全に固定するのではなく、プロトタイプ、利用者フィードバック、運用経験を通じて継続的に更新する。

## 2. 目的

- AI Agent時代の成果物駆動プロジェクト運営を支える、オープンなドキュメントフレームワーク **SpecDojo** を整備する。

- システムの構築・改修・運用・保守に必要となり得る文書群を体系化する。
  ただし、すべての文書作成を強制するものではなく、プロジェクトの目的、規模、リスク、公共性、運用責任に応じて、必要な成果物を選択できるようにする。

- Waterfall / PMBOK 型の成果物体系を参考にしつつ、工程順序を固定しない、非線形・選択式の成果物駆動フレームワークを提供する。
  要件定義、設計、実装、テスト、運用のどこから始めても、必要に応じて Why、What、How に戻り、成果物間の整合性を保ちながらプロジェクトを前進できる状態を目指す。

- 人とAI Agentが共有可能なプロジェクト成果物を中核に据え、開発の生産性、品質、透明性、継続性を高める。
  これにより、多様なプロジェクトが限られた人材・時間・予算の中でも、社会課題の解消に資するシステムを継続的に構築・改善できる状態を目指す。

- 特に、シビックテック、地域活動、教育、福祉、行政、NPO、OSSなど、公共性や継続性が重要なプロジェクトにおいて、市民、専門家、行政、開発者、AI Agentが協働しやすい共通の型を提供する。

- SpecDojo では、オープンソースとして公開することを中核的な方針とする。
  成果物駆動プロジェクト運営の知見を特定の組織や製品に閉じず、誰もが自由に利用・改善・再利用できる共通基盤として提供する。
  これにより、車輪の再発明を防ぎ、各現場の知見を SpecDojo に還元しやすくし、文書体系や開発プロセスのブラックボックス化を抑制する。

## 3. 必要性

課題解決型プロジェクトにおけるドキュメンテーションは、担当者の経験や力量に依存しやすく、標準化・再利用・継続的更新が進みにくい。

その結果、多くのプロジェクトでは以下の課題が生じている。

- 目的や解決すべき課題が曖昧なまま、実装やツール選定が先行する
- 要件や設計意図の伝達不足による認識齟齬・手戻り
- 仕様や判断経緯の不足による属人化・ブラックボックス化
- 担当者交代や長期運用時の保守・改修困難化
- 変更時に、どの要件・設計・実装・運用ルールへ影響するか追跡できない
- シビックテックや地域活動において、善意や個人の熱量に依存し、継続的な運用や引き継ぎが難しくなる

特に、社会課題の解消に取り組むプロジェクトでは、限られたリソースの中で、継続的にシステムを改善し続ける必要がある。
そのためには、プロジェクトの知識を人とAI Agentが共有・活用できる形で残し、引き継ぎ、再利用できる基盤が必要である。

SpecDojo は、開発文書を単なる納品物や記録ではなく、人とAI Agentが協働するための共有知識基盤として位置付ける。

また、SpecDojo は Waterfall 型の成果物を網羅的に参照可能にするが、Waterfall 型の進行順序や全ドキュメント作成を強制しない。
プロジェクトは、課題整理、要件定義、プロトタイプ、既存システム改修、運用課題、障害対応など、どこから始めてもよい。重要なのは、開始地点ではなく、各成果物が Why、What、How、判断根拠、変更履歴に接続され、後から人間とAI Agentが追跡・再利用できることである。

このような成果物体系により、SpecDojo は、AI coding のための仕様管理にとどまらず、AI Agent時代におけるプロジェクト運営、上流工程、変更管理、継続改善を支える基盤となることを目指す。

## 4. 期待効果

SpecDojo の整備により、成果物駆動プロジェクト運営を実践可能にし、社会課題の解消に資する課題解決型プロジェクトの成功確率、実行速度、透明性、継続性を高めることを期待する。

短期的には、以下の効果を目指す。

- プロジェクトの立ち上げから初回リリースまでの期間短縮
- Why、What、How の接続による、目的・要件・仕様・設計・実装・テストの一貫性向上
- AI Agent の活用による、限られた体制での開発実行力向上
- 仕様・設計意図・判断経緯・変更履歴の共有による、継続的な改善・引き継ぎの容易化
- 開発先行、運用先行、課題整理先行など、さまざまな開始地点からでも、必要な上流成果物へ戻れる柔軟性の確保
- シビックテックや地域活動における、非エンジニアを含む参加者の貢献機会拡大

中長期的には、以下の効果を目指す。

- シビックテック、地域活動、教育、福祉、行政、NPO、OSS、企業内業務改善等における課題解決型プロジェクトの創出数増加
- 社会課題の解消・軽減につながるシステムの実装件数増加
- 小規模組織、非専門家を含むチーム、地域コミュニティによるシステム開発・改善の参加機会拡大
- 価値あるプロジェクトが属人化・資金不足・人材不足により停滞または消滅するリスクの低減
- ユーザー企業、地域団体、教育機関、NPO等が、外部依存だけに頼らず、自ら課題を定義し、AI Agentと協働して改善を進められる状態の拡大
- AI Agent時代に適した、透明で再利用可能なプロジェクト成果物体系の普及

効果は、短期指標と中長期指標に分けて確認する。なお、下記の目標値は、
AI Agent を前提とした成果物駆動プロジェクト運営が十分に機能した場合の挑戦的な初期仮説であり、実測値に基づき継続的に見直す。

## 5. 前提条件

SpecDojo の策定および活用にあたっては、以下を前提とする。

- SpecDojo は、人間の判断や責任を代替するものではなく、人とAI Agentが協働するための基盤であること
- SpecDojo は、AI Agentに上流工程の意思決定を委ねるものではなく、課題、目的、選択肢、判断根拠を整理し、人間の意思決定を支援するものであること
- 対象範囲は、企画、課題定義、要件定義、設計、実装、テスト、運用・保守、継続改善を含むシステムライフサイクル全体とすること
- SpecDojo は、Waterfall / PMBOK 型の成果物体系を参考にするが、Waterfall 型の工程順序を強制しないこと
- すべてのドキュメント作成を強制せず、プロジェクトの目的、規模、リスク、公共性、運用責任に応じて、必要な成果物を選択すること
- プロジェクトは、要件定義、設計、実装、テスト、運用、課題整理、プロトタイプなど、どこから開始してもよいこと
- どこから開始した場合でも、成果物を Why、What、How、判断根拠、変更履歴に接続し、後から追跡・再利用できる状態を目指すこと
- ドキュメントは、人間にとっての理解容易性と、AI Agentにとっての解釈容易性を両立すること
- 特定の開発手法、言語、基盤、生成AI製品に過度に依存しないこと
- SpecDojo はオープンソースとして公開し、企業、個人、行政、教育機関、地域コミュニティ、NPO、OSS プロジェクト等が自由に利用・改善・再利用できること
- 各現場の知見を SpecDojo に還元しやすい構造とし、継続的に改善される共通基盤を目指すこと
