# プロジェクト概要: SpecDojo

## 1. 背景

生成AIおよびAI Agentの発展により、サービスやシステムの企画、要件定義、設計、実装、テスト、運用・保守まで、AIが支援・実行できる領域が拡大している。

一方で、AI Agentが適切に作業するためには、要件や仕様だけでなく、なぜそのサービス・システムが必要なのか、誰のどの課題を解決するのか、どのような成果を目指すのか、どの判断に基づいて仕様が決まったのかが、構造的に記録されている必要がある。

従来のドキュメントは、主に人間同士の理解や合意形成を目的としていた。しかし今後は、人間に加えてAI Agentも読み手・実行主体となるため、ドキュメントには、人が理解しやすいことに加え、AI Agentが解釈し、参照し、タスク実行に活用できることが求められる。

## 2. 必要性

課題解決型プロジェクトにおけるドキュメンテーションは、担当者の経験や力量に依存しやすく、標準化・再利用・継続的更新が進みにくい。

その結果、多くのプロジェクトでは以下の課題が生じている。

- 目的や解決すべき課題が曖昧なまま、実装やツール選定が先行する
- 要件や設計意図の伝達不足による認識齟齬・手戻り
- 仕様や判断経緯の不足による属人化・ブラックボックス化
- 担当者交代や長期運用時の保守・改修困難化
- 変更時に、どの要件・設計・実装・運用ルールへ影響するか追跡できない
- シビックテックや地域活動において、善意や個人の熱量に依存し、継続的な運用や引き継ぎが難しくなる

特に、社会課題の解消に取り組むプロジェクトでは、限られたリソースの中で、継続的にサービスやシステムを改善し続ける必要がある。そのためには、プロジェクトの知識を人とAI Agentが共有・活用できる形で残し、引き継ぎ、再利用できる基盤が必要である。

SpecDojo は、開発文書を単なる納品物や記録ではなく、人とAI Agentが協働するための共有知識基盤として位置付ける。

## 3. 基本的な考え方

本プロジェクトでは、人間とAI Agentが共有できる成果物体系を中心に、プロジェクトを構築・改修・運用し、継続的に改善していくスタイルを **成果物駆動プロジェクト運営** と位置付ける。

SpecDojo は、Waterfall や PMBOK で重視されてきた成果物、責任分担、変更管理、品質管理、トレーサビリティを参考にする。ただし、Waterfall のように工程順序を固定するものではない。要件定義から始めても、プロトタイプ開発から始めても、運用課題から始めてもよい。

SpecDojo の基本思想は、**Why before What** である。
何を作るかを決める前に、なぜ作るのか、誰のために作るのか、どの成果を目指すのかを明らかにする。ただし、Why は最初から固定するものではなく、学びに応じて継続的に更新する。

## 4. 実現したいこと

- AI Agent時代の成果物駆動プロジェクト運営を支える、オープンなドキュメントフレームワーク **SpecDojo** を整備する。

- サービスやシステムの構築・改修・運用・保守に必要となり得る文書群を体系化するとともに、規約、テンプレート、サンプル、各種ツール等を通じて、ライフサイクル全体を一貫して支援する。
  ただし、すべての文書作成を強制せず、プロジェクトの目的、規模、リスク、公共性、運用責任に応じて、必要な成果物を選択できるようにする。

- Waterfall / PMBOK 型の成果物体系を参考にしつつ、工程順序を固定しない、非線形・選択式の成果物駆動フレームワークを提供する。

- 人とAI Agentが共有可能な成果物を中核に据え、開発の生産性、品質、透明性、継続性を高める。

- 特に、シビックテック、地域活動、教育、福祉、行政、NPO、OSSなど、公共性や継続性が重要なプロジェクトにおいて、市民、専門家、行政、開発者、AI Agentが協働しやすい共通の型を提供する。

- SpecDojo をオープンソースとして公開し、誰もが自由に利用・改善・再利用できる共通基盤として育てる。

## 5. 期待効果

SpecDojo の整備により、成果物駆動プロジェクト運営を実践可能にし、社会課題の解消に資する課題解決型プロジェクトの成功確率、実行速度、透明性、継続性を高めることを期待する。

短期的には、以下の効果を目指す。

- プロジェクトの立ち上げから初回リリースまでの期間短縮
- Why、What、How の接続による、目的・要件・仕様・設計・実装・テストの一貫性向上
- AI Agent の活用による、限られた体制での開発実行力向上
- 仕様・設計意図・判断経緯・変更履歴の共有による、継続的な改善・引き継ぎの容易化
- 開発先行、運用先行、課題整理先行など、さまざまな開始地点からでも、必要な上流成果物へ戻れる柔軟性の確保
- シビックテックや地域活動における、非エンジニアを含む参加者の貢献機会拡大

中長期的には、以下の効果を目指す。

- シビックテック、地域活動、教育、福祉、行政、NPO、OSS、企業内業務改善等における課題解決型プロジェクトの創出数増加
- 社会課題の解消・軽減につながるサービスやシステムの実装件数増加
- 小規模組織、非専門家を含むチーム、地域コミュニティによるシビックテック開発・改善の参加機会拡大
- 価値あるプロジェクトが属人化・資金不足・人材不足により停滞または消滅するリスクの低減
- AI Agent時代に適した、透明で再利用可能なプロジェクト成果物体系の普及

効果は、短期指標と中長期指標に分けて確認する。短期的には、SpecDojo を適用したプロジェクトの QCD、すなわち品質、コスト、納期への効果を確認する。中長期的には、SpecDojo を通じて実現されたプロジェクトが、社会課題の解消や地域・公共活動の継続性にどの程度貢献したかを確認する。

なお、下記の目標値は、AI Agent を前提とした成果物駆動プロジェクト運営が十分に機能した場合の挑戦的な初期仮説であり、実測値に基づき継続的に見直す。

### 5.1. 短期指標

短期的には、SpecDojo を適用したプロジェクトにおける QCD への効果を確認する。

| 観点     | 指標                               | 目標    | 測定方法                                                                                           |
| -------- | ---------------------------------- | ------- | -------------------------------------------------------------------------------------------------- |
| Quality  | Why に基づく目的達成度             | 90%以上 | プロジェクト定義で定めた目的・課題・成功条件に対して、初回リリース時点で実現できた項目の割合を確認 |
| Cost     | 同等規模テーマに必要な作業工数     | 50%削減 | 成果物作成、実装、レビュー、修正に要した作業時間を比較                                             |
| Delivery | 企画開始から初回リリースまでの期間 | 50%短縮 | 企画開始日から初回リリース日までの日数を比較                                                       |

### 5.2. 中長期指標

中長期的には、SpecDojo を通じて実現されたプロジェクトが、社会課題の解消、地域・公共活動の継続性、参加機会の拡大にどの程度貢献したかを確認する。

| 観点           | 指標                                                           | 目標        | 測定方法                                                         |
| -------------- | -------------------------------------------------------------- | ----------- | ---------------------------------------------------------------- |
| 社会課題の解消 | 社会課題の解消・軽減につながるサービスやシステムの継続運用件数 | 年率30%増加 | 活用事例、導入報告、運用状況、利用者フィードバック等から確認     |
| 継続性         | 価値あるプロジェクトが一定期間後も継続運用・改善されている割合 | 70%以上     | 一定期間後のリリース、Issue対応、保守、文書更新状況を確認        |
| 参加機会       | 非エンジニア・市民参加者が成果物更新や改善提案に関与した件数   | 年率30%増加 | 課題定義、要件、運用ルール、議事録、判断記録、改善提案件数を集計 |
| 公共性・透明性 | 判断根拠、成果物、変更履歴を公開しているプロジェクトの割合     | 70%以上     | 公開リポジトリ、公開文書、事例記事、導入報告等から確認           |

参考指標として、SpecDojo の導入プロジェクト数、テンプレート再利用件数、公開事例数、外部貢献件数を継続的に確認する。

## 6. 前提条件

SpecDojo の策定および活用にあたっては、以下を前提とする。

- SpecDojo は、人間の判断や責任を代替するものではなく、人とAI Agentが協働するための基盤であること
- AI Agentは、上流工程の意思決定者ではなく、課題、目的、選択肢、判断根拠を整理し、人間の意思決定を支援する存在であること
- ドキュメントは、人間にとっての理解容易性と、AI Agentにとっての解釈容易性を両立すること
- SpecDojo は、特定の開発手法、言語、基盤、生成AI製品に過度に依存しないこと
- 各プロジェクトは、目的、規模、リスク、公共性、運用責任に応じて、SpecDojo の成果物体系を取捨選択して適用すること
- SpecDojo は、オープンソースとして公開し、実際の活用事例や各現場の知見を取り込みながら継続的に改善すること
