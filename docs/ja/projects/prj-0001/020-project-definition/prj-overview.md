# プロジェクト概要: SpecDojo

SpecDojo は、人と AI Agent が共有できる仕様体系を通じて、プロジェクトの立ち上げ、改善、運用を支援し、限られた体制でも判断根拠と成果物を継続的に引き継げるようにするオープンなドキュメントフレームワークである。

## 1. 背景

社会課題の解消や地域・公共活動の継続には、多様な主体が協力してプロジェクトを立ち上げ、改善し続ける仕組みが必要である。しかし、その担い手となる市民、専門家、行政、開発者のリソースは限られており、知識の属人化・断絶・引き継ぎ困難という壁が繰り返し生じている。

こうした状況において、生成AIおよびAI Agentの発展は、限られた体制でも多くのことを実現できる可能性を広げている。企画から要件・仕様定義、設計、実装、テスト、運用・保守まで、AIが支援・実行できる領域が拡大しているからである。

## 2. 必要性

しかし、この可能性を活かすには、プロジェクトの知識や判断を継続的に共有・更新できる形にする必要がある。その中心的な課題が、言語化の難しさである。言語化は、担当者の経験や力量に依存しやすく、標準化・再利用・継続的更新が進みにくい。

その結果、多くのプロジェクトでは以下の問題が生じている。

- 目的や解決すべき課題が曖昧なまま、実装やツール選定が先行する
- 要件や設計意図の伝達不足による認識齟齬・手戻り
- 仕様や判断経緯の不足による属人化・ブラックボックス化
- 担当者交代や長期運用時の保守・改修困難化
- 変更時に、どの要件・設計・実装・運用ルールへ影響するか追跡できない
- 個人の善意や熱量に依存し、継続的な運用や引き継ぎが難しくなる

加えて、AIの力を引き出すためには、人間同士の理解や合意形成を目的とした言語化のあり方を変える必要がある。今後は、人間に加えてAI Agentも実行主体となるため、人が理解しやすいことに加え、AI Agentが解釈し、参照し、タスク実行に活用できることが求められる。

## 3. 基本的な考え方

こうした課題に対して、SpecDojo は、言語化の結果であるドキュメントを、単なる納品物や記録ではなく、人とAI Agentが協働するための共有知識基盤として位置付け、再構築する。

1. **Specification-Driven（仕様駆動）**: 要求、要件、仕様、設計を分断せず、人間とAI Agentが共有できる仕様体系として整理し、それを中心にプロジェクトを構築・改修・運用する。
2. **Why before What（Whatの前にWhy）**: 何を作るかを決める前に、なぜ作るのか、誰のために作るのか、どの成果を目指すのかを明らかにする。
3. **Start Anywhere, Evolve Always（どこからでも、育て続ける）**: どこから始めてもよい。Why は最初から固定せず、学びに応じて継続的に更新する。

SpecDojo は、Waterfall や PMBOK で重視されてきた成果物、責任分担、変更管理、品質管理、トレーサビリティを参考にする。ただし、Waterfall のように工程順序を固定するものではない。要件定義から始めても、プロトタイプ開発から始めても、運用課題から始めてもよい。

## 4. 実現したいこと

- AI Agent 時代の仕様駆動プロジェクト運営を支える、オープンなドキュメントフレームワーク **SpecDojo** を整備する。

- サービスやシステムの構築・改修・運用・保守に必要となり得る文書群を体系化するとともに、規約、テンプレート、サンプル、各種ツール等を通じて、ライフサイクル全体を一貫して支援する。ただし、すべての文書作成を強制せず、プロジェクトの目的、規模、リスク、公共性、運用責任に応じて、必要な成果物を選択できるようにする。

- 特に、シビックテック、地域活動、教育、福祉、行政、NPO、OSSなど、公共性や継続性が重要なプロジェクトにおいて、市民、専門家、行政、開発者、AI Agentが協働しやすい共通の型を提供する。

- SpecDojo をオープンソースとして公開し、誰もが自由に利用・改善・再利用できる共通基盤として育てる。

### 4.1. 対象利用者と利用場面

| 対象利用者                                                          | 解決したい課題                                                   | 主な利用場面                                                 |
| ------------------------------------------------------------------- | ---------------------------------------------------------------- | ------------------------------------------------------------ |
| プロジェクトオーナー、業務担当者、Business Analyst                  | 目的、利用者、業務課題、期待効果、受入観点を関係者と共有したい   | 新規プロジェクトの立ち上げ、要求整理、スコープ合意、変更判断 |
| Architect、開発者、Quality Engineer、運用担当者                     | 上流の判断根拠と後続の設計・実装・テスト・運用成果物を追跡したい | 成果物作成、レビュー、影響分析、引き継ぎ、継続改善           |
| 小規模組織、地域コミュニティ、行政、教育機関、NPO、OSS プロジェクト | 限られた人員でも、公開・再利用可能な型を使って活動を継続したい   | 最小構成の選択、サンプル参照、公開、外部協働                 |
| AI Agent                                                            | 明示された目的、制約、入力、出力形式に基づいて作業を支援したい   | 草案作成、整合性確認、レビュー支援、更新候補の提示           |

AI Agent は利用主体の一つであるが、目的、優先順位、受入可否、公開可否の最終判断は人間が行う。

### 4.2. 初期スコープの概要

初期スコープは、人と AI Agent が同じ文書体系を参照し、仕様駆動プロジェクトの主要成果物を作成・レビュー・改善できる状態を成立させることに置く。

| 区分   | 概要                                                                                                                                                                                                                                      |
| ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 対象   | 文書分類・配置・命名・メタデータ、standards・guides・rulebooks・recipes・samples・templates、プロジェクト管理文書、代表的なサンプル、Schedule・実行管理・文書検証を支援する補助ツール、README・LICENSE・Issue / Pull Request 等の公開導線 |
| 対象外 | 特定業務システムの本番開発、特定の開発手法・言語・クラウド・生成 AI 製品への最適化、全文書種別の初期完成保証、有償サービスや本格的な利用者サポートの運営、個別組織の機密情報の収集                                                        |

詳細な対象範囲、対象期間、境界判断、変更方針は [プロジェクトスコープ](./prj-scope.md) を正とする。

### 4.3. 本書で合意する事項

本書では、後続計画の前提として次の事項を合意対象とする。

- プロジェクトの目的を「人と AI Agent が共有・更新できる仕様体系と運用基盤の整備」とすること
- 主な利用者、利用場面、初期スコープ、対象外の方向性
- 文書体系・ルールを中核とし、サンプルと補助ツールを段階的に組み合わせる方針
- OSS として公開・再利用し、利用実績とフィードバックに基づき継続改善する方針
- 初期公開では全文書の完成ではなく、主要成果物を作成・レビューできる状態を優先すること

正式な立ち上げ認可と権限委譲は [プロジェクト憲章](./prj-charter.md)、詳細な受入条件と GO / Not GO 判断は [成功基準と受入条件](./prj-success-criteria-and-acceptance-criteria.md) を正とする。本書の合意だけでは、本格実行開始または外部公開を認可したことにはならない。

## 5. 期待効果

SpecDojo の整備により、仕様駆動プロジェクト運営を実践可能にし、課題解決型プロジェクトの成功確率、実行速度、透明性、継続性を高めることを期待する。

本プロジェクトの短期的な成功は、SpecDojo 自体の初期公開に必要な仕様体系、文書体系、ルール、サンプル、AI Agent 向け指示、管理基盤が整い、人と AI Agent が同じ仕様と判断根拠を参照して作業できる状態になることで判定する。外部プロジェクトの成果や社会課題への影響は、公開後に継続的に確認する中長期的な効果として扱う。

下記の目標値は、AI Agent を前提とした仕様駆動プロジェクト運営が十分に機能した場合の初期仮説であり、実測値に基づき継続的に見直す。初期公開時点では達成度を判定し、工数・期間は実績値を記録する。削減率や短縮率は、同等規模の仕様整備・改善実績をベースラインとして確定した後に判定する。

### 5.1. 短期指標

短期的には、SpecDojo 自体の初期公開に向けた QCD への効果を確認する。

| 観点     | 指標                                     | 目標            | 測定方法                                                                         |
| -------- | ---------------------------------------- | --------------- | -------------------------------------------------------------------------------- |
| Quality  | 初期公開に必要な成功基準の達成度         | 90%以上         | 成功基準と受入条件で定めた項目に対して、初期公開時点で実現できた項目の割合を確認 |
| Cost     | 同等規模の仕様整備・改善に必要な作業工数 | 次回以降50%削減 | 初期公開時は実績値を記録し、ベースライン確定後に同等規模タスクと比較             |
| Delivery | 企画開始から初期公開可能状態までの期間   | 次回以降50%短縮 | 初期公開時は実績値を記録し、ベースライン確定後に同等規模タスクと比較             |

### 5.2. 中長期指標

中長期的には、公開後の波及効果を確認する。公開直後は母数が小さいため、まず累積件数を確認し、導入事例が一定数に達した後に年率増加などの成長指標へ移行する。

| 観点           | 指標                                                           | 初期目標     | 測定方法                                                         |
| -------------- | -------------------------------------------------------------- | ------------ | ---------------------------------------------------------------- |
| 社会課題の解消 | 社会課題の解消・軽減につながるサービスやシステムの継続運用件数 | 累積3件以上  | 活用事例、導入報告、運用状況、利用者フィードバック等から確認     |
| 継続性         | 価値あるプロジェクトが一定期間後も継続運用・改善されている割合 | 70%以上      | 一定期間後のリリース、Issue対応、保守、文書更新状況を確認        |
| 参加機会       | 非エンジニア・市民参加者が成果物更新や改善提案に関与した件数   | 累積10件以上 | 課題定義、要件、運用ルール、議事録、判断記録、改善提案件数を集計 |
| 公共性・透明性 | 判断根拠、仕様、変更履歴を公開しているプロジェクトの割合       | 70%以上      | 公開リポジトリ、公開文書、事例記事、導入報告等から確認           |

参考指標として、SpecDojo の導入プロジェクト数、テンプレート再利用件数、公開事例数、外部貢献件数を継続的に確認する。

### 5.3. 成功判定の輪郭

初期整備の成功は、少なくとも次の状態が確認できることを輪郭とする。

- 主要な文書体系と各参考資料の役割を説明できる
- 優先度の高い成果物を、人と AI Agent が参考資料に基づいて作成・レビューできる
- スコープ、成功基準、前提・制約、管理計画、体制、Schedule が接続されている
- lint とレビューにより重大な文書構造不備を検出できる
- 公開対象に機密情報を含めず、README、LICENSE、Issue / Pull Request 等の公開・改善導線を確認できる

短期指標の達成率は進捗と効果を測る指標であり、必須受入条件の未達を許容する基準ではない。具体的な判定基準、証跡、判定時期、責任者は [成功基準と受入条件](./prj-success-criteria-and-acceptance-criteria.md) で管理する。

## 6. 前提条件

SpecDojo の策定および活用にあたっては、以下を前提とする。

- SpecDojo は、人間の判断や責任を代替するものではなく、人と AI Agent が協働するための基盤であること
- AI Agentは、上流工程の意思決定者ではなく、課題、目的、選択肢、判断根拠を整理し、人間の意思決定を支援する存在であること
- ドキュメントは、人間にとっての理解容易性と、AI Agentにとっての解釈容易性を両立すること
- SpecDojo は、特定の開発手法、言語、基盤、生成AI製品に過度に依存しないこと
- 各プロジェクトは、目的、規模、リスク、公共性、運用責任に応じて、SpecDojo の成果物体系を取捨選択して適用すること (詳細なスコープ判断は [プロジェクトスコープ](./prj-scope.md) を正とする）
- SpecDojo は、オープンソースとして公開し、実際の活用事例や各現場の知見を取り込みながら継続的に改善すること

### 6.1. 技術・運用上の前提と制約

| 区分       | 内容                                                                                          | プロジェクトへの影響                                                                                                                                                                                   |
| ---------- | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 文書形式   | 人と AI Agent の双方が参照しやすい Markdown、YAML、JSON を中心に、Git で変更履歴を管理する    | 文書は差分確認と機械検証が可能な構造を優先する                                                                                                                                                         |
| 補助ツール | Node.js / npm で実行できる CLI、lint、スキーマ検証、Schedule・実行管理を補助に用いる          | ツールが利用できない場合でも文書自体を参照できる構成を維持する。初期段階では lint とレビューを品質確認の必須手段とする ([前提・制約・依存関係](./prj-assumptions-constraints-dependencies.md) ACD-202) |
| 公開基盤   | GitHub リポジトリと GitHub Pages を主な公開・参照導線とする                                   | URL、公開設定、外部サービス仕様の変更を継続的に確認する ([前提・制約・依存関係](./prj-assumptions-constraints-dependencies.md) ACD-201)                                                                |
| 技術中立性 | 特定の開発手法、言語、クラウド、生成 AI 製品を必須条件にしない                                | 製品固有の内容は例示または利用プロジェクト側の文書に分離する ([前提・制約・依存関係](./prj-assumptions-constraints-dependencies.md) ACD-101)                                                           |
| 公開適性   | 公開リポジトリに個人情報、非公開情報、認証情報、機密情報を含めない                            | 公開前レビューと差分確認を必須とする ([前提・制約・依存関係](./prj-assumptions-constraints-dependencies.md) ACD-102)                                                                                   |
| 実行体制   | 初期段階は個人・小規模運用を前提にする。管理責務 `PO`、公開判断は最終的に人間が行うこととする | 全成果物の同時完成を目指さず、優先順位と段階導入で負荷を制御する ([前提・制約・依存関係](./prj-assumptions-constraints-dependencies.md) ACD-004, ACD-104)                                              |

詳細な監視方法、変更トリガー、対応責任は [前提・制約・依存関係](./prj-assumptions-constraints-dependencies.md) を正とする。
