# 前提・制約・依存関係

## 1. 前提条件

| ID      | 種別 | 内容                                                                            | 影響                                           | 監視方法                               | トリガー                                     | 責任者     | 対応方針                                 |
| ------- | ---- | ------------------------------------------------------------------------------- | ---------------------------------------------- | -------------------------------------- | -------------------------------------------- | ---------- | ---------------------------------------- |
| ACD-001 | 前提 | SpecDojo は人間の判断を代替せず、人と AI が協働するための文書基盤として整備する | 責任境界が曖昧になると品質・公開判断がぶれる   | 管理計画、RACI、参考資料の確認 | Agent に承認や最終判断を委ねる記述が出た場合 | `PO`, `QE` | 記述を修正し、判断責任を `PO` に戻す     |
| ACD-002 | 前提 | 対象範囲はシステムライフサイクル全体を想定する                                  | 文書体系が一部工程に偏る可能性がある           | 成果物カタログと rulebook 一覧の確認   | 主要工程の文書が欠落した場合                 | `ARC`      | 優先順位を付けて不足文書を登録する       |
| ACD-003 | 前提 | オープンソースとして公開し、自由に利用・改善・再利用できる共通基盤とする        | 公開不適切情報やライセンス未確定が公開を妨げる | 公開前レビュー、LICENSE、README の確認 | 公開対象に未判断情報が含まれる場合           | `PO`, `QE` | 公開対象から除外または修正する           |
| ACD-004 | 前提 | 初期段階では個人・小規模運用で進める                                            | 作業負荷が集中し、継続性が下がる               | Schedule、未完了項目の確認        | 高優先度項目が継続的に滞留する場合           | `PO`       | スコープを調整し、外部参加導線を検討する |

## 2. 制約事項

| ID      | 種別 | 内容                                                             | 影響                             | 監視方法                               | トリガー                                 | 責任者      | 対応方針                                 |
| ------- | ---- | ---------------------------------------------------------------- | -------------------------------- | -------------------------------------- | ---------------------------------------- | ----------- | ---------------------------------------- |
| ACD-101 | 制約 | 特定の開発手法、言語、基盤、生成 AI 製品に過度に依存しない       | 再利用性が下がる                 | rulebook、guideline、sample の記述確認 | 特定製品前提の必須記述が増えた場合       | `ARC`, `QE` | 汎用表現へ修正し、必要なら例示扱いにする |
| ACD-102 | 制約 | 公開リポジトリには個人情報、非公開情報、機密情報を記載しない     | 公開停止や信頼低下につながる     | 公開前レビュー、差分確認               | 個人名、連絡先、非公開情報が含まれる場合 | `PO`, `QE`  | 削除、匿名化、公開対象外化を行う         |
| ACD-103 | 制約 | 初期公開では全成果物の完成ではなく、主要文書体系の成立を優先する | 完成範囲の期待値がぶれる         | 成功基準、成果物カタログ、登録簿の確認 | 未整備文書の扱いが曖昧な場合             | `PO`, `ARC` | 未整備項目を登録簿または WBS で管理する  |
| ACD-104 | 制約 | 管理責務と公開判断は `PO` に集約する                             | 複数人運用時に判断負荷が集中する | pm-organization、pm-raci の確認        | 外部参加者や複数人運用が始まる場合       | `PO`        | `PM`、`OPS` 等の採用を再検討する         |

## 3. 依存関係

| ID      | 種別 | 内容                                                              | 影響                             | 監視方法                                | トリガー                           | 責任者      | 対応方針                                 |
| ------- | ---- | ----------------------------------------------------------------- | -------------------------------- | --------------------------------------- | ---------------------------------- | ----------- | ---------------------------------------- |
| ACD-201 | 依存 | GitHub リポジトリと GitHub Pages に公開・参照導線を依存する       | 公開方法や参照性に影響する       | README、Pages 設定、リンク確認          | 公開先やURLが変わった場合          | `ARC`, `PO` | README、リンク、公開手順を更新する       |
| ACD-202 | 依存 | Markdown lint とローカル補助ツールに品質確認を依存する            | 品質判定の再現性に影響する       | lint 実行結果、ツール実行結果の確認     | lint やツールが失敗する場合        | `QE`, `ARC` | 失敗原因を修正し、必要ならルールを見直す |
| ACD-203 | 依存 | AI Agent の出力品質は入力文書と参考資料の明確さに依存する | 生成物のばらつきや誤解釈が増える | rulebook、recipe、sample、template、レビュー結果の確認 | 出力の不整合や過剰補完が増えた場合 | `BA`, `QE`  | 参考資料を改善する     |
| ACD-204 | 依存 | 利用者・貢献者からの改善は Issue / PR 導線に依存する              | 外部改善が集まりにくくなる       | README、CONTRIBUTING、Issue / PR の確認 | 外部受付開始時                     | `PO`, `BA`  | 最小の貢献ルールとレビュー方針を整備する |

## 4. 影響評価と対応方針

| 種別                 | 主な影響領域                          | 対応方針                                                     |
| -------------------- | ------------------------------------- | ------------------------------------------------------------ |
| 前提が崩れた場合     | スコープ、品質、公開判断、責任分担    | プロジェクト登録簿へ記録し、必要に応じて変更要求として扱う   |
| 制約に抵触した場合   | 公開可否、ライセンス、再利用性        | 該当成果物を修正または公開対象から除外する                   |
| 依存先が変化した場合 | URL、ツール実行、生成物、外部参加導線 | 影響範囲を確認し、README、成果物カタログ、管理計画を更新する |

## 5. 監視・変更管理

- 本書は、スコープ、管理計画、成果物カタログ、公開方針を変更したときに見直す。
- 高影響の項目は、プロジェクト登録簿、リスク登録簿、変更要求ログに接続する。
- 前提、制約、依存関係に変更があった場合は、成功基準、Schedule、RACI への影響を確認する。
