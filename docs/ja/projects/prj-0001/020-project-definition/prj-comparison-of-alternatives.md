# 代替案比較: SpecDojo

## 1. 比較目的と前提条件

本書は、SpecDojo の初期整備アプローチについて、複数の代替案を比較し、採択理由を説明可能な形で残すための文書である。

比較対象は、文書体系、ルール、サンプル、AI 向け指示、補助ツール、公開導線をどの順序・比重で整備するかである。

本書の根拠資料は次のとおり。比較対象案・評価軸・採択理由は、これらの決定事項と整合させる。

- [[prj-0001:prj-issues-and-approach|プロジェクト課題と解決アプローチ]]: 解決策候補（A-01〜A-04）と採用アプローチを引き継ぎ、本書で評価軸により再評価する。
- [[prj-0001:prj-scope|プロジェクトスコープ]]: 比較対象範囲と境界の判断基準。

前提条件は次のとおり。

- SpecDojo は、仕様駆動開発を支えるオープンなドキュメントフレームワークとして整備する。
- 初期段階では個人・小規模運用を前提とする。
- 特定技術や特定組織に過度に依存しない。
- 人と AI Agent の双方が参照・再利用できることを重視する。

## 2. 比較対象案

| 案ID   | 案                         | 概要                                                         | 対象範囲                                          |
| ------ | -------------------------- | ------------------------------------------------------------ | ------------------------------------------------- |
| ALT-01 | 文書体系・ルールブック優先 | 成果物分類、構成、命名、メタデータ、rulebook を先に整備する  | standards、guides、rulebooks、成果物カタログ      |
| ALT-02 | サンプル・導入体験優先     | 利用者が真似できるサンプル、README、導入手順を先に整備する   | samples、README、導入ガイド、サンプルプロジェクト |
| ALT-03 | 補助ツール・自動化優先     | スケジュール、生成物、lint、チェックを先に整備する           | specdojo コマンド、generated、実行管理            |
| ALT-04 | 特定プロジェクト実践優先   | 特定のサンプルまたは実プロジェクトを進めながら標準を抽出する | サンプルプロジェクト、派生ルール、実例            |

`ALT-01`〜`ALT-04` は、[[prj-0001:prj-issues-and-approach|プロジェクト課題と解決アプローチ]] の `A-01`〜`A-04` を比較用の粒度にそろえて再掲したものである。

## 3. 評価軸と評価基準

| 評価軸     | 重み | 判定基準                                                                   | 備考                     |
| ---------- | ---- | -------------------------------------------------------------------------- | ------------------------ |
| 効果       | 5    | SpecDojo の目的である仕様駆動開発の再利用性向上にどれだけ効くか            | 最重要                   |
| コスト     | 3    | 個人・小規模運用で実行可能な作業量か                                       | 低コストな案ほど高評価   |
| 期間       | 3    | 初回公開または GO / Not GO 判断までに成果が出るか                          | 短期間で進む案ほど高評価 |
| リスク     | 4    | 手戻り、属人化、公開不備、品質低下のリスクが低いか                         | 低リスクな案ほど高評価   |
| 運用適合   | 4    | OSS 公開後の継続改善、Issue / PR、AI Agent 利用に適合するか                | 高いほど高評価           |
| 技術実現性 | 4    | 特定技術へ過度に依存せず、文書体系・参考資料・補助ツールとして実現できるか | 高いほど高評価           |

評価は `High` / `Middle` / `Low` で行い、いずれも「その評価軸で採択判断に有利か」を示す。総合評価は採択判断に必要な説明として扱い、機械的な加重合計ではなく、根拠資料との整合と初期公開への影響を踏まえて判断する。

## 4. 比較結果と採択理由

| 案ID   | 効果   | コスト | 期間   | リスク | 運用適合 | 技術実現性 | 総合評価 | 判定     |
| ------ | ------ | ------ | ------ | ------ | -------- | ---------- | -------- | -------- |
| ALT-01 | High   | Middle | Middle | High   | High     | High       | High     | 採択     |
| ALT-02 | Middle | Middle | High   | Middle | High     | Middle     | Middle   | 一部採択 |
| ALT-03 | Middle | Low    | Middle | Low    | Middle   | Middle     | Middle   | 一部採択 |
| ALT-04 | Middle | Low    | Low    | Low    | Middle   | Low        | Low      | 非採択   |

採択案は ALT-01 とし、ALT-02 と ALT-03 を補完的に組み合わせる。

採択理由:

- SpecDojo の中核は、特定プロジェクトではなく、再利用可能な文書体系と作成ルールである。
- rulebook を先に整備することで、recipe、sample、template、補助ツールの品質基準をそろえやすい。
- 成果物カタログ、スケジュール、品質管理と接続しやすく、AI Agent の作業入力としても安定する。
- 特定技術や特定組織へ固定せず、Markdown、YAML、生成ビュー、lint など既存の文書運用で段階的に実現できる。

技術的実現性と影響:

技術的な外部依存は、初期段階では Markdown、YAML、既存の lint / format、生成ビューに留め、特定クラウド、特定業務システム、非公開組織情報には依存しない方針とする。後続成果物への影響は、文書体系、rulebook / recipe / sample / template、補助ツールの整備順に現れる。

| 案ID   | 技術的実現性                                                                                 | 主な影響                                                                               |
| ------ | -------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| ALT-01 | 既存の文書配置、frontmatter、rulebook、成果物カタログを軸に進められる                        | 文書構造の整合が先に安定し、後続の sample / template / 自動化の判断基準になる          |
| ALT-02 | 既存 sample を拡張して進められるが、基準が薄いとサンプル間の粒度が揺れる                     | 導入体験は早く改善する一方、rulebook との差分管理が必要になる                          |
| ALT-03 | lint、生成物、スケジュールとの接続で実現できるが、ルール未確定部分の自動化は手戻りが出やすい | 整合確認は効率化する一方、先に自動化範囲を固定すると文書体系変更への追従コストが増える |
| ALT-04 | 実プロジェクトの知見抽出は可能だが、汎用化の再設計が必要になる                               | 具体性は高まる一方、公開・再利用できる一般形への変換負荷が大きい                       |

非採択または一部採択の理由:

| 案ID   | 理由                                                             | 再評価条件                                          |
| ------ | ---------------------------------------------------------------- | --------------------------------------------------- |
| ALT-02 | サンプルだけでは判断基準が残らず、文書体系の一貫性が崩れやすい   | 初回利用者向け導入体験を強化する段階で比重を上げる  |
| ALT-03 | ツール優先にすると、ルール未確定のまま自動化し、手戻りが増える   | スケジュール / 生成物の運用が安定した段階で強化する |
| ALT-04 | 特定事例に寄りすぎると、汎用フレームワークとしての説明力が下がる | 代表事例や導入事例を増やす段階で再評価する          |

## 5. リスクとトレードオフ

| 区分         | 内容                                                     | 軽減策                                                       |
| ------------ | -------------------------------------------------------- | ------------------------------------------------------------ |
| トレードオフ | ルール優先により、初期の利用体験が硬く見える可能性がある | 主要文書には recipe、sample、template を必要に応じて併設する |
| トレードオフ | ツール整備が後回しになり、手作業確認が残る               | 品質管理計画で lint とレビューを必須化する                   |
| リスク       | ルールが増えすぎて導入負荷が高くなる                     | 最小構成、サンプル、導入ガイドを整備する                     |
| リスク       | 特定技術に依存しない方針により、具体性が不足する         | サンプルでは具体例を示し、ルール本文では汎用性を保つ         |

案ごとのリスクとトレードオフ:

| 案ID   | 主なリスク / トレードオフ                                          | 軽減策 / 扱い                                                                         |
| ------ | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------- |
| ALT-01 | 利用者が完成イメージをつかむ前に、規約理解の負荷が先に立つ         | 主要成果物から recipe、sample、template を段階的に併設し、導入導線を補う              |
| ALT-02 | サンプルが先行し、rulebook と成果物カタログの基準から外れる        | サンプルは rulebook の派生物として管理し、基準未整備の領域は `_TODO_` 化する          |
| ALT-03 | 自動化対象を早期に固定し、文書体系変更時の手戻りが増える           | 初期は format / lint / 生成ビューなど低依存の範囲に絞り、判定ロジックは後続で拡張する |
| ALT-04 | 特定事例の事情が混入し、汎用フレームワークとしての再利用性が下がる | 実例はサンプルとして分離し、標準本文には一般化した判断基準のみ反映する                |

## 6. 決定と見直し

- 現時点の採択方針は「文書体系・ルールブック優先、サンプルと補助ツールを段階的に補完」とする。
- 本比較は採択判断の支援であり、採択案の最終判断は人間の `PO` が行う。AI Agent は比較分析の支援に留める。
- 初回公開後、利用者フィードバック、Issue / PR、導入事例を踏まえて、サンプル優先またはツール優先への比重変更を再評価する。
- 方針を変更する場合は、プロジェクト登録簿または決定記録に判断理由を残す。
