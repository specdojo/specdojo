# 代替案比較: SpecDojo

## 1. 比較目的と前提条件

本書は、SpecDojo の初期整備アプローチについて、複数の代替案を比較し、採択理由を説明可能な形で残すための文書である。

比較対象は、Why 起点の合意形成、仕様の共有知識基盤化、必要な地点からの継続的な育成、共通の型・補助手段の整備、固定工程による標準化を、初期整備でどう組み合わせて実践するかである。

本書の根拠資料は次のとおり。比較対象案・評価軸・採択理由は、これらの決定事項と整合させる。

- [[prj-0001:prj-issues-and-approach|プロジェクト課題と解決アプローチ]]: 解決策候補（A-01〜A-05）と採用アプローチを引き継ぎ、本書で評価軸により再評価する。
- [[prj-0001:prj-scope|プロジェクトスコープ]]: 比較対象範囲と境界の判断基準。

前提条件は次のとおり。

- SpecDojo は、仕様駆動開発を支えるオープンなドキュメントフレームワークとして整備する。
- 初期段階では個人・小規模運用を前提とする。
- 特定技術や特定組織に過度に依存しない。
- 人と AI Agent の双方が参照・再利用できることを重視する。
- 初期公開では、外部 SaaS、行政システム、個別団体の業務システムとのデータ連携や認証連携は対象外とする。

## 3. 評価軸と評価基準

| 評価軸     | 重み | 判定基準                                                                   | 備考                     |
| ---------- | ---- | -------------------------------------------------------------------------- | ------------------------ |
| 効果       | 5    | SpecDojo の目的である仕様駆動開発の再利用性向上にどれだけ効くか            | 最重要                   |
| コスト     | 3    | 個人・小規模運用で実行可能な作業量か                                       | 低コストな案ほど高評価   |
| 期間       | 3    | 初回公開または GO / Not GO 判断までに成果が出るか                          | 短期間で進む案ほど高評価 |
| リスク     | 4    | 手戻り、属人化、公開不備、品質低下のリスクが低いか                         | 低リスクな案ほど高評価   |
| 運用適合   | 4    | OSS 公開後の継続改善、Issue / PR、AI Agent 利用に適合するか                | 高いほど高評価           |
| 技術実現性 | 4    | 特定技術へ過度に依存せず、文書体系・参考資料・補助ツールとして実現できるか | 高いほど高評価           |

評価は `High` / `Middle` / `Low` で行い、いずれも「その評価軸で採択判断に有利か」を示す。総合評価は採択判断に必要な説明として扱い、機械的な加重合計ではなく、根拠資料との整合と初期公開への影響を踏まえて判断する。

評価軸は、[[prj-0001:prj-issues-and-approach|プロジェクト課題と解決アプローチ]] の主要課題に対応させる。効果は、各案が同書で「主に対応する課題」として挙げる課題 ID（`ALT-01`: `P-01`、`P-02`、`P-06`／`ALT-02`: `P-02`〜`P-05`、`P-07`／`ALT-03`: `P-03`〜`P-06`／`ALT-04`: `P-01`〜`P-07`／`ALT-05`: `P-01`〜`P-05`）への寄与度を見る。リスクと運用適合は、`P-05`（変更影響の追跡）と `P-07`（AI Agent の活用）を阻害しないかを重視する。技術実現性は、[[prj-0001:prj-scope|プロジェクトスコープ]] で定義した文書フレームワーク、補助ツール、公開・再利用の導線の範囲で成立するかを評価する。

## 4. 比較結果と採択理由

| 案ID   | 効果   | コスト | 期間   | リスク | 運用適合 | 技術実現性 | 総合評価 | 判定     |
| ------ | ------ | ------ | ------ | ------ | -------- | ---------- | -------- | -------- |
| ALT-01 | High   | Middle | Middle | Middle | High     | High       | High     | 採択     |
| ALT-02 | High   | Middle | Middle | High   | High     | High       | High     | 採択     |
| ALT-03 | High   | Low    | Middle | Middle | High     | High       | High     | 採択     |
| ALT-04 | Middle | Low    | Middle | Low    | Middle   | Middle     | Middle   | 一部採択 |
| ALT-05 | Low    | Middle | High   | Low    | Low      | Middle     | Low      | 非採択   |

採択案は ALT-01、ALT-02、ALT-03 を一体の基本方針として採択し、ALT-04 の共通の型と補助手段によって段階的に実践可能にする。

採択理由:

- SpecDojo の中心課題は、プロジェクトの知識や判断を継続的に共有・更新できる形で言語化することが難しい点にあり、Why の合意形成（ALT-01）と仕様の共有知識基盤化（ALT-02）を同時に満たす必要がある。
- 個人・小規模運用を前提とすると、全成果物を固定工程で揃えるより、必要な地点から始めて継続的に育てる（ALT-03）方が現実的であり、担い手への負荷集中も避けやすい。
- 共通の型と補助手段（ALT-04）は三原則の実践を再現可能にするために必要だが、整備自体を目的化しないよう、本質課題への寄与と初期公開の優先度に応じて段階的に導入する。
- 全文書種別の完成よりも、初期公開に必要な文書体系、主要ルール、主要サンプル、管理計画、公開導線の成立を優先できる。
- 技術的な外部依存を Markdown、YAML、既存の lint / format、生成ビューに留め、特定技術や特定組織へ固定しない方針と両立できる。

技術的実現性と影響:

技術的な外部依存は、初期段階では Markdown、YAML、既存の lint / format、生成ビューに留め、特定クラウド、特定業務システム、非公開組織情報には依存しない方針とする。後続成果物への影響は、Why の合意形成、仕様の共有知識基盤化、必要な地点からの着手、共通の型と補助手段の整備順に現れる。

| 案ID   | 技術的実現性                                                                                                | 主な影響                                                                               |
| ------ | ----------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| ALT-01 | 既存のプロジェクト定義文書のテンプレートと合意記録の枠組みで進められる                                      | Why と課題への合意が先に安定し、後続の文書体系・成果物選択の判断基準になる             |
| ALT-02 | 既存の文書配置、frontmatter、rulebook、成果物カタログを軸に進められる                                       | 文書構造と参照関係が先に安定し、後続の sample / template / 自動化の判断基準になる      |
| ALT-03 | 既存 sample やテンプレートを流用しつつ、着手順序と選択条件を rulebook 側で明示すれば実現できる              | 成果物ごとの選択制が定着する一方、選択基準が曖昧だと着手順序が案件ごとにばらつく       |
| ALT-04 | lint、生成物、recipe / sample / template との接続で実現できるが、ルール未確定部分の自動化は手戻りが出やすい | 整合確認は効率化する一方、先に自動化範囲を固定すると文書体系変更への追従コストが増える |
| ALT-05 | 既存の文書配置と成果物カタログを軸に、作成順序をチェックリスト化すれば実現できる                            | 開始手順と完了条件は統一しやすいが、案件に応じた選択と途中地点からの改善が難しくなる   |

非採択または一部採択の理由:

| 案ID   | 扱い     | 理由                                                                                                           | 再評価条件                                                               |
| ------ | -------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| ALT-04 | 一部採択 | 補助手段の整備を先行させると、Why や仕様の合意が固まる前に型が固定され、本質課題との対応が弱くなるおそれがある | 主要成果物の rulebook と参照関係が安定した段階で比重を上げる             |
| ALT-05 | 非採択   | 目的・規模・リスクに応じた成果物選択と、任意の地点から始めて育てる方針（ALT-03）に反するため                   | 初回公開後、固定順序を必要とする利用者要望が明確になった段階で再評価する |

## 5. リスクとトレードオフ

| 区分         | 内容                                                         | 軽減策                                                             |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------------ |
| トレードオフ | Why の合意形成を先行させるため、初期段階の実装着手が遅くなる | 合意形成の範囲を必要最小限にし、並行して着手可能な成果物から進める |
| トレードオフ | 初期公開では全文書種別の完成を保証しない                     | 未整備項目を成果物カタログと後続作業で管理する                     |
| リスク       | 合意形成や仕様の共有知識基盤化が形骸化し、文書量だけが増える | 各成果物が P-01〜P-07 のどの課題に寄与するか確認する               |
| リスク       | 特定技術に依存しない方針により、具体性が不足する             | サンプルでは具体例を示し、ルール本文では汎用性を保つ               |

案ごとのリスクとトレードオフ:

| 案ID   | 主なリスク / トレードオフ                                                        | 軽減策 / 扱い                                                                         |
| ------ | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| ALT-01 | 合意形成に時間をかけすぎると、着手が遅れる                                       | 合意対象を「何が課題で、どれを、どう解く方針か」に絞り、詳細設計は後続文書に委ねる    |
| ALT-02 | 仕様体系の整備が先行し、利用者が完成イメージをつかむ前に規約理解の負荷が先に立つ | 主要成果物から recipe、sample、template を段階的に併設し、導入導線を補う              |
| ALT-03 | 着手順序や選択基準が曖昧だと、案件ごとに整備の粒度がばらつく                     | rulebook と成果物カタログで選択条件と最小構成を明示する                               |
| ALT-04 | 自動化対象を早期に固定し、文書体系変更時の手戻りが増える                         | 初期は format / lint / 生成ビューなど低依存の範囲に絞り、判定ロジックは後続で拡張する |
| ALT-05 | 固定順序が実情に合わず、案件に応じた選択や途中地点からの改善を妨げる             | 段階採用はせず、初期公開後の要望に応じて限定的な範囲でのみ再評価する                  |

## 6. 決定と見直し

- 現時点の採択方針は「Why 起点の合意形成と仕様の共有知識基盤化を中核とし、必要な地点から始めて継続的に育てながら、共通の型と補助手段を段階的に整備する」とする。
- 本比較は採択判断の支援であり、採択案の最終判断は人間の `PO` が行う。AI Agent は比較分析の支援に留める。
- 初回公開後、利用者フィードバック、Issue / PR、導入事例を踏まえて、共通の型と補助手段（ALT-04）への比重変更を再評価する。
- 方針を変更する場合は、プロジェクト登録簿または決定記録に判断理由を残す。
