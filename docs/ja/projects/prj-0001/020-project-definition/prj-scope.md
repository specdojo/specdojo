# プロジェクトスコープ: SpecDojo

本書は [[prj-0001:prj-overview|プロジェクト概要: SpecDojo]] で示した背景、必要性、実現したいことを受け、SpecDojo の初期公開に含める活動領域、成果物群、公開準備、対象外の境界を定義する。

## 1. 対象業務

本プロジェクトでは、仕様駆動プロジェクト運営を実践・再利用するために必要な文書体系と運用基盤の整備を対象とする。主な利用者は、プロジェクトの立ち上げ・改善・運用を行う市民、専門家、行政、開発者、AI Agent とし、各利用者が同じ仕様と判断根拠を参照できる状態を目指す。

| 利用者             | 利用場面                                                     | 利用者影響                                                     |
| ------------------ | ------------------------------------------------------------ | -------------------------------------------------------------- |
| 市民、専門家、行政 | 地域・公共性のあるプロジェクトの立ち上げ、合意形成、引き継ぎ | 目的、判断根拠、必要な成果物を共有し、担当交代後も参照できる   |
| 開発者             | サービスやシステムの構築、改修、運用・保守                   | 要件から運用までの成果物を選択し、変更影響を追跡できる         |
| AI Agent           | 文書の参照、タスク実行の支援、成果物の改善                   | 人間と同じ仕様体系を解釈し、判断根拠を参照して作業を支援できる |

- システムライフサイクル全体を支える文書分類、配置、命名、メタデータ、派生関係の整理。
- 初期公開に必要な成果物の rulebook、recipe、sample、template の整備。
- プロジェクト定義、プロジェクトマネジメント、設計、実装、テスト、運用・保守に関する初期公開向け文書テンプレートの整備。
- 人と AI Agent が同じ文書を参照して作業できるようにするための記述方針、品質基準、運用ルールの整備。
- SpecDojo 自体のプロジェクト管理、成果物カタログ、Schedule、実行管理の整備。
- 公開リポジトリとして利用・改善・再利用しやすい README、ライセンス、貢献導線、公開範囲の整理。

本スコープで利用者に影響する範囲は、文書の作成・参照・レビュー・改善に関わる導線である。個別プロジェクトの業務手順そのものを変更するのではなく、各プロジェクトが必要な成果物を選択し、合意形成と引き継ぎに使える型を提供する。

本書の合意対象は、SpecDojo の初期公開に含める活動領域、成果物群、公開準備、対象外の境界である。PO は対象範囲と対象外の承認に、ARC は外部連携や補助ツールを含む技術的境界の確認に、PM は初期公開までの計画前提の確認に本書を用いる。個別成果物ごとの詳細な受入条件、設計、実装、テスト手順は後続文書で扱う。

## 2. 対象システム

本プロジェクトの対象は、アプリケーションシステムそのものではなく、仕様駆動開発を支える文書フレームワークと補助ツール群である。

| 対象                 | 内容                                                                                              | 境界                                                   |
| -------------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------ |
| 文書フレームワーク   | 文書分類、配置、命名、メタデータ、派生関係、rulebook、recipe、sample、template、AI Agent 向け指示 | 個別プロジェクトの業務システムや本番データは含めない   |
| プロジェクト運営文書 | プロジェクト定義、管理計画、責任分担、スケジュール、実行管理のための成果物                        | 個別組織の内部手順そのものは含めない                   |
| 補助ツールと生成物   | 文書の参照、生成、実行管理、整合確認を支援するコマンド、管理基盤、生成物                          | 特定の業務を実行するアプリケーション機能は含めない     |
| 公開・再利用の導線   | オープンソースとして利用、改善、再利用するための説明、ライセンス、貢献導線                        | 有償サービスの提供や外部サービスとの業務連携は含めない |

外部連携は、公開リポジトリ上で文書、サンプル、補助ツールを利用・改善できる範囲に限る。初期公開では、外部 SaaS、行政システム、個別団体の業務システムとのデータ連携や認証連携は対象外とする。

## 3. 対象期間

- 初期対象期間は、SpecDojo の初回公開に必要な仕様体系、文書体系、ルール、サンプル、AI Agent 向け指示、管理基盤、公開導線が整うまでとする。
- 本格実行開始の GO / Not GO 判断は、プロジェクト定義文書とプロジェクトマネジメント文書が整備された時点で行う。
- 初回公開後の改善、利用者フィードバック対応、外部貢献受付、事例追加は継続改善フェーズとして扱う。
- 具体的な初回公開日は `_TODO_`: 初期公開計画で決定する。

## 4. スコープ外

| 対象（Out）                            | 理由                                                                                       | 補足                                                   |
| -------------------------------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------ |
| 特定業務システムの本番開発             | SpecDojo は文書フレームワークであり、特定システムの実装は利用プロジェクト側の責務とする    | サンプル実装や例示は必要最小限に留める                 |
| 特定の開発手法・言語・基盤への最適化   | 幅広いプロジェクトで再利用できる共通の型を提供するため                                     | 個別技術の詳細は利用プロジェクト側で扱う               |
| 人間の判断や責任の代替                 | SpecDojo と AI Agent は、課題、選択肢、判断根拠の整理を支援する位置付けのため              | 意思決定と責任は人間が担う                             |
| 個別組織の機密情報や内部手順の収集     | オープンソースとして公開・再利用するため                                                   | 公開可能な一般化済み知見のみ扱う                       |
| すべての文書種別の完成保証             | プロジェクトの目的、規模、リスク、公共性、運用責任に応じて成果物を選択できるようにするため | 初期公開では必要な文書体系と代表的な参考資料を優先する |
| 外部サービスや個別団体システムとの連携 | 初期公開では文書体系と公開可能な運用基盤の整備を優先するため                               | 連携が必要な場合は、公開後の利用事例に基づき再検討する |

## 5. 境界の判断基準

初期スコープでは、「人と AI Agent が同じ文書体系を参照し、仕様駆動開発の主要成果物を作成・レビュー・改善できる状態」を優先する。

迷った場合は、次の順で判断する。

1. 人と AI Agent が同じ仕様と判断根拠を参照して協働できることを優先する。
2. 特定プロジェクト固有の実装詳細よりも、複数プロジェクトで再利用できる文書体系と共通の型を優先する。
3. プロジェクトの目的、規模、リスク、公共性、運用責任に応じて成果物を選択できることを優先する。
4. 公開・再利用を妨げる機密情報、特定技術への過度な依存、人間の判断責任を代替する内容は初期スコープから外す。

## 6. スコープ変更方針

| 項目     | 内容                                                                                               |
| -------- | -------------------------------------------------------------------------------------------------- |
| 変更入口 | 利用者、関係者、または AI Agent による改善提案を、変更要求ログまたは該当する Issue / PR に記録する |
| 影響評価 | 業務価値、利用者影響、公開・再利用への影響、スケジュール影響、人間の判断責任への影響を確認する     |
| 承認者   | `PO` が対象範囲・対象外の変更を最終判断し、`BA` は利用者影響と受入観点を整理する                   |
| 記録先   | 変更内容、判断理由、影響範囲、影響を受ける成果物を変更要求ログまたは該当する Issue / PR に残す     |

初期公開前にスコープを追加する場合は、優先順位またはスコープ外項目の見直しを同時に行い、初期公開に必要な最小範囲を維持する。
