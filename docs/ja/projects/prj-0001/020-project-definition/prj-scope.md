# プロジェクトスコープ: SpecDojo

## 1. 対象業務

本プロジェクトでは、仕様駆動プロジェクト運営を実践・再利用するために必要な文書体系と運用基盤の整備を対象とする。主な利用者は、プロジェクトの立ち上げ・改善・運用を行う市民、専門家、行政、開発者、AI Agent とし、各利用者が同じ仕様と判断根拠を参照できる状態を目指す。

- システムライフサイクル全体を支える文書分類、配置、命名、メタデータ、派生関係の整理。
- 各成果物の rulebook、recipe、sample、template の整備。
- プロジェクト定義、プロジェクトマネジメント、設計、実装、テスト、運用・保守に関する文書テンプレートの整備。
- 人と AI Agent が同じ文書を参照して作業できるようにするための記述方針、品質基準、運用ルールの整備。
- SpecDojo 自体のプロジェクト管理、成果物カタログ、Schedule、実行管理の整備。
- 公開リポジトリとして利用・改善・再利用しやすい README、ライセンス、貢献導線、公開範囲の整理。

本スコープで利用者に影響する範囲は、文書の作成・参照・レビュー・改善に関わる導線である。個別プロジェクトの業務手順そのものを変更するのではなく、各プロジェクトが必要な成果物を選択し、合意形成と引き継ぎに使える型を提供する。

## 2. 対象システム

本プロジェクトの対象は、アプリケーションシステムそのものではなく、仕様駆動開発を支える文書フレームワークと補助ツール群である。

| 対象             | 内容                                                                                        |
| ---------------- | ------------------------------------------------------------------------------------------- |
| 文書体系         | `docs/ja/specdojo` 配下の standards、guides、rulebooks、instructions、samples           |
| プロジェクト文書 | `docs/ja/projects/prj-0001` 配下のプロジェクト定義、管理計画、組織、Schedule、管理台帳 |
| サンプル         | SpecDojo の利用イメージを示すサンプルプロジェクト、サンプル成果物                           |
| 補助ツール       | Schedule / 実行管理 / 生成物確認を支援する `specdojo` 関連コマンドおよび生成物        |
| 公開基盤         | GitHub リポジトリ、GitHub Pages、README、LICENSE、Issue / PR 導線                           |

## 3. 対象期間

- 初期対象期間は、SpecDojo の初回公開に必要な文書体系、主要ルール、主要サンプル、管理計画、公開導線が整うまでとする。
- 本格実行開始の GO / Not GO 判断は、プロジェクト定義文書とプロジェクトマネジメント文書が整備された時点で行う。
- 初回公開後の改善、利用者フィードバック対応、外部貢献受付、事例追加は継続改善フェーズとして扱う。
- 具体的なマイルストーン日は [[prj-0001:sch-milestones|マイルストーン]] を正とする。

## 4. スコープ外

| 対象（Out）                              | 理由                                                                                    | 補足                                           |
| ---------------------------------------- | --------------------------------------------------------------------------------------- | ---------------------------------------------- |
| 特定業務システムの本番開発               | SpecDojo は文書フレームワークであり、特定システムの実装は利用プロジェクト側の責務とする | サンプル実装や例示は必要最小限に留める         |
| 特定の開発手法・言語・クラウドへの最適化 | 幅広いプロジェクトで再利用できる共通基盤とするため                                      | 個別技術の詳細はサンプルまたは利用側文書で扱う |
| 有償サービス前提の運用設計               | OSS として自由に利用・改善・再利用できることを優先するため                              | 必要な場合は代替手段を併記する                 |
| 個別組織の機密情報や内部手順の収集       | 公開リポジトリとして扱うため                                                            | 公開可能な一般化済み知見のみ扱う               |
| 全文書種別の完成保証                     | 初期段階では主要文書体系と代表的なルール・サンプルを優先するため                        | 未整備文書は登録簿で管理する            |
| 利用者サポート窓口の本格運用             | 初回公開前は内部整備を優先するため                                                      | 外部参加受付後に Issue / PR 方針を見直す       |

## 5. 境界の判断基準

初期スコープでは、「人と AI Agent が同じ文書体系を参照し、仕様駆動開発の主要成果物を作成・レビュー・改善できる状態」を優先する。

迷った場合は、次の順で判断する。

1. 文書体系と参考資料の再利用性を高めるものを優先する。
2. 特定プロジェクト固有の実装詳細よりも、複数プロジェクトで使える共通ルールを優先する。
3. 公開可能性、保守容易性、AI 利用可能性を損なうものは初期スコープから外す。

## 6. スコープ変更方針

- 変更提案は、プロジェクト登録簿または Issue / Pull Request で記録する。
- 主要成果物の追加・削除、公開方針、ライセンス方針、文書体系の大幅変更は `PO` が最終判断する。
- 対象利用者、対象成果物、公開範囲、外部連携有無に影響する変更は、業務価値、利用者影響、スケジュール影響、保守責任を確認してから判断する。
- 変更時は、成果物カタログ、Schedule、RACI、管理計画への影響を確認する。
- 初期公開前にスコープを追加する場合は、優先順位またはスコープ外項目の見直しを同時に行う。
