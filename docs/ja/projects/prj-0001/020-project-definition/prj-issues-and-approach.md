# プロジェクト課題と解決アプローチ: SpecDojo

本書は、[[prj-0001:prj-overview|プロジェクト概要]] で示した必要性を起点とし、[[prj-0001:prj-scope|プロジェクトスコープ]] と [[prj-0001:prj-assumptions-constraints-dependencies|前提・制約・依存関係]] を前提に、SpecDojo の利用者（市民・専門家・行政、開発者、AI Agent）が直面する主要課題と、その解決アプローチを業務観点で整理する。詳細な設計・実装・テスト設計は扱わず、関係者が「何が課題で、どれを、どう解く方針か」を合意するための判断材料を示す。主要課題の整理と解決アプローチの方針承認は PO（Product Owner）が行い、BA は利用者影響と受入観点の整理に、ARC は技術的課題と実現方式の方向性の識別に、PM は後続作業計画やリスク管理の入力として利用する。

本書の合意対象は、初期公開までに優先して扱う主要課題、採用する解決アプローチ、先送りする事項、追加判断が必要な事項である。個別業務システムの設計、特定技術の採否、詳細な公開作業手順は、本書の合意対象に含めない。

## 1. 課題一覧

中心にある課題は、プロジェクトの知識や判断を、関係者が継続的に共有・更新できる形で言語化することが難しい点にある。その結果として利用者に生じている、観察可能な問題を次に示す。

| 課題ID | 課題                                                                           | 影響                                                                    | 優先度 | 備考                         |
| ------ | ------------------------------------------------------------------------------ | ----------------------------------------------------------------------- | ------ | ---------------------------- |
| P-01   | 目的や解決すべき課題が共有されないまま、実装やツール選定が先行する             | 利用者価値と一致しない成果物が作られ、方針変更や作り直しが発生する      | High   | Why に関する合意形成の問題   |
| P-02   | 要件や設計意図が関係者間で正確に伝わらず、認識齟齬と手戻りが発生する           | 確認・修正の工数が増え、品質と意思決定の一貫性が低下する                | High   | 知識伝達の問題               |
| P-03   | 仕様や判断経緯が残らず、プロジェクトが属人化・ブラックボックス化する           | 過去の判断を再検証できず、同じ議論や誤りが繰り返される                  | High   | 知識保持の問題               |
| P-04   | 担当者交代や長期運用時に、保守・改修に必要な知識を引き継げない                 | 作業の停滞、誤った変更、保守コストの増加が発生する                      | High   | 知識継承の問題               |
| P-05   | 変更時に、影響を受ける要件・設計・実装・運用ルールを追跡できない               | 変更漏れや不整合が残り、品質低下や障害につながる                        | High   | 変更追跡の問題               |
| P-06   | プロジェクト運営が特定個人の経験、善意、熱量に依存し、継続できない             | 担い手の負荷集中や離脱によって、活動や成果物の維持・改善が止まる        | High   | 継続性の問題                 |
| P-07   | 人間向けに記述された知識を AI Agent が解釈・参照し、安定して作業に利用できない | AI Agent の生成・更新結果がばらつき、人間による確認・修正の負担が増える | High   | 人と AI Agent の協働上の問題 |

## 2. 原因（仮説でも可）

| 課題ID     | 原因区分 | 内容                                                                                               | 根拠 / 補足                                                                  |
| ---------- | -------- | -------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| P-01       | 仮説     | 目的、対象者、期待成果、判断根拠を明示する方法が担当者の経験や力量に依存している                   | プロジェクト概要の「必要性」で示した中心的な課題に基づく                     |
| P-02       | 仮説     | 関係者が同じ意味で参照できる文書構造、用語、記述粒度が定まらず、情報が複数箇所に分散する           | 文書の粒度、構成、記述ルールがプロジェクトごとにばらつく可能性がある         |
| P-03       | 仮説     | 仕様と判断経緯を関連付けて記録し、変更に合わせて更新する方法と責任が定まっていない                 | 記録を納品物ではなく継続利用する知識として扱う仕組みが必要                   |
| P-04       | 仮説     | 作業の背景や判断根拠が成果物ではなく担当者個人の記憶、経験、口頭説明に残る                         | P-03 の属人化が担当者交代時に顕在化すると考えられる                          |
| P-05       | 仮説     | 要件、設計、実装、テスト、運用ルール間の関係を識別し、変更時に確認する方法が定まっていない         | スコープでライフサイクル全体の文書体系と派生関係を対象としている             |
| P-06       | 仮説     | 文書の作成・レビュー・改善手順と責任分担が再現可能な形になっておらず、個人の自発的な活動に依存する | プロジェクト概要の背景と必要性で示したリソース制約および継続困難に基づく     |
| P-07       | 仮説     | 文書が人間同士の説明に最適化され、AI Agent が必要とする構造、文脈、制約、参照関係が不足している    | 人と AI Agent が同じ仕様と判断根拠を参照するというスコープの判断基準に基づく |
| P-01〜P-07 | 仮説     | 言語化した知識を標準化・再利用し、継続的に更新する共通基盤がない                                   | 個別原因を横断する構造的な原因。利用事例と利用者確認による検証が必要         |

## 3. 解決策候補

[[prj-0001:prj-overview|プロジェクト概要]] の「基本的な考え方」を、本質課題を解くために必要な能力の候補として具体化する。A-01〜A-05 は相互排他的な運営・開発戦略ではなく、採択する戦略がどのような比重と仕組みで満たすかを評価する解決能力である。予測型、アジャイル型、仕様駆動型、Agent Loop 型、最小統制型などの戦略候補は、既存方式の調査を含めて [[prj-0001:prj-comparison-of-alternatives|代替案比較]] で構成・比較する。

| 案ID | 案                                 | 具体化する考え方              | 概要                                                                                                                                   | 主に対応する課題 | 期待できる効果                                                           |
| ---- | ---------------------------------- | ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | ---------------- | ------------------------------------------------------------------------ |
| A-01 | Why 起点で合意を形成する           | Why before What               | 実装やツールを決める前に、解決すべき課題、対象者、目的、期待成果、判断根拠を明文化し、関係者が合意する                                 | P-01、P-02、P-06 | 手段の先行を防ぎ、利用者価値と意思決定の一貫性を保てる                   |
| A-02 | 仕様を共有知識基盤として構成する   | Specification-Driven          | 要求、要件、仕様、設計、実装、テスト、運用、判断経緯を識別可能な形で関連付け、人と AI Agent が同じ仕様と判断根拠を参照できるようにする | P-02〜P-05、P-07 | 知識の共有・継承と変更追跡が可能になり、人と AI Agent の認識を揃えられる |
| A-03 | 必要な地点から始めて継続的に育てる | Start Anywhere, Evolve Always | 目的、規模、リスク、既存成果物に応じて必要な地点・成果物から着手し、学びや変更に合わせて Why と関連文書を継続的に更新する              | P-03〜P-06       | 固定工程や文書一式の負担を避けながら、知識を陳腐化させず維持・改善できる |
| A-04 | 人間の判断と検証証拠をゲートにする | 責任ある Human-in-the-loop    | AI Agent の提案・実行結果を主張として扱い、人間の承認と機械検証の証拠が揃った場合だけ状態を進める                                      | P-01、P-05〜P-07 | 判断責任を人間に残し、誤った完了判定や不整合の拡大を抑えられる           |
| A-05 | 共通の型と補助手段で再現可能にする | A-01〜A-04 を横断的に支援     | 文書体系、rulebook、recipe、sample、template、メタデータ、参照関係、品質確認、実行状態管理を必要性と優先度に応じて整備する             | P-01〜P-07       | 言語化の経験差を抑え、人と AI Agent が同じ方針を再現しやすくなる         |

## 4. 採用アプローチと理由

本書では A-01〜A-05 を、戦略選定に用いる必要能力として採用する。これは特定の開発方式やタスク管理ツールの採択ではない。PO は主要課題と必要能力を承認し、[[prj-0001:prj-comparison-of-alternatives|代替案比較]] では、各戦略がこれらをどの程度満たすかを、費用、期間、リスク、運用適合、技術実現性、成熟度とともに評価する。BA は Why と利用者課題の対応を確認し、ARC は方式境界と外部依存を識別し、PM は後続作業とリスクを分解し、QE は検証証拠と完了条件を確認する。

| 採用対象                              | 判定               | 理由                                                                                    |
| ------------------------------------- | ------------------ | --------------------------------------------------------------------------------------- |
| A-01 Why 起点の合意形成               | 必要能力として採用 | 目的や解決課題が曖昧なまま手段が先行する問題を直接扱い、戦略を評価する基準になるため    |
| A-02 仕様を共有知識基盤として構成     | 必要能力として採用 | 知識と判断の共有、継承、変更追跡、人と AI Agent の協働をライフサイクル全体で支えるため  |
| A-03 必要な地点から始めて継続的に育成 | 必要能力として採用 | 案件特性に応じた着手と、学びや変更による継続更新を可能にするため                        |
| A-04 人間の判断と検証証拠             | 必要能力として採用 | Agent の自動実行と人間の判断責任を分離し、誤った状態遷移を防ぐため                      |
| A-05 共通の型と補助手段               | 必要能力として採用 | A-01〜A-04 を属人的な努力ではなく、選択・再利用・検証可能な形で実践できるようにするため |

必要能力の優先順位は、本質課題への効果、Why との整合、知識の共有・追跡可能性、人間の判断責任、継続的な更新、案件に応じた選択、AI 利用可能性の順とする。どの戦略と支援方式で実現するかは本書で先取りせず、外部の有力方式を含む [[prj-0001:prj-comparison-of-alternatives|代替案比較]] で決定する。

## 5. トレードオフ/リスク

トレードオフは採用に伴い受容する事項、リスクは所有者が監視する事項として扱う。各リスクは、状態変化と軽減策の実施状況を [[prj-0001:pjr-index|プロジェクト登録簿]]（リスク登録簿ビュー）へ転記して追跡する。

| 区分         | 内容                                                                         | 軽減策                                                   | 所有者 |
| ------------ | ---------------------------------------------------------------------------- | -------------------------------------------------------- | ------ |
| トレードオフ | 初期段階では全文書種別の完成よりも、主要文書体系の成立を優先する             | 未整備項目を成果物カタログと後続作業で管理する           | `PO`   |
| トレードオフ | 汎用性を優先するため、特定技術の詳細手順は薄くなる                           | サンプルや利用プロジェクト側文書に委譲する               | `ARC`  |
| トレードオフ | 成果物の選択制を残すため、利用者には判断の手間が残る                         | 適用条件、対象外、前提を rulebook と成果物説明に明記する | `BA`   |
| リスク       | 文書体系や補助ツールの整備自体が目的化し、利用者の本質課題との対応が弱くなる | 各成果物が P-01〜P-07 のどの課題に寄与するか確認する     | `ARC`  |
| リスク       | rulebook、recipe、sample、template の整合性が崩れる                          | 後続の品質確認とレビューで確認する                       | `QE`   |
| リスク       | AI Agent 向けの参考資料が曖昧で出力品質がばらつく                            | 参考資料を継続改善する                                   | `ARC`  |
| リスク       | AI Agent に承認や最終判断を委ねる表現が混入し、責任分担が曖昧になる          | 判断者と Agent の役割を成果物内で区別する                | `PO`   |
| リスク       | 公開資料に個人情報、機密情報、非公開の組織情報が混入する                     | 公開前に差分と記載情報を確認し、必要に応じて一般化する   | `QE`   |
| リスク       | 個人・小規模運用に負荷が集中する                                             | 優先順位を絞り、外部参加導線を段階的に整備する           | `PM`   |

## 7. 見直しと変更管理

本書は原因を仮説として扱う暫定的な合意であり、確定情報や環境変化に応じて見直す。見直しの契機、確認する観点、対応方針を次に示す。

| 見直しの契機                                                 | 確認する観点                                       | 対応方針                                                         |
| ------------------------------------------------------------ | -------------------------------------------------- | ---------------------------------------------------------------- |
| 課題または原因の仮説が利用事例・利用者確認で反証された       | 反証された P-ID と影響、依存する A-ID の妥当性     | 課題・原因・採用アプローチを更新し、必要なら判断軸を再評価する   |
| プロジェクトスコープまたは前提・制約・依存関係が変更された   | スコープ外・境界の判断基準との整合、成立条件の変化 | 影響する課題・解決策を洗い直し、代替案比較へ再評価を依頼する     |
| 採用アプローチに影響する決定が記録された                     | 決定内容と本書の方針の差分                         | 方針を決定に合わせて改訂し、決定記録と相互参照する               |
| 成果物カタログの棚卸でルール・サンプルの欠落や重複が判明した | P-05 の整合性維持リスクへの影響                    | 是正対象と優先度を次の検討事項と後続作業へ反映する               |
| リスクが顕在化した                                           | 該当リスクの所有者、軽減策の実施状況               | リスク登録簿で状態を更新し、必要なら課題として課題一覧へ昇格する |

各項目の所有者が成立状況と変化を確認し、PM がプロジェクト全体への影響を取りまとめる。主要課題、採用アプローチ、優先順位の変更は PO が最終判断し、見直し結果と判断根拠を [[prj-0001:pjr-index|プロジェクト登録簿]] または決定記録に残す。
