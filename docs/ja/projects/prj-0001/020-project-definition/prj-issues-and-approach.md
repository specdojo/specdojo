# プロジェクト課題と解決アプローチ: SpecDojo

本書は、[[prj-0001:prj-scope|プロジェクトスコープ]] と [[prj-0001:prj-assumptions-constraints-dependencies|前提・制約・依存関係]] を前提に、SpecDojo の利用者（市民・専門家・行政、開発者、AI Agent）が直面する主要課題と、その解決アプローチを業務観点で整理する。詳細な設計・実装・テスト設計は扱わず、関係者が「何が課題で、どれを、どう解く方針か」を合意するための判断材料を示す。本書は [[prj-0001:prj-scope|プロジェクトスコープ]] で定義した利用者の影響と [[prj-0001:prj-assumptions-constraints-dependencies|前提・制約・依存関係]] の成立条件を踏まえ、主要課題の整理と解決アプローチの方針承認を PO（Product Owner）が行うための記録である。BA は利用者影響と受入観点を整理し、ARC は技術的課題と実現方式の方向性の識別基準として、PM は後続作業計画やリスク管理の入力として利用する。

本書の合意対象は、初期公開までに優先して扱う主要課題、採用する解決アプローチ、先送りする事項、追加判断が必要な事項である。個別業務システムの設計、特定技術の採否、詳細な公開作業手順は、本書の合意対象に含めない。

## 1. 課題一覧

| 課題ID | 課題                                                                       | 影響                                                                              | 優先度 | 備考                                               |
| ------ | -------------------------------------------------------------------------- | --------------------------------------------------------------------------------- | ------ | -------------------------------------------------- |
| P-01   | 開発文書の粒度、構成、記述ルールがプロジェクトごとにばらつきやすい         | 市民、専門家、行政、開発者が成果物を再利用・レビュー・引き継ぎしにくくなる        | High   | 文書体系と rulebook の整備が必要                   |
| P-02   | 要件、仕様、設計意図、判断経緯が構造化されず、AI Agent が作業に使いにくい  | AI Agent の生成物品質がばらつき、人間の確認・修正の手戻りが増える                 | High   | rulebook / recipe / sample / template の整備が必要 |
| P-03   | プロジェクト開始時に必要な文書と順序が分かりにくい                         | 立ち上げが遅れ、重要な前提や制約が抜け、GO / Not GO 判断の材料が不足する          | High   | 主要成果物と参考資料の対応を明確にする             |
| P-04   | OSS として公開・改善・再利用に参加するための入口や判断基準が読み取りにくい | 利用者や貢献者が参加しにくく、公開後の継続改善と再利用が進まない                  | Medium | README、LICENSE、Issue / PR 方針が必要             |
| P-05   | ルール、サンプル、生成物が増えるほど整合性維持が難しくなる                 | 古いルールや矛盾したサンプルが残り、利用者と Agent が参照する根拠の信頼性が下がる | High   | 品質管理、成果物カタログ、生成物方針が必要         |

## 2. 原因（仮説でも可）

| 課題ID | 原因区分 | 内容                                                                                                 | 根拠 / 補足                                                 |
| ------ | -------- | ---------------------------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| P-01   | 事実     | 文書の作成・参照・レビュー・改善に関わる導線を共通化しないと、担当者ごとの判断に依存しやすい         | スコープの対象業務・利用者影響に基づく                      |
| P-02   | 仮説     | AI Agent は、曖昧な入力や制約不足に弱く、文書構造の差を吸収しきれない                                | ACD-A01 の役割分担を前提に、参考資料の整備で軽減可能        |
| P-03   | 事実     | プロジェクト定義、管理計画、設計、実装、テスト、運用・保守に関する成果物が多く、作成順序の判断が必要 | スコープの対象業務・対象システムに基づく                    |
| P-04   | 仮説     | OSS 公開だけでは、貢献方法やレビュー基準が分からず参加障壁が残る                                     | ACD-D03、ACD-D04 が未確定であり、初回公開前の導線整備が必要 |
| P-05   | 事実     | 文書種別が増えるほど、リンク、ID、ルール、サンプルの更新漏れが起きやすい                             | ACD-D02、ACD-C04 の成立条件に基づく                         |

## 3. 解決策候補

| 案ID | 案                                   | 概要                                                        | 期待できる効果                           |
| ---- | ------------------------------------ | ----------------------------------------------------------- | ---------------------------------------- |
| A-01 | 文書体系・ルールブック中心で整備する | 成果物分類、構成、命名、メタデータ、rulebook を先に整備する | 再利用性とレビュー可能性が高まる         |
| A-02 | サンプル中心で整備する               | 実例を多く作り、利用者が真似できる形を優先する              | 導入しやすく、完成イメージを伝えやすい   |
| A-03 | 補助ツール中心で整備する             | スケジュール、生成物、チェックを自動化する                  | 実行管理と整合性確認を効率化できる       |
| A-04 | プロジェクト個別支援として整備する   | 特定プロジェクトの成果物を作りながら標準化する              | 実務適合性は高いが、汎用性が下がりやすい |

## 4. 採用アプローチと理由

採用アプローチは、A-01 を中核に、A-02 と A-03 を段階的に組み合わせる方針とする。PO はこの方針により主要課題と解決方針を承認し、ARC は文書体系・参照資料・補助ツールの責務境界を識別し、PM は初期公開までの後続作業とリスクを分解する。

| 採用対象                        | 判定     | 理由                                                                         |
| ------------------------------- | -------- | ---------------------------------------------------------------------------- |
| A-01 文書体系・ルールブック中心 | 採用     | SpecDojo の中核は、再利用可能な文書体系と作成ルールであるため                |
| A-02 サンプル中心               | 一部採用 | ルールだけでは導入しにくいため、主要文書にはサンプルを用意する               |
| A-03 補助ツール中心             | 一部採用 | スケジュール / 生成物の整合性確認に有効だが、初期は文書体系を優先する        |
| A-04 プロジェクト個別支援       | 非採用   | 特定プロジェクトに寄りすぎると汎用フレームワークとしての再利用性が下がるため |

判断軸の優先順位は、再利用性、AI 利用可能性、公開・改善しやすさ、保守容易性、導入しやすさの順とする。

## 5. トレードオフ/リスク

| 区分         | 内容                                                                | 軽減策                                                 |
| ------------ | ------------------------------------------------------------------- | ------------------------------------------------------ |
| トレードオフ | 初期段階では全文書種別の完成よりも、主要文書体系の成立を優先する    | 未整備項目を成果物カタログと後続作業で管理する         |
| トレードオフ | 汎用性を優先するため、特定技術の詳細手順は薄くなる                  | サンプルや利用プロジェクト側文書に委譲する             |
| リスク       | rulebook、recipe、sample、template の整合性が崩れる                 | 後続の品質確認とレビューで確認する                     |
| リスク       | AI Agent 向けの参考資料が曖昧で出力品質がばらつく                   | 参考資料を継続改善する                                 |
| リスク       | AI Agent に承認や最終判断を委ねる表現が混入し、責任分担が曖昧になる | 判断者と Agent の役割を成果物内で区別する              |
| リスク       | 公開資料に個人情報、機密情報、非公開の組織情報が混入する            | 公開前に差分と記載情報を確認し、必要に応じて一般化する |
| リスク       | 個人・小規模運用に負荷が集中する                                    | 優先順位を絞り、外部参加導線を段階的に整備する         |
