# プロジェクト課題と解決アプローチ: SpecDojo

本書は、[[prj-0001:prj-scope|プロジェクトスコープ]] と [[prj-0001:prj-assumptions-constraints-dependencies|前提・制約・依存関係]] を前提に、SpecDojo の利用者（市民・専門家・行政、開発者、AI Agent）が直面する主要課題と、その解決アプローチを業務観点で整理する。詳細な設計・実装・テスト設計は扱わず、関係者が「何が課題で、どれを、どう解く方針か」を合意するための判断材料を示す。本書は [[prj-0001:prj-scope|プロジェクトスコープ]] で定義した利用者の影響と [[prj-0001:prj-assumptions-constraints-dependencies|前提・制約・依存関係]] の成立条件を鑑み、主要課題の整理と解決アプローチの方針承認を PO（Product Owner）が行うための記録であり、ARC は技術的課題と実現方式の方向性の識別基準として、PM は後続作業計画やリスク管理の入力として利用する。

## 1. 課題一覧

| 課題ID | 課題                                                                      | 影響                                               | 優先度 | 備考                                                 |
| ------ | ------------------------------------------------------------------------- | -------------------------------------------------- | ------ | ---------------------------------------------------- |
| P-01   | 開発文書の粒度、構成、記述ルールがプロジェクトごとにばらつきやすい        | 再利用、レビュー、引き継ぎ、AI 利用が難しくなる    | High   | 文書体系と rulebook の整備が必要                     |
| P-02   | 要件、仕様、設計意図、判断経緯が構造化されず、AI Agent が作業に使いにくい | 生成物品質のばらつき、誤解釈、手戻りが増える       | High   | rulebook / recipe / sample / template の整備が必要   |
| P-03   | プロジェクト開始時に必要な文書と順序が分かりにくい                        | 立ち上げが遅れ、重要な前提や制約が抜ける           | High   | docs-structure-guide、成果物カタログ、管理計画で補う |
| P-04   | OSS として公開・改善・再利用するための導線が不足しやすい                  | 利用者や貢献者が参加しにくく、継続改善が進まない   | Medium | README、LICENSE、Issue / PR 方針が必要               |
| P-05   | ルール、サンプル、生成物が増えるほど整合性維持が難しくなる                | 古いルールや矛盾したサンプルが残り、信頼性が下がる | High   | 品質管理、成果物カタログ、生成物方針が必要           |

## 2. 原因（仮説でも可）

| 課題ID | 原因区分 | 内容                                                                     | 根拠 / 補足                              |
| ------ | -------- | ------------------------------------------------------------------------ | ---------------------------------------- |
| P-01   | 事実     | 一般的な開発文書は組織や担当者の経験に依存しやすい                       | prj-overview の必要性に記載              |
| P-02   | 仮説     | AI Agent は、曖昧な入力や制約不足に弱く、文書構造の差を吸収しきれない    | 参考資料の整備で軽減可能                  |
| P-03   | 事実     | 企画、定義、管理、設計、テスト、運用の文書が多く、作成順序の判断が必要   | docs-structure-guide と dct-index が必要 |
| P-04   | 仮説     | OSS 公開だけでは、貢献方法やレビュー基準が分からず参加障壁が残る         | 初回公開前の導線整備が必要               |
| P-05   | 事実     | 文書種別が増えるほど、リンク、ID、ルール、サンプルの更新漏れが起きやすい | 品質管理計画で監視する                   |

## 3. 解決策候補

| 案ID | 案                                   | 概要                                                        | 期待できる効果                           |
| ---- | ------------------------------------ | ----------------------------------------------------------- | ---------------------------------------- |
| A-01 | 文書体系・ルールブック中心で整備する | 成果物分類、構成、命名、メタデータ、rulebook を先に整備する | 再利用性とレビュー可能性が高まる         |
| A-02 | サンプル中心で整備する               | 実例を多く作り、利用者が真似できる形を優先する              | 導入しやすく、完成イメージを伝えやすい   |
| A-03 | 補助ツール中心で整備する             | スケジュール、生成物、チェックを自動化する             | 実行管理と整合性確認を効率化できる       |
| A-04 | プロジェクト個別支援として整備する   | 特定プロジェクトの成果物を作りながら標準化する              | 実務適合性は高いが、汎用性が下がりやすい |

## 4. 採用アプローチと理由

採用アプローチは、A-01 を中核に、A-02 と A-03 を段階的に組み合わせる方針とする。

| 採用対象                        | 判定     | 理由                                                                         |
| ------------------------------- | -------- | ---------------------------------------------------------------------------- |
| A-01 文書体系・ルールブック中心 | 採用     | SpecDojo の中核は、再利用可能な文書体系と作成ルールであるため                |
| A-02 サンプル中心               | 一部採用 | ルールだけでは導入しにくいため、主要文書にはサンプルを用意する               |
| A-03 補助ツール中心             | 一部採用 | スケジュール / 生成物の整合性確認に有効だが、初期は文書体系を優先する  |
| A-04 プロジェクト個別支援       | 非採用   | 特定プロジェクトに寄りすぎると汎用フレームワークとしての再利用性が下がるため |

判断軸の優先順位は、再利用性、AI 利用可能性、公開・改善しやすさ、保守容易性、導入しやすさの順とする。

## 5. トレードオフ / リスク

| 区分         | 内容                                                             | 軽減策                                            |
| ------------ | ---------------------------------------------------------------- | ------------------------------------------------- |
| トレードオフ | 初期段階では全文書種別の完成よりも、主要文書体系の成立を優先する | 未整備項目を成果物カタログ、WBS、登録簿で管理する |
| トレードオフ | 汎用性を優先するため、特定技術の詳細手順は薄くなる               | サンプルや利用プロジェクト側文書に委譲する        |
| リスク       | rulebook、recipe、sample、template の整合性が崩れる              | 品質管理計画とレビューで確認する                  |
| リスク       | AI Agent 向けの参考資料が曖昧で出力品質がばらつく                | 参考資料を継続改善する                            |
| リスク       | 個人・小規模運用に負荷が集中する                                 | 優先順位を絞り、外部参加導線を段階的に整備する    |
