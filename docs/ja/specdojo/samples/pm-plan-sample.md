# プロジェクト管理計画 サンプル

本書は、駄菓子屋きぬや販売管理システム構築プロジェクトの計画化、進捗確認、課題・リスク管理、変更管理の運用方針を、PM が実行管理に使える粒度で定義する最小サンプルである。

## 1. 概要

本プロジェクトは、販売、在庫、つけ管理の初期リリースを対象とし、小規模な兼務体制で進める。PM は計画、進捗、課題、リスク、変更要求の管理プロセスを起案・運用し、PO は目的、優先順位、公開方針、変更要求の採否を判断する。

採用する Role code の語彙は `pm-organization.md` と `pm-roles.yaml` を正とする。実行主体の割り当ては `pm-members.yaml` で管理し、本書には個人名や agent 名を複製しない。

AI Agent は文書草案、差分整理、整合確認を支援してよい。ただし、最終承認、公開可否、変更要求の採否は人間の PO が判断する。

## 2. 管理方針（スコープ / スケジュール / コスト）

### 2.1. 基本方針

| 管理領域 | 管理単位 | PM の運用方針 | 最終判断 | 逸脱時対応 |
| --- | --- | --- | --- | --- |
| スコープ | 成果物、変更要求、未決事項 | 初期リリース対象を販売、在庫、つけ管理に限定し、追加要望は変更要求として整理する | PO | EC、会計連携、複数店舗対応の追加はスコープ変更として扱う |
| スケジュール | Schedule、マイルストーン、依存関係 | 主要成果物の順序と期限を確認し、2営業日超の遅延を課題化する | PM、必要時 PO | 店頭試用日へ影響する場合は再計画案を作成する |
| コスト | 作業負荷、外部依存、運用負荷 | 金額予算ではなく、家族と小規模支援者で継続できる作業量かを確認する | PO | 有償サービスや継続費用が必要な場合は判断事項として起票する |
| 品質 | rulebook、レビュー、検証結果 | 文書構造、受入条件、公開適性を確認し、後続作業に使える粒度へ整える | QE、必要時 PO | 重大な不整合や公開不適切情報は是正対象にする |
| 変更 | 変更要求、決定記録、影響範囲 | 変更内容、理由、影響先、採否案を分離して整理する | PO | 採否決定後に成果物と Schedule を更新する |

### 2.2. スコープ管理

- 初期リリースの対象は、店頭販売記録、在庫確認、つけ残高確認に限定する。
- 仕入れ先連携、本格会計、EC、複数店舗対応は対象外として扱い、要望が出た場合は変更要求に分離する。
- 下流成果物へ渡す入力は、対象、前提、制約、未決事項に分けて記録する。
- 判断待ちの内容は本文に混在させず、課題または決定記録へ転記できる単位にする。

### 2.3. スケジュール管理

- PM は、要件整理、設計、実装、テスト、店頭試用、公開準備の順序と依存関係を Schedule で管理する。
- Schedule の `owner` には Role code を使用し、member nickname、agent 名、個人名は使わない。
- 期限超過、入力不足、レビュー待ち、検証失敗が後続へ影響する場合は課題として記録する。
- 店頭試用日へ影響する遅延は、再計画案と判断が必要な事項を PO へ提示する。

### 2.4. コスト管理

| 観点 | 管理内容 | 判断条件 | 記録先 |
| --- | --- | --- | --- |
| 作業負荷 | 文書作成、動作確認、店頭試用準備に必要な時間 | 学校・店舗の予定と両立できない作業量になった | 課題 |
| 外部依存 | 生成 AI、GitHub、公開基盤、補助ツールへの依存 | 有償化、利用制限、代替困難性が発生した | リスク、変更要求 |
| 運用負荷 | 店頭での問い合わせ、記録確認、家族への説明 | 店主または家族だけで対応困難な頻度になった | 課題、変更要求 |
| 再作業負荷 | 文書体系、画面、手順の作り直し | 主要方針変更で複数成果物の再編集が必要になった | Schedule、課題 |

## 3. 品質・リスク・課題管理方針

### 3.1. 品質管理

| 観点 | 基準 | 確認方法 | 主な責任 |
| --- | --- | --- | --- |
| 構造 | frontmatter、章構成、必須表が rulebook に従う | rulebook、sample、lint を確認する | PM、QE |
| 一貫性 | Role code、RACI、Schedule、受入条件が矛盾しない | 関連文書を横断確認する | PM、ARC |
| 計画化可能性 | 内容がタスク化、順序付け、進捗確認に使える | 入力、出力、依存、未決事項を確認する | PM |
| 公開適性 | 個人情報、非公開情報、認証情報を含めない | 公開前レビューで確認する | PO、QE |

### 3.2. リスク管理

- PM は、初期リリース、店頭試用、公開判断、継続運用へ影響する可能性がある事項をリスクとして記録する。
- リスクは、影響、発生可能性、対応方針、責任 Role code、期限を含めて管理できる粒度にする。

| 記録トリガー | 例 | 初期対応 |
| --- | --- | --- |
| 店頭試用へ影響する | タブレット準備が間に合わない | 代替端末または紙運用の併用を検討する |
| 利用者へ影響する | 店主が操作手順を覚えにくい | 操作手順の見直しと店頭練習を計画する |
| 公開判断へ影響する | サンプルデータに個人情報が混入する | 公開前に匿名化または削除する |

### 3.3. 課題管理

- PM は、すでに発生し、対応しなければ作業が進まない事項を課題として扱う。
- 課題には、状態、優先度、対応 Role code、期限、次アクションを記録する。

| 課題区分 | 判断基準 | PM の初期対応 |
| --- | --- | --- |
| 入力不足 | 要件、受入条件、制約が未決で設計へ進めない | 不足情報と判断者を明示する |
| 整合不備 | Role code、責任分担、Schedule が矛盾している | 正本を特定し、修正対象を切り分ける |
| 検証失敗 | lint、schema、動作確認が失敗している | 失敗内容、対象、再実行条件を記録する |
| 判断待ち | PO の採否または公開判断が必要 | 判断材料、選択肢、影響範囲を整理する |

### 3.4. 変更管理

- 初期リリース対象、主要成果物、Role code、責任分担、店頭試用日、公開方針へ影響する変更は変更要求として扱う。
- PM は、変更内容、理由、影響先、代替案、推奨案、採否判断者を整理する。
- 誤字修正、リンク修正、表記ゆれ修正は通常更新として扱う。ただし、意味や責務が変わる場合は変更要求に含める。

## 4. 体制と意思決定

### 4.1. 採用ロール

| Role code | 管理計画での主な責務 |
| --- | --- |
| PO | 目的、優先順位、公開方針、変更要求の採否を判断する |
| PM | 計画化、進捗確認、課題・リスク管理、変更要求の影響整理を担う |
| BA | 店主、家族利用者、常連客の利用価値と受入条件を確認する |
| ARC | 文書体系、構成、依存関係、技術制約との整合を確認する |
| DEV | 実装・設定・技術的変更作業の実現可能性を確認する |
| QE | 品質基準、レビュー、検証、公開適性を確認する |
| UX | 店頭操作、説明、導線の分かりやすさを確認する |
| OPS | 店頭導入、公開、運用、変更適用の影響を確認する |

### 4.2. 意思決定

| 判断対象 | 起案・整理 | 最終判断 | 相談先 | 証跡 |
| --- | --- | --- | --- | --- |
| 作業順序、進捗確認、ブロック解消 | PM | PM | ARC、QE、関連 Role code | Schedule、課題 |
| 初期リリース範囲、優先順位の変更 | PM | PO | BA、ARC、QE | 変更要求、決定記録 |
| 品質基準、検証方針、公開前確認 | QE、PM | PO | ARC、OPS | レビュー結果、決定記録 |
| AI Agent の出力採否 | PM | PO または成果物の A | 関連 Role code | 差分、レビュー結果 |
| 公開可否、外部提供 | OPS、PM | PO | ARC、QE | 公開前チェック、決定記録 |

## 5. レポーティングと見直し

### 5.1. レポーティング

| 管理領域 | 指標 | 報告タイミング | 主な報告先 | 閾値超過時対応 |
| --- | --- | --- | --- | --- |
| 進捗 | 完了、着手中、ブロック、期限超過 | 作業節目、Schedule 更新時 | PO、関連 Role code | ブロック理由と再計画案を整理する |
| 課題 | open 件数、高優先度、期限超過、判断待ち | 課題発生時、作業節目 | PO、PM | 対応 Role code、期限、次アクションを明確にする |
| リスク | 高影響リスク、顕在化兆候、対応期限 | リスク識別時、見直し時 | PO、PM、QE | 回避、軽減、受容、移管の方針を整理する |
| 変更要求 | スコープ、Schedule、公開方針への影響 | 変更要求起票時 | PO、ARC、OPS | 影響範囲と採否案を決定記録へ接続する |
| 品質 | lint、レビュー指摘、不整合 | 主要成果物更新時 | PM、QE、ARC | 正本を修正し、必要な生成物を再生成する |

### 5.2. 見直し条件

| 更新トリガー | 見直し内容 | 主な責任 |
| --- | --- | --- |
| 組織定義またはロール定義を変更した | 判断者、Schedule `owner`、RACI、報告先との整合を確認する | PM、ARC |
| 初期リリース範囲または優先順位を変更した | 管理単位、変更要求、見直し条件、GO / Not GO 判断への影響を確認する | PM、PO |
| 成果物カタログまたは Schedule を大きく変更した | 進捗管理単位、依存関係、報告タイミング、再計画手順を確認する | PM |
| 課題滞留、リスク顕在化、変更要求増加が継続した | エスカレーション先、判断期限、PM 責務の専任化要否を確認する | PM、PO |
| 店頭試用または公開準備を開始した | 公開前確認、問い合わせ、運用負荷、個人情報・機密情報の扱いを確認する | PO、OPS、QE |

### 5.3. GO / Not GO 判断

- 初期リリース対象、優先順位、受入条件が矛盾していないこと。
- 採用 Role code、Schedule `owner`、RACI、実行主体の割り当てが整合していること。
- 主要な課題、リスク、変更要求、未決事項が識別され、対応方針または判断待ちとして分離されていること。
- 必要な検証とレビューの結果が確認されていること。
- 公開対象に個人情報、非公開情報、認証情報、秘密情報が含まれていないこと。

## 7. 継続確認事項

| 論点 | 現状 | 対応方針 | 判断者 |
| --- | --- | --- | --- |
| PM 責務の専任化 | 小規模運用では専任化しない | 進捗遅延、課題滞留、リスク顕在化が継続した場合に見直す | PO |
| 管理台帳への転記単位 | 課題、リスク、変更要求、決定記録を分離する | PM が初期分類し、必要に応じて該当台帳へ転記する | PM |
| 初回公開判断 | _UNDECIDED_: 公開可否と公開範囲は未了 | GO / Not GO 判断時に検証結果と公開不適切情報の有無を確認する | PO |
