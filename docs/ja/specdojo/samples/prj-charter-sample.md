# プロジェクト憲章: 駄菓子屋きぬや 販売管理システム

本書は、駄菓子屋きぬや 販売管理システム構築プロジェクトの立ち上げを認可し、詳細計画と初期準備を進める権限を委譲するための憲章である。店主代表が PO として、目的、優先順位、予算枠、リリース可否を最終判断する。

## 1. 本書の目的

- 店頭販売、在庫、つけ管理を一元化する小規模プロジェクトの立ち上げを認可する。
- 開発担当に、要件整理、詳細計画、初期準備、見積もり作成を進める権限を委譲する。
- 本書の承認は、本格実行開始、リリース、追加支出、主要スコープ変更を承認するものではない。

## 2. 認可対象

| 項目            | 内容                                                     |
| --------------- | -------------------------------------------------------- |
| プロジェクト名  | 駄菓子屋きぬや 販売管理システム構築                      |
| プロジェクト ID | `prj-0001`                                               |
| 認可対象        | 立ち上げ、詳細計画策定、初期準備、概算見積もり作成       |
| 認可しない範囲  | 本格開発開始、リリース判断、外部支出、主要スコープ変更   |
| 直接根拠        | `prj-overview-sample`、`prj-stakeholder-register-sample` |
| 承認責任        | 店主代表（`PO`）                                         |

## 3. プロジェクトの目的

駄菓子屋きぬやでは、販売記録、在庫確認、常連客のつけ管理をノートと記憶に頼っている。プロジェクトの目的は、店主代表が無理なく使える仕組みで日々の販売、在庫、つけ残高を記録し、家族利用者が店番する場合も同じ手順で確認できる状態を作ることである。

| 観点     | 認可判断で確認する期待効果                             | 根拠                              |
| -------- | ------------------------------------------------------ | --------------------------------- |
| 販売記録 | 日々の販売内容を記録し、売上と在庫の確認に使える       | `prj-overview-sample`             |
| 在庫     | 人気商品の欠品や在庫過多を減らすための記録を始められる | `prj-overview-sample`             |
| つけ管理 | 常連客のつけ残高を短時間で確認できる                   | `prj-overview-sample`             |
| 継続運用 | 店主代表と家族利用者が同じ手順で利用できる             | `prj-stakeholder-register-sample` |

## 6. ハイレベル成功基準

| 観点       | 成功判定の候補                                   | 詳細化先                                       |
| ---------- | ------------------------------------------------ | ---------------------------------------------- |
| 欠品削減   | 人気商品の欠品回数を記録し、改善効果を確認できる | `prj-success-criteria-and-acceptance-criteria` |
| 在庫適正化 | 廃棄、値引き、長期滞留を月次で確認できる         | `prj-success-criteria-and-acceptance-criteria` |
| つけ確認   | 顧客検索から残高確認までの時間を測定できる       | `prj-success-criteria-and-acceptance-criteria` |
| 継続運用   | 営業日の記録継続状況を確認できる                 | `prj-success-criteria-and-acceptance-criteria` |

## 7. 初期ステークホルダー

| ID                      | 関係者         | 関与区分                         | 対応 Role code | 認可判断における主な責任                         |
| ----------------------- | -------------- | -------------------------------- | -------------- | ------------------------------------------------ |
| `STH-OWNER`             | 店主代表       | スポンサー / 意思決定者 / 利用者 | `PO`           | 目的、優先順位、予算枠、リリース可否を判断する   |
| `STH-FAMILY-USER`       | 家族利用者代表 | 利用者                           | `BA`           | 店番時の操作性と引継ぎ方法を確認する             |
| `STH-DEVELOPMENT`       | 開発担当       | 実行担当者                       | `PM`           | 合意した対象範囲に従い、計画と変更影響を共有する |
| `STH-REGULAR-CUSTOMERS` | 常連客         | 影響受容者                       | なし           | つけ残高や販売方法の変更から影響を受ける         |
| `STH-SUPPLIER`          | 仕入れ先       | 外部協力者                       | なし           | 初期リリースでは現行連絡を継続する               |

詳細は `prj-stakeholder-register-sample` を参照する。

## 8. 権限委譲

| 項目             | 決裁者           | 実行責任者       | 協議先             | 証跡                   |
| ---------------- | ---------------- | ---------------- | ------------------ | ---------------------- |
| 立ち上げ認可     | 店主代表（`PO`） | 開発担当（`PM`） | 家族利用者代表     | 本書の承認履歴         |
| 詳細計画策定     | 店主代表（`PO`） | 開発担当（`PM`） | 家族利用者代表     | `pm-plan`、議事録      |
| 概算見積もり作成 | 店主代表（`PO`） | 開発担当（`PM`） | 必要時に技術支援者 | 見積もりメモ、決定記録 |
| 本格実行開始     | 店主代表（`PO`） | 開発担当（`PM`） | 家族利用者代表     | GO / Not GO 決定記録   |

次の事項は店主代表（`PO`）の承認を必要とする。

- 初期対象範囲の変更
- 費用上限の確定または増額
- 外部サービス利用や機材購入
- リリース日と本格実行開始の判断
- つけ管理など常連客への影響がある運用変更

## 9. 主要前提・制約

| 区分 | 内容                                                                                     | 詳細化先                                   |
| ---- | ---------------------------------------------------------------------------------------- | ------------------------------------------ |
| 前提 | 単一店舗で運用する                                                                       | `prj-assumptions-constraints-dependencies` |
| 前提 | 店主代表と家族利用者が、同じ手順で販売、在庫、つけ残高を確認する                         | `prj-assumptions-constraints-dependencies` |
| 前提 | 店舗の販売手順を大きく変えず、現行運用に近い操作を優先する                               | `prj-assumptions-constraints-dependencies` |
| 制約 | 初期リリースでは EC、複数店舗、本格会計連携、仕入れ先システム連携を含めない              | `prj-assumptions-constraints-dependencies` |
| 制約 | 費用上限は `_UNDECIDED_:` 詳細計画策定時に店主代表が承認する。承認前の外部支出は行わない | 未決事項                                   |
| 制約 | 顧客情報は必要最小限にし、公開サンプルには個人情報を含めない                             | `prj-assumptions-constraints-dependencies` |

## 11. 本格実行開始の GO / Not GO 判断

本格実行開始は、次の観点を確認して店主代表（`PO`）が判断する。

| 判断観点     | 確認内容                                                         | 記録先                                         |
| ------------ | ---------------------------------------------------------------- | ---------------------------------------------- |
| 目的整合     | 販売、在庫、つけ管理の初期目的と対象範囲が一致している           | 決定記録                                       |
| 予算枠       | 費用上限、外部支出、機材購入の要否が承認されている               | 決定記録                                       |
| スケジュール | リリース目標日と受入確認の時期が合意されている                   | `pm-plan`                                      |
| 体制         | 店主代表、家族利用者代表、開発担当の確認タイミングが決まっている | `pm-raci`                                      |
| 品質         | 操作確認、つけ残高確認、記録継続の判定方法が決まっている         | `prj-success-criteria-and-acceptance-criteria` |
