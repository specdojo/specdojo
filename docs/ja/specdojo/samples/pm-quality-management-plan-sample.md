# 品質管理計画 サンプル

本書は、駄菓子屋きぬや販売管理システム構築プロジェクトにおける品質目標、レビュー方針、品質メトリクス、検査基準、是正プロセスを、PM が計画・進捗・課題・リスク管理へ接続できる粒度で定義する最小サンプルである。

## 1. 概要

本計画は、販売、在庫、つけ管理の初期リリースを対象に、成果物、参考資料、レビュー結果、公開前確認の品質管理方針を定義する。PM は品質確認の結果を課題、リスク、変更要求、決定記録へ接続できる単位に分離し、QE と PO が判断できる材料を整える。

採用する Role code の語彙は `pm-organization.md` と `pm-roles.yaml` を正とする。AI Agent は草案作成、差分整理、整合確認を支援してよいが、公開可否、重大不適合、変更要求の採否は人間の PO が判断する。

## 2. 品質目標

| 項目 | 目標値 | 期限 / タイミング | 判定方法 | 責任ロール |
| --- | --- | --- | --- | --- |
| 文書構造適合 | 対象成果物の frontmatter、章構成、必須表が rulebook に適合している | 主要成果物レビュー時 | rulebook、sample、Markdown lint を確認する | PM、QE |
| 文書間整合 | 初期リリース対象、受入条件、RACI、Schedule が矛盾していない | 主要成果物更新時 | 関連文書を横断確認する | PM、ARC、QE |
| レビュー指摘残 | 公開判断時点で重大指摘 0 件 | 公開前 | レビュー記録と課題ログを確認する | QE |
| 公開適性 | 個人情報、非公開情報、認証情報を含まない | 初回公開前 | 公開前レビューと差分確認を行う | PO、QE |
| AI 利用可能性 | rulebook、sample、template が次作業に使える粒度である | 参考資料更新時 | 章構成、制約、記入例を確認する | PM、BA、QE |

## 3. レビュープロセス

### 3.1. レビュー種別

| 種別 | 主な対象 | 実施ロール | 頻度 / タイミング | 出口条件 |
| --- | --- | --- | --- | --- |
| 構造レビュー | frontmatter、見出し、リンク、表 | PM、QE、ARC | 新規成果物作成時 | rulebook とファイル命名に対する不整合がない |
| 内容レビュー | 目的、対象範囲、品質目標、未決事項 | PO、BA、QE | 主要成果物更新時 | 初期リリースの目的と受入条件に矛盾しない |
| 整合性レビュー | RACI、Schedule、管理計画、生成物 | PM、ARC、QE | 関連文書をまたぐ更新時 | 参照先、責任分担、生成元が矛盾していない |
| 公開前レビュー | 公開対象文書、サンプル、README | PO、OPS、QE | 初回公開前 | 個人情報、非公開情報、認証情報、未判断の公開事項がない |

### 3.2. レビュー手順

1. PM はレビュー対象、根拠文書、関連文書、必要な検証を確認する。
2. 実施ロールは rulebook、sample、RACI、Schedule との整合を確認する。
3. 指摘事項はレビューコメント、課題ログ、プロジェクト登録簿のいずれかに記録する。
4. 是正後、QE は重大指摘が残っていないことを確認する。
5. 公開可否または重大不適合の判断が必要な場合は、PM が判断材料を整理し PO へ上げる。

## 4. 品質メトリクス

| 指標 | 算出方法 | 閾値 | 計測頻度 | 報告先 |
| --- | --- | --- | --- | --- |
| Markdown lint 結果 | `npm run -s lint:md` の失敗件数 | 0 件 | 主要成果物更新時 | PM、QE |
| 重大レビュー指摘残 | 公開判断を妨げる未解決指摘件数 | 0 件 | 公開前 | PO、QE |
| 関連リンク不整合 | 存在しない参照または古い成果物名の件数 | 0 件 | 主要成果物更新時 | ARC、QE |
| rulebook 不適合 | 対象成果物が必須章または必須表を満たさない件数 | 0 件 | 新規成果物作成時 | PM、QE |
| 公開不適切情報 | 個人情報、非公開情報、認証情報の検出件数 | 0 件 | 初回公開前 | PO、QE |

## 5. 検査基準と是正

### 5.1. 検査基準

| 観点 | 受入基準 |
| --- | --- |
| 構造 | frontmatter、章構成、必須表、リンクが rulebook に沿っている |
| 内容 | 品質目標、レビュー方針、検査基準、是正方法が読み取れる |
| 整合 | 管理計画、RACI、Schedule、受入条件と矛盾していない |
| 追跡 | 指摘、課題、リスク、変更要求、決定記録へ接続できる |
| 公開 | 個人情報、非公開情報、認証情報を含まない |
| AI 利用 | AI Agent が参照する構造、制約、出力条件が明確である |

### 5.2. 是正プロセス

| 不適合区分 | 例 | 是正方法 | 再判定 |
| --- | --- | --- | --- |
| 軽微 | 誤字、表記ゆれ、軽微なリンク表示 | 通常の文書更新で修正する | PM または QE |
| 構造不適合 | 必須見出し不足、frontmatter 不備 | rulebook に合わせて修正する | QE、ARC |
| 内容不整合 | 受入条件や RACI との矛盾 | 正となる文書を確認し影響範囲を修正する | PO、QE |
| 公開不適合 | 個人情報、認証情報、未判断の公開事項 | 公開対象から除外または内容を修正する | PO、QE |
| 重大 | 初期リリース範囲や公開方針に影響する不備 | 課題または変更要求として扱う | PO |

### 5.3. 生成物の扱い

- `generated` 配下の成果物は、正本から派生する補助成果物として扱う。
- 正本と生成物が矛盾する場合は、正本を修正してから生成物を再作成する。
- 生成物を手修正した場合は、再生成時に失われる可能性があるため、必要な変更は生成元へ反映する。
- 生成物の不整合が公開判断へ影響する場合は、PM が課題として記録する。

## 6. 品質上の役割分担

| Role code | 品質管理上の責務 |
| --- | --- |
| PO | 品質基準、重大不適合、公開可否、変更要求の最終判断を行う |
| PM | 品質確認を計画、進捗、課題、リスク、変更要求へ接続する |
| BA | 利用者視点、受入条件、サンプルの分かりやすさを確認する |
| ARC | 文書体系、ID、命名、依存関係、生成物との整合を確認する |
| QE | rulebook 適合、レビュー指摘、検証結果、公開適性を確認する |
| OPS | 公開準備、配布、運用負荷、外部提供への影響を確認する |

## 8. 見直し条件

| 更新トリガー | 見直し内容 | 責任ロール |
| --- | --- | --- |
| 管理計画の品質方針を変更した | 品質目標、レビュー種別、メトリクスを見直す | PM、QE |
| 初期リリース範囲または受入条件を変更した | 検査基準、内容レビュー、公開前確認の対象を見直す | PO、PM、QE |
| Role code または RACI を変更した | 品質上の役割分担と再判定者を見直す | PM、ARC、QE |
| 重大な不整合が発生した | 是正プロセス、課題化条件、変更要求化条件を見直す | PM、PO、QE |
| 初回公開を行う | 公開前レビュー、公開不適切情報の確認範囲を見直す | PO、OPS、QE |
