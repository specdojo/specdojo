# プロジェクトスコープ: 駄菓子屋きぬや 販売管理システム

## 1. 対象業務

本プロジェクトでは、単一店舗の店頭販売を継続しやすくするため、販売記録、店頭在庫、常連顧客のつけ管理を対象とする。対象利用者は店主代表、家族利用者代表、開発担当とし、店主代表以外が店番する場合でも同じ手順で販売・在庫・つけ残高を確認できる状態を目指す。

- 駄菓子の店頭販売を記録する。
- 入荷、棚卸、廃棄を含む店頭在庫を管理する。
- 常連顧客の簡易情報とつけ残高を確認する。
- 日次の売上、在庫低下、つけ残高を確認する。

## 2. 対象システム

| 対象         | 内容                                                           |
| ------------ | -------------------------------------------------------------- |
| 販売登録     | 商品選択、数量入力、支払い方法選択、販売記録                   |
| 在庫管理     | 入荷登録、棚卸、在庫一覧、在庫低下の確認                       |
| 顧客管理     | 常連顧客の登録、つけ残高の記録、残高確認                       |
| 簡易レポート | 日次売上一覧、在庫低下一覧、つけ残高一覧                       |
| 運用補助     | 店主代表と家族利用者代表が同じ手順を参照できる最小限の操作メモ |

## 3. 対象期間

- 初期対象期間は、最小機能の初期リリースから、販売・在庫・つけ管理の記録が2週間継続できることを確認するまでとする。
- 初期リリース前は、店頭販売、在庫、つけ管理に必要な最小機能と操作手順の整備を対象とする。
- 初期リリース後は、記録継続状況、不足機能、店主代表の操作負荷を確認し、次フェーズの改善候補を整理する。
- 具体的な日付は _UNDECIDED_: プロジェクト計画で決定する。

## 4. スコープ外

| 対象（Out）           | 理由                                                         | 補足                                 |
| --------------------- | ------------------------------------------------------------ | ------------------------------------ |
| 複数店舗管理          | 単一店舗の運用定着を優先するため                             | 運用定着後に次フェーズで検討する     |
| EC サイトやネット注文 | 店頭販売の記録継続を優先するため                             | 店舗運営への効果確認後に要否判断する |
| 本格的な会計連携      | 初期は販売記録とつけ管理の責務に限定するため                 | 会計処理は既存手順で代替する         |
| 仕入れ先システム連携  | 仕入れ判断の基礎データを蓄積する段階であり連携範囲が未確定   | 仕入れ改善の課題整理後に検討する     |
| 複雑なポイント制度    | 店頭手順を増やさず、店主代表が迷わず使えることを優先するため | 必要性が明確になった段階で再検討する |

## 5. 境界の判断基準

初期リリースでは、「店頭での販売・在庫・つけの記録が止まらないこと」を最優先とする。

迷った場合は、次の順で判断する。

1. 店主代表または家族利用者代表が日々使う業務を優先する。
2. 販売・在庫・つけ残高の記録継続に直接必要なものを優先する。
3. 店頭手順を複雑にする機能は初期スコープから外す。
4. 外部連携や高度な分析は、記録が安定した後の次フェーズで扱う。

## 6. スコープ変更方針

| 項目     | 内容                                                                       |
| -------- | -------------------------------------------------------------------------- |
| 変更入口 | 店主代表、家族利用者代表、開発担当が課題一覧または変更要求として提案する   |
| 影響評価 | 店頭手順、操作時間、初期リリース時期、データ移行、再説明の必要性を確認する |
| 承認者   | PO が最終判断し、BA は利用者影響と受入観点を整理する                       |
| 記録先   | DEC、課題一覧、または変更要求ログへ、判断理由と影響範囲を残す              |

初期リリース前にスコープを追加する場合は、追加対象と同時に延期または除外する対象を示し、初期リリースの目的を維持できるかを確認する。
