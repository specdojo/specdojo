# コミュニケーション計画 サンプル

本書は、駄菓子屋きぬや販売管理システム構築プロジェクトにおける進捗、課題、リスク、変更要求、判断事項の報告・連絡・会議体・エスカレーションを、PM が計画化と進捗確認に使える粒度で定義する最小サンプルである。

## 1. 概要

本計画は、販売、在庫、つけ管理の初期リリースに向けた報告経路、情報要求、会議体、証跡、エスカレーション条件を定義する。PM は進捗、課題、リスク、変更要求、決定事項を管理・報告に接続できる単位へ分離し、PO が判断できる材料を整える。

小規模運用では同じ実行主体が複数責務を兼務する場合があるため、本文では責任主体を Role code で記述する。実行主体の割り当ては `pm-members.yaml` を正本とし、本書には人物名、agent 名、個人連絡先を記載しない。

AI Agent は草案作成、差分整理、整合確認、レビュー支援を行ってよい。ただし、最終承認、公開可否、変更要求の採否は人間の PO が判断する。

## 2. ロール別コミュニケーション分類

| Role code | コミュニケーション上の位置づけ | 主な関心事項 | 受信情報 | 発信情報 | 頻度 / タイミング |
| --- | --- | --- | --- | --- | --- |
| PO | 最終判断者 | 目的、優先順位、公開方針、変更要求の採否 | 主要判断事項、重大な課題・リスク、公開前確認結果 | 採否判断、優先順位、公開可否 | 重要判断時、公開前 |
| PM | 計画・進捗・課題・リスクの運用者 | 作業順序、期限超過、課題化、再計画 | 進捗、ブロック理由、レビュー結果 | 進捗報告、再計画案、課題・リスク整理結果 | 作業節目、阻害発生時 |
| BA | 利用者価値・受入条件の確認者 | 店頭運用、利用者向け説明、受入条件 | 目的、対象範囲、利用者向け成果物 | 業務観点の懸念、改善提案 | 利用者向け成果物更新時 |
| QE | 品質基準・検証の確認者 | rulebook 適合、文書間矛盾、公開適性 | レビュー対象、差分、検証結果 | レビュー指摘、公開前確認結果 | 主要成果物更新時、公開前 |
| OPS | 公開・運用影響の確認者 | 公開手順、問い合わせ、継続運用負荷 | 公開候補、運用負荷、利用者向け案内 | 公開準備状況、運用上の懸念 | 公開準備時、運用変更時 |

## 3. 会議体計画

| 会議名 | 目的 | 参加ロール | 頻度 | 成果物 / 証跡 |
| --- | --- | --- | --- | --- |
| 作業計画レビュー | 次に扱う成果物、順序、依存、ブロック理由を確認する | PM、必要に応じて PO、QE | 作業節目 | Schedule、実行結果、課題 |
| 成果物レビュー | 成果物の構造、内容、リンク、rulebook 適合を確認する | PM、QE、観点に応じて BA | 主要成果物更新時 | Pull Request、レビューコメント、修正差分 |
| 課題・リスク確認 | 高優先度の課題、顕在化しそうなリスク、期限超過を確認する | PM、必要に応じて PO、QE | 阻害発生時、作業節目 | PJR、課題ログ、リスク登録簿 |
| 変更要求確認 | 初期リリース範囲、公開方針、店頭試用日への影響を確認する | PM、PO、必要に応じて BA、OPS、QE | 変更要求起票時 | 変更要求ログ、決定記録 |
| 公開前確認 | 公開範囲、検証結果、公開不適切情報、運用負荷を確認する | PO、PM、OPS、QE | 初回公開前 | 公開前チェック結果、決定記録、README |

## 4. 報告計画

### 4.1. 報告種別

| 報告種別 | 受信ロール | 目的 | 期限 / タイミング | 形式 | 責任ロール |
| --- | --- | --- | --- | --- | --- |
| 進捗報告 | PO、関連 Role code | 完了、着手中、ブロック、次アクションを共有する | 作業節目 | 実行結果、Schedule 差分 | PM |
| 課題報告 | PO、PM | 発生済みの阻害要因、対応 Role code、期限を共有する | 課題発生時、期限超過時 | PJR、課題ログ | PM |
| リスク報告 | PO、PM、QE | 高影響リスク、顕在化兆候、対応期限を共有する | リスク識別時、見直し時 | PJR、リスク登録簿 | PM |
| 変更要求報告 | PO、OPS、QE | 変更内容、理由、影響先、採否案を共有する | 変更要求起票時 | 変更要求ログ、決定記録 | PM |
| 公開準備報告 | PO、PM、OPS、QE | 公開可否判断に必要な状態を共有する | 初回公開前 | 公開前チェック結果、README | OPS、必要時 PM |

### 4.2. 報告に含める内容

- 完了した成果物、変更内容、検証結果。
- 未完了の成果物、ブロック理由、次アクション、判断者。
- 初期リリース範囲、店頭試用、品質、公開判断への影響。
- 課題、リスク、変更要求、決定記録へ転記すべき事項。
- 期限超過または依存未解決がある場合の再計画案。

## 5. 連絡チャネルとエスカレーション

### 5.1. 連絡チャネル

| チャネル | 用途 | 使用条件 | 証跡 |
| --- | --- | --- | --- |
| PJR | 課題、リスク、変更要求、決定事項、未決事項の管理 | 状態、優先度、担当、期限、結論を追跡したい場合 | PJR 正本、生成ビュー |
| Pull Request | 成果物変更のレビュー、差分確認、承認 | 複数ファイル変更、公開前確認、レビュー指摘の解消 | PR、レビューコメント、コミット差分 |
| Issue | 外部からの問い合わせ、改善要望、課題受付 | 外部参加や利用者入力を受け付ける場合 | Issue、コメント、関連 PR |
| 実行結果 | タスク実施内容、変更ファイル、検証結果の記録 | exec plan に基づく成果物更新時 | exec result |
| README / ドキュメント | 利用者・貢献者への公開情報提供 | 導入手順、利用方法、貢献方法を知らせる場合 | README、ガイド、サンプル |

### 5.2. エスカレーション

| 区分 | 一次対応 | エスカレーション条件 | 最終判断 | 証跡 |
| --- | --- | --- | --- | --- |
| スコープ変更 | PM | 初期リリース対象、目的、優先順位が変わる | PO | 変更要求ログ、決定記録 |
| Schedule 影響 | PM | 店頭試用日または公開準備へ影響する遅延がある | PM、必要時 PO | Schedule、課題、進捗報告 |
| 品質問題 | QE | rulebook 不適合、文書間矛盾、公開不適切情報がある | PO | レビュー記録、検証結果 |
| 利用者視点の不備 | BA | 受入条件、利用者向け説明、店頭運用に見直しが必要 | PO | レビュー記録、Issue、変更要求 |
| 公開・運用影響 | OPS | 公開手順、問い合わせ、継続運用負荷に影響する | PO | 公開前チェック、決定記録 |
| AI Agent 出力 | PM | 根拠不明、過剰補完、責務境界の誤認がある | PO または成果物の A | 指示内容、生成結果、レビュー結果 |

## 6. 情報管理と公開方針

### 6.1. 証跡管理

- 重要判断は、PJR、決定記録、変更要求、Pull Request、Issue、実行結果のいずれかに残す。
- 課題、リスク、変更要求、決定事項は本文に混在させず、追跡できる単位へ分離する。
- 生成ビューと正本が矛盾する場合は、正本を更新してから生成物を再作成する。
- 報告の責任主体は Role code で記述し、実行主体や兼務割り当ては本書へ複製しない。

### 6.2. 公開情報の扱い

- 公開リポジトリには、人物名、個人連絡先、非公開の組織情報、認証情報、秘密情報を記載しない。
- AI Agent への入力にも、公開できない情報を含めない。
- 外部向け説明は、README、導入ガイド、サンプル、Issue / PR ルールへ集約する。
- 公開前確認では、成果物本文、生成物、差分、添付ファイルに公開不適切情報が含まれていないことを確認する。

## 7. 見直し条件

| 更新トリガー | 見直し内容 | 責任ロール |
| --- | --- | --- |
| `pm-plan.md` の管理方針を変更した | 報告種別、会議体、エスカレーション条件を更新する | PM |
| 採用 Role code または責任分担を変更した | ロール別分類、報告責任、相談先、最終判断者を更新する | PM、ARC |
| 成果物カタログまたは Schedule を大きく変更した | 作業計画レビュー、進捗報告、変更要求報告の対象を更新する | PM |
| 課題滞留、リスク顕在化、変更要求増加が継続した | 報告頻度、エスカレーション条件、PO 判断のタイミングを見直す | PM、PO |
| 外部参加者の Issue / Pull Request 受付を開始した | 外部向けチャネル、応答方針、レビュー導線を追加する | BA、QE、OPS |
| 初回公開または主要リリースを行う | 公開前確認、公開準備報告、運用影響の連絡先を更新する | PO、OPS、QE |

## 9. 継続確認事項

| 論点 | 現状 | 対応方針 | 判断者 |
| --- | --- | --- | --- |
| 外部参加者向け正式チャネル | _UNDECIDED_: Issue / PR の正式運用開始条件は未了 | 初回公開前に受付範囲、応答方針、レビュー導線を決める | PO |
| 進捗報告ファイルの作成頻度 | _UNDECIDED_: 作業節目の実行結果で足りるか未判断 | 報告量が増えた場合に定期進捗報告の形式を決める | PM |
| Release notes の運用 | _UNDECIDED_: 配置、形式、作成責任は未了 | 主要リリース前に公開準備報告とあわせて判断する | PO、OPS |
