# 代替案比較: 駄菓子屋きぬや 販売管理システム

本書は、販売・在庫・つけの記録を継続できる状態へ移行する戦略と、その開発作業を管理する方式を分けて比較し、採択理由を残す最小の完成例である。

## 1. 比較目的と前提条件

| 決定ID | 比較レイヤー       | 決定すること                                 | 決定しないこと       |
| ------ | ------------------ | -------------------------------------------- | -------------------- |
| D-01   | 運営・開発戦略     | 2 週間以内に店番で試せる改善をどう実現するか | 個別の作業管理ツール |
| D-02   | 開発作業の状態管理 | 開発作業、判断、依存の正本をどこに置くか     | 店舗の日次業務手順   |

D-01 を先に決め、D-02 は採択戦略を小規模な開発体制で実行する支援方式として判断する。

根拠資料は、初期対象を定める `prj-scope-sample`、P-01〜P-03 と必要能力を示す `prj-issues-and-approach-sample`、単一店舗・タブレット 1 台・店主代表の最終判断を示す `prj-assumptions-constraints-dependencies-sample` である。

前提条件:

- 初期リリースは単一店舗の販売・在庫・つけ管理に限定する。
- `PO` が最終的な業務判断を行い、2 週間以内に実際の店番で試行を始める。
- 開発担当 1 名から開始し、有償サービスを必須にしない。

候補は方式名から先に選ばず、次のカテゴリと基準から絞った。

| 決定ID | 候補カテゴリ                                      | 候補に含める基準                                             |
| ------ | ------------------------------------------------- | ------------------------------------------------------------ |
| D-01   | 予測型、適応型、業務改善型、ハイブリッド型        | 記録継続と 2 週間以内の試行への道筋を単独で説明できる        |
| D-02   | リポジトリ内管理、Git ホスティング統合、汎用 SaaS | 作業状態、担当、履歴を管理でき、小規模体制で利用を開始できる |

戦略の整理には [PMI Standards](https://www.pmi.org/standards) と [アジャイルソフトウェア開発宣言](https://agilemanifesto.org/iso/ja/manifesto.html)、作業管理候補には [GitHub Issues](https://docs.github.com/en/issues/tracking-your-work-with-issues/learning-about-issues/about-issues)、[Jira](https://www.atlassian.com/software/jira/features/project-management)、[Linear](https://linear.app/docs) の公式資料を使用した。市場占有率は評価していないため、これらを市場の唯一の代表とはみなさない。

## 2. 比較対象案

### 2.1. D-01: 運営・開発戦略

| 案ID   | 案                       | 目的達成への道筋                                                 | 主な構成要素                       | 対象範囲                   |
| ------ | ------------------------ | ---------------------------------------------------------------- | ---------------------------------- | -------------------------- |
| STR-01 | 予測型・計画先行         | 全機能と移行手順を先に確定し、設計・実装・受入を順に完了する     | 全体計画、要件一式、段階レビュー   | 初期対象の全機能           |
| STR-02 | アジャイル・価値提供先行 | 短い反復で動く機能を提供し、店番の結果から次を決める             | 優先順位、短期反復、利用者レビュー | 価値の高い機能から順次     |
| STR-03 | 業務運用改善先行         | 紙の様式と手順を統一し、記録の安定後に必要な範囲をシステム化する | 紙様式、操作手順、日次確認         | 現行の販売・在庫・つけ運用 |
| STR-04 | 最小仕様＋反復           | 記録項目・責任・受入条件を合意し、最小機能を短い反復で改善する   | 最小仕様、受入条件、タブレット記録 | 記録継続に必要な機能と運用 |

### 2.2. D-02: 開発作業の状態管理

| 方式ID | 方式                  | カテゴリ             | 管理対象・責務                 | 強み                               | 主な制約                             |
| ------ | --------------------- | -------------------- | ------------------------------ | ---------------------------------- | ------------------------------------ |
| TSK-01 | リポジトリ内 Markdown | リポジトリ内管理     | 開発作業、判断、完了条件の正本 | 既存環境で開始でき、追加費用がない | 通知と対話的な操作が弱い             |
| TSK-02 | GitHub Issues         | Git ホスティング統合 | Issue、担当、議論、PR 接続     | コード変更と外部協働を接続できる   | リポジトリ内の正本との責務分離が必要 |
| TSK-03 | Jira                  | 汎用 SaaS            | workflow、計画、依存、レポート | 複雑な統制や複数チームへ拡張できる | 現体制には設定・運用負荷が大きい     |
| TSK-04 | Linear                | 製品開発向け SaaS    | Issue、Project、依存、開発連携 | 軽量な製品開発フローを利用できる   | 外部 SaaS と正本の同期が必要         |

## 3. 評価軸と評価基準

### 3.1. D-01 の評価軸

| 評価軸     | 重み | 判定基準                                   | 根拠となる課題・制約             |
| ---------- | ---- | ------------------------------------------ | -------------------------------- |
| 効果       | 5    | P-01〜P-03 を横断して改善できるか          | 記録不統一、残高確認、在庫過不足 |
| コスト     | 3    | 小規模体制で開始・維持できるか             | 開発担当 1 名                    |
| 期間       | 4    | 2 週間以内に店番で試せるか                 | 初期試行期限                     |
| リスク     | 4    | 記録停止、移行漏れ、手戻りを抑えられるか   | 記録継続性                       |
| 運用適合   | 5    | 店主代表と家族利用者代表が継続利用できるか | 利用者負荷                       |
| 技術実現性 | 3    | タブレット 1 台と小規模実装で成立するか    | 端末・体制制約                   |

### 3.2. D-02 の評価軸

| 評価軸         | 判定基準                                           |
| -------------- | -------------------------------------------------- |
| 正本性         | 現在の作業、判断、完了条件を一意に確認できるか     |
| 外部協働       | 担当、議論、通知、コード変更と接続できるか         |
| 導入・運用負荷 | 1 名で開始し、状態を継続更新できるか               |
| 可逆性         | 利用停止後も標準的なファイルから作業を再開できるか |
| 成熟度         | 公式資料と運用方法を確認できるか                   |

評価は `High` / `Middle` / `Low` とし、採択に有利なほど高くする。重みは優先順位を示すが、機械的な合計だけで決定しない。

## 4. 比較結果と採択理由

### 4.1. D-01: 運営・開発戦略

| 案ID   | 効果   | コスト | 期間 | リスク | 運用適合 | 技術実現性 | 総合評価 | 判定     |
| ------ | ------ | ------ | ---- | ------ | -------- | ---------- | -------- | -------- |
| STR-01 | High   | Low    | Low  | Middle | Middle   | Middle     | Middle   | 非採択   |
| STR-02 | Middle | Middle | High | Middle | High     | High       | High     | 一部採択 |
| STR-03 | Middle | High   | High | High   | High     | High       | Middle   | 一部採択 |
| STR-04 | High   | Middle | High | High   | High     | High       | High     | 採択     |

STR-04 を採択し、STR-02 の短期反復と STR-03 の紙による切り戻しを取り込む。最小仕様によって記録方法と受入条件を揃えながら、2 週間以内に操作上の学びを得られるためである。STR-01 は試行開始が遅れるため採択せず、法令・会計連携など事前確定が必要な範囲が生じた場合に再評価する。

### 4.2. D-02: 開発作業の状態管理

| 方式ID | 正本性 | 外部協働 | 導入・運用負荷 | 可逆性 | 成熟度 | 判定           |
| ------ | ------ | -------- | -------------- | ------ | ------ | -------------- |
| TSK-01 | High   | Low      | High           | High   | Middle | 正本として採択 |
| TSK-02 | Middle | High     | High           | High   | High   | 段階採用       |
| TSK-03 | Middle | High     | Low            | Middle | High   | 非採択         |
| TSK-04 | Middle | High     | Middle         | Middle | High   | 非採択         |

TSK-01 を正本とし、TSK-02 は外部参加または PR 連携が必要になった時点で受付・議論に用いる。Jira と Linear は現体制では外部管理と同期の負荷が便益を上回るため採択せず、複数チームの統制や横断レポートが必要になった場合にカテゴリ全体を再評価する。

### 4.3. 技術的実現性と影響

| 決定ID | 技術的実現性                                                     | 主な影響                                    |
| ------ | ---------------------------------------------------------------- | ------------------------------------------- |
| D-01   | 最小仕様、タブレット記録、短期レビューを既存環境で実施できる     | 初期合意と反復ごとの受入確認が必要になる    |
| D-02   | Markdown を正本にし、必要時だけ GitHub Issues と ID で接続できる | 転記条件と Issue のクローズ条件が必要になる |

## 5. リスクとトレードオフ

| 区分         | 内容                                                             | 軽減策                                        |
| ------------ | ---------------------------------------------------------------- | --------------------------------------------- |
| トレードオフ | 初期合意と短期反復の両方を行うため、紙だけの改善より準備が増える | 最小仕様を記録項目、責任、受入条件に限定する  |
| リスク       | タブレット操作に慣れず、営業中の記録が止まる                     | 紙の記録を切り戻し手段として残す              |
| リスク       | Markdown と GitHub Issues で状態が重複する                       | Markdown を正本、Issue を受付・議論と定義する |

## 6. 決定と見直し

- D-01 は最小仕様＋反復、D-02 は Markdown を正本として採択し、GitHub Issues は必要時に段階採用する。
- 最終判断は `PO` が行い、`DEV` は比較分析と実施案の作成を支援する。
- 2 週間の試行で営業日の 90% 以上に記録できない場合は、紙運用の比重と最小仕様を再評価する。
- 開発担当の増加、並行作業、依存見落としが発生した場合は、特定製品を自動採用せず、作業管理カテゴリ全体を再評価する。
- 方針変更は、理由、影響範囲、移行方法とともに Markdown の変更記録へ残す。
