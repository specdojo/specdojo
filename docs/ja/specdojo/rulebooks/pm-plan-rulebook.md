# プロジェクト管理計画 作成ルール

Project Management Plan Documentation Rulebook

本ドキュメントは、プロジェクト全体の計画化、進捗確認、課題・リスク管理、変更管理の方針を定義する `pm-plan.md` を作成・更新するためのルールを定める。構造・必須項目・禁止事項は本 rulebook を正とし、具体的な作り方は recipe、粒度の例は sample、雛形は template を参照する。

## 1. 全体方針

- `pm-plan.md` は、PM が計画、進捗、課題、リスク、変更要求を管理するための上位方針を定義する。
- 管理方針は、タスク化、順序付け、所要時間見積もり、進捗確認、管理台帳への転記に使える粒度で記述する。
- 責任主体は Role code で記述し、採用 Role code の正本は `pm-organization.md` と `pm-roles.yaml` に置く。
- 小規模運用では PM 責務を専任 member に固定しない場合がある。採用 Role code と実行主体の割り当てを混同しない。
- AI Agent は文書作成、差分整理、レビュー支援を担ってよいが、最終承認、公開可否、変更要求の採否、説明責任は人間の判断に残す。
- 品質管理、コミュニケーション、RACI、管理台帳の詳細を本文に過剰展開せず、上位方針と委譲先を明確にする。

## 2. 位置づけと用語定義

`pm-plan.md` は、プロジェクト管理成果物群の中核となる上位計画である。個別計画や管理台帳の内容を複製せず、各管理領域の運用方針、判断責任、逸脱時対応、見直し条件を定義する。

| 文書                            | 役割                                           | `pm-plan.md` との関係                        |
| ------------------------------- | ---------------------------------------------- | -------------------------------------------- |
| `pm-organization.md`            | 採用 Role code と最終判断の集約先を定義する    | 責任主体と `owner` 語彙の前提                |
| `pm-roles.yaml`                 | Role code の語彙を machine-readable に管理する | Schedule `owner`、RACI、判断者に使う値の正本 |
| `pm-members.yaml`               | 実行主体と Role code の対応を管理する          | 専任・兼務・agent 割り当ての正本             |
| `pm-raci.md`                    | 成果物・プロセスごとの責任分担を定義する       | `pm-plan` 自身と管理プロセスの責任分担の正本 |
| `pm-quality-management-plan.md` | 品質目標、レビュー、検証、是正を定義する       | 品質管理の詳細委譲先                         |
| `pm-communication-plan.md`      | 報告、連絡、会議体、エスカレーションを定義する | レポーティング詳細の委譲先                   |
| PJR / 管理台帳                  | 課題、リスク、変更要求、決定記録を管理する     | 本文から分離すべき管理事項の記録先           |

## 3. ファイル命名・ID規則

- 推奨パス: `docs/ja/projects/<project-id>/030-project-management/010-management-plan/pm-plan.md`
- 推奨ファイル名: `pm-plan.md`
- 推奨 ID: `<project-id>:pm-plan`
- `rulebook` は `pm-plan-rulebook` を指定する。
- `based_on` には、少なくとも `<project-id>:pm-organization` と `<project-id>:pm-roles` を含めることを推奨する。

## 6. 記述ガイド

### 6.1. 概要

- 計画の目的、適用範囲、前提となる組織定義、PM と PO の責務境界を 2〜4 文で記述する。
- PM は計画・進捗・課題・リスク・変更要求を管理し、PO は目的・優先順位・公開方針・変更採否などの最終判断を担うことを分けて書く。
- 小規模運用では PM 責務が専任 member に固定されない場合があるため、実行主体の割り当ては `pm-members.yaml` へ委譲する。
- AI Agent の支援範囲と、人間が担う最終判断・説明責任を明記する。

### 6.2. 管理方針（スコープ / スケジュール / コスト）

「基本方針」では、管理領域ごとの管理単位、PM の運用方針、最終判断、逸脱時対応を表で示す。

| 管理領域 | 管理単位 | PM の運用方針 | 最終判断 | 逸脱時対応 |
| -------- | -------- | ------------- | -------- | ---------- |

- スコープ管理では、成果物の追加・削除・統合、対象外事項、未決論点を、変更要求や課題へ分離する方針を書く。
- スケジュール管理では、依存関係、順序、所要時間見積もり、期限超過、Schedule `owner` の Role code 使用を明記する。
- コスト管理では、金額予算が未設定でも、作業負荷、外部依存、運用負荷、再作業負荷を制約として扱う。

コスト管理の推奨フォーマット:

| 観点 | 管理内容 | 判断条件 | 記録先 |
| ---- | -------- | -------- | ------ |

### 6.3. 品質・リスク・課題管理方針

- 品質管理では、構造、一貫性、計画化可能性、公開適性など、PM が計画運用上確認する最低基準を表で示す。
- リスク管理では、将来発生する可能性があり、スコープ、スケジュール、品質、公開判断、継続運用へ影響する事項を記録対象にする。
- 課題管理では、すでに発生しており、対応しなければ作業が進まない事項を記録対象にする。
- 変更管理では、目的、スコープ、主要成果物、Role code、責任分担、公開方針、Schedule へ影響する変更を変更要求として扱う。

推奨フォーマット:

| 記録トリガー | 例  | 初期対応 |
| ------------ | --- | -------- |

| 課題区分 | 判断基準 | PM の初期対応 |
| -------- | -------- | ------------- |

### 6.4. 体制と意思決定

- 採用ロールでは、`pm-organization.md` と `pm-roles.yaml` を正本とし、本書には管理計画での責務の使い分けだけを書く。
- 意思決定では、判断対象、起案・整理、最終判断、相談先、証跡を表で示す。
- RACI では、`pm-raci.md` を正本とし、`pm-plan` 自身の責任分担だけを再掲してよい。

推奨フォーマット:

| 判断対象 | 起案・整理 | 最終判断 | 相談先 | 証跡 |
| -------- | ---------- | -------- | ------ | ---- |

### 6.5. レポーティングと見直し

- レポーティングでは、進捗、課題、リスク、変更要求、品質などの指標、報告タイミング、報告先、閾値超過時対応を表で示す。
- 見直し条件では、組織・ロール変更、スコープ変更、Schedule 変更、課題滞留、公開開始などのトリガーと見直し内容を対応付ける。
- GO / Not GO 判断では、PM が PO に提示すべき確認事項を箇条書きにする。

推奨フォーマット:

| 管理領域 | 指標 | 報告タイミング | 主な報告先 | 閾値超過時対応 |
| -------- | ---- | -------------- | ---------- | -------------- |

| 更新トリガー | 見直し内容 | 主な責任 |
| ------------ | ---------- | -------- |

### 6.6. 関連ドキュメント

- 正本、委譲先、管理台帳を分けて記載する。
- 存在する文書は `[[id|title]]` 形式で記載し、未作成または一般概念として扱う文書はバッククォートで仮置きする。
- 関連文書の内容を本文に丸写ししない。

### 6.7. 継続確認事項

- 未決事項、暫定判断、後続対応、判断者を表で管理する。
- 進捗、課題、リスク、変更要求、決定記録へ転記すべき事項を本文に埋めっぱなしにしない。
- `_TODO_:`, `_UNDECIDED_:`, `_ASSUMPTION_:` を使用する場合は、判断者または次アクションを添える。

推奨フォーマット:

| 論点 | 現状 | 対応方針 | 判断者 |
| ---- | ---- | -------- | ------ |

## 7. 禁止事項

| 禁止事項                                                                               | 理由                                       |
| -------------------------------------------------------------------------------------- | ------------------------------------------ |
| PM の管理方針を書かず、管理領域だけを列挙する                                          | 計画化、進捗確認、課題化に使えないため     |
| `PO`、`PM`、`QE` などの判断責任を曖昧にする                                            | 承認待ち、課題滞留、責任分散が起きるため   |
| `pm-organization.md` や `pm-roles.yaml` にない Role code を判断者や `owner` として使う | 組織定義・ロール定義と矛盾するため         |
| member nickname、人名、agent 名を Schedule `owner` や RACI の責任主体として扱う        | Role code と実行主体を混同するため         |
| 課題、リスク、変更要求、決定事項を本文に混在させ、管理台帳へ分離しない                 | 報告・追跡・再計画ができなくなるため       |
| Agent に最終承認、公開可否、変更要求の採否、説明責任を委ねる                           | 人間の判断責任を代替してしまうため         |
| 個人情報、非公開情報、認証情報、秘密情報を記載する                                     | 公開範囲と安全性に反するため               |
| 実装詳細、DB 定義、API 詳細を本文に含める                                              | プロジェクト管理計画の責務から逸脱するため |
