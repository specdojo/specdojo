# 品質管理計画 作成ルール

Quality Management Plan Documentation Rulebook

本ドキュメントは、プロジェクトにおける品質目標、レビュー方針、品質メトリクス、検査基準、是正プロセスを定義する `pm-quality-management-plan.md` を作成・更新するためのルールを定める。構造・必須項目・禁止事項は本 rulebook を正とし、具体的な作り方は recipe、粒度の例は sample、雛形は template を参照する。

## 1. 全体方針

- `pm-quality-management-plan.md` は、成果物と参考資料を完了・公開候補として扱うための品質目標、レビュー、検証、是正を定義する。
- 内容は PM が計画化、順序付け、進捗確認、課題化、リスク化、変更要求化に使える粒度で記述する。
- 品質確認の実作業は `QE` を中心に行い、重大不適合、公開可否、変更要求の採否などの最終判断は `PO` に残す。
- 品質管理上の責任主体は Role code で記述し、member nickname、人名、agent 名を責任主体にしない。
- AI Agent は草案作成、差分整理、整合確認、レビュー支援を担ってよいが、最終承認、公開可否、重大不適合の採否は人間の判断に残す。
- 品質メトリクスは、算出方法、閾値、計測頻度、報告先を判定可能な形で記述する。

## 2. 位置づけと用語定義

`pm-quality-management-plan.md` は、プロジェクト管理計画が示す品質・リスク・課題管理方針を、レビュー、検証、是正、公開前確認へ落とし込む個別計画である。組織定義や RACI を複製せず、品質管理で必要な責務、判断条件、記録先を定義する。

| 文書                                   | 役割                                               | `pm-quality-management-plan.md` との関係                 |
| -------------------------------------- | -------------------------------------------------- | -------------------------------------------------------- |
| `pm-plan.md`                           | 全体管理方針、品質・リスク・課題管理方針を定義する | 本計画の上位方針                                         |
| `pm-organization.md` / `pm-roles.yaml` | 採用 Role code と責務語彙を管理する                | 品質管理上の責任ロール、相談先、報告先に使う語彙の正本   |
| `pm-raci.md`                           | 成果物・プロセス別の責任分担を定義する             | 品質管理計画自身とレビュー関連プロセスの責任分担の確認元 |
| `pm-communication-plan.md`             | 報告、連絡、会議体、エスカレーションを定義する     | 品質指摘、是正、公開前確認の報告経路                     |
| PJR / 管理台帳                         | 課題、リスク、変更要求、決定事項を管理する         | 不適合や未決事項から派生する記録先                       |
| `generated` 成果物                     | 正本から派生する補助成果物                         | 正本との整合確認と再生成判断の対象                       |

## 6. 記述ガイド

### 6.1. 概要

- 計画の目的、適用範囲、上位計画、品質管理の責務境界を 2〜4 文で記述する。
- PM は品質確認を計画、進捗、課題、リスク、変更要求へ接続することを示す。
- QE は rulebook 適合、文書間矛盾、検証、公開適性を確認し、PO は重大判断と公開可否を判断することを分けて書く。
- AI Agent の支援範囲と、人間が担う最終判断を明記する。

### 6.2. 品質目標

品質目標では、項目、目標値、期限 / タイミング、判定方法、責任ロールを表で示す。目標値は定量値または判定可能な条件で書き、定性表現だけで終わらせない。

推奨フォーマット:

| 項目 | 目標値 | 期限 / タイミング | 判定方法 | 責任ロール |
| ---- | ------ | ----------------- | -------- | ---------- |

扱う観点には、文書構造適合、文書間整合、AI 利用可能性、公開適性、レビュー指摘残、生成物整合を含めることを推奨する。

### 6.3. レビュープロセス

「レビュー種別」では、種別、主な対象、実施ロール、頻度 / タイミング、出口条件を表で示す。

推奨フォーマット:

| 種別 | 主な対象 | 実施ロール | 頻度 / タイミング | 出口条件 |
| ---- | -------- | ---------- | ----------------- | -------- |

「レビュー手順」では、番号付き手順で、対象特定、根拠確認、指摘記録、是正、再判定、PO 判断への接続を記述する。

### 6.4. 品質メトリクス

指標、算出方法、閾値、計測頻度、報告先を表で示す。ツールコマンドが確定している場合は記載してよいが、未確定のコマンドを確定事項として書かない。

推奨フォーマット:

| 指標 | 算出方法 | 閾値 | 計測頻度 | 報告先 |
| ---- | -------- | ---- | -------- | ------ |

扱う指標には、Markdown lint、schema 検証、docs build、重大レビュー指摘残、関連リンク不整合、rulebook 不適合、文書間矛盾、公開不適切情報を含めることを推奨する。

### 6.5. 検査基準と是正

「検査基準」では、観点と受入基準を表で示す。構造、内容、整合、追跡、公開、AI 利用などを分ける。

推奨フォーマット:

| 観点 | 受入基準 |
| ---- | -------- |

「是正プロセス」では、不適合区分、例、是正方法、再判定を表で示す。重大不適合は通常更新に混ぜず、PJR、課題、リスク、変更要求、決定記録へ接続する。

推奨フォーマット:

| 不適合区分 | 例  | 是正方法 | 再判定 |
| ---------- | --- | -------- | ------ |

「生成物の扱い」では、`generated` 成果物と正本の関係、矛盾時の対処、再生成判断、手修正の扱いを記述する。

### 6.6. 品質上の役割分担

- 採用 Role code を行として、品質管理上の責務を記述する。
- 未採用 Role code、member nickname、人名、agent 名を行にしない。
- Role code の定義全文を複製せず、品質管理上の使い分けに絞る。

推奨フォーマット:

| Role code | 品質管理上の責務 |
| --------- | ---------------- |

### 6.7. 関連ドキュメント

- 上位計画、正本、委譲先、管理台帳、生成ビューを分けて記載する。
- 存在するプロジェクト文書は `[[id|title]]` 形式で記載し、未作成または一般概念として扱う文書はバッククォートで仮置きする。
- 関連文書の内容を本文に丸写ししない。

### 6.8. 見直し条件

更新トリガー、見直し内容、責任ロールを表で示す。上位管理方針、成果物カタログ、参考資料、公開方針、採用 Role code、RACI、重大不整合をトリガーに含める。

推奨フォーマット:

| 更新トリガー | 見直し内容 | 責任ロール |
| ------------ | ---------- | ---------- |

### 6.9. 未決事項

- 未決事項、暫定判断、後続対応、判断者を表で管理する。
- _TODO_:, _UNDECIDED_:, _ASSUMPTION_: を使用する場合は、判断者または次アクションを添える。
- 課題、リスク、変更要求、決定記録へ転記すべき事項を本文に埋めっぱなしにしない。

推奨フォーマット:

| 論点 | 現状 | 対応方針 | 判断者 |
| ---- | ---- | -------- | ------ |

## 7. 禁止事項

| 禁止事項                                                         | 理由                                             |
| ---------------------------------------------------------------- | ------------------------------------------------ |
| 品質目標を定性のみで記載する                                     | 判定不能になり、完了判断や公開判断に使えないため |
| メトリクスの算出方法、閾値、計測頻度を省略する                   | 再現性が失われるため                             |
| レビューの出口条件を「問題ないこと」だけで済ませる               | レビュー完了の判断基準が不明確になるため         |
| 是正プロセスや再判定者を未定義にする                             | 不適合が放置されるため                           |
| 課題、リスク、変更要求、決定事項を本文に混在させる               | 管理台帳で追跡できなくなるため                   |
| member nickname、人名、agent 名を責任主体として使う              | Role code と実行主体を混同するため               |
| Agent に最終承認、公開可否、重大不適合の採否を委ねる             | 人間の判断責任を代替してしまうため               |
| 個人名、個人連絡先、非公開組織情報、認証情報、秘密情報を記載する | 公開範囲と安全性に反するため                     |
| 実装詳細、DB 定義、API 詳細を本文に含める                        | 品質管理計画の責務から逸脱するため               |
