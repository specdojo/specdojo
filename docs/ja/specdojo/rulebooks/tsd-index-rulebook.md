# 技術スタック定義（TSD Index）作成ルール

Technology Stack Definition Index (TSD Index) Documentation Rules

`tsd-index.md` は、プロダクトが採用する技術の **全体インベントリ（一覧表）** を一箇所に集約する入口ドキュメントである。採用技術の担当責務を関係者間で合意するための最小限の一覧を定義し、詳細は `tsd-<term>.md` にリンクで繋ぐ。

## 1. 全体方針

- 1プロジェクト（システム）に対して `tsd-index.md` は1つだけ作成する
- 採用技術の全区分を漏れなく記載し、詳細ドキュメント（`tsd-<term>.md`）への入口とする
- 表形式（Markdown）で管理し、機械判読用途には使わない
- ベンダー固有の接続情報・機密値・バージョン固定値は記載しない
- 採用が確定した技術のみを記載し、候補・検討中は含めない

## 2. 位置づけ

`tsd-index` と `tsd-<term>` の関係：

| ドキュメント | 役割                               | 粒度                          |
| ------------ | ---------------------------------- | ----------------------------- |
| `tsd-index`  | 採用技術の全体インベントリ（入口） | 区分・技術名・責務を1行で表す |
| `tsd-<term>` | 特定技術領域の詳細設定・構成       | 複数セクションで詳述          |

`tsd-index` は全区分を俯瞰でき、必要に応じて `tsd-<term>` へリンクする。詳細設定・手順・コードブロックは `tsd-<term>` に委ねる。

## 3. ファイル命名・ID規則

- ファイル名: `tsd-index.md`（固定）
- `id`: `tsd-index`（固定）
- 配置先: プロジェクトのアーキテクチャディレクトリ（例: `030-architecture/020-infrastructure/`）

## 4. 推奨 Frontmatter 項目

| 項目       | 値                               | 必須 |
| ---------- | -------------------------------- | ---- |
| `id`       | `tsd-index`                      | ○    |
| `type`     | `architecture`                   | ○    |
| `status`   | `draft` / `ready` / `deprecated` | ○    |
| `rulebook` | `tsd-index-rulebook`             | 推奨 |
| `based_on` | 根拠となる仕様 ID（配列）        | 任意 |

## 5. 本文構成（標準テンプレ）

| 章番号 | 章タイトル       | 必須 | 説明                              |
| ------ | ---------------- | ---- | --------------------------------- |
| 1      | 技術スタック一覧 | ○    | 全区分を網羅する一覧表（表1つ）   |
| 2      | 補足             | 任意 | `tsd-<term>` へのリンク・注意事項 |

- `技術スタック一覧` は表1つで構成する（区分ごとに複数表に分割しない）
- `補足` には詳細ドキュメントへのリンクを記載し、技術固有の説明は書かない

## 6. 記述ガイド

### 6.1. 技術スタック一覧表

必須列：

| 列名     | 説明                                                               |
| -------- | ------------------------------------------------------------------ |
| 区分     | カテゴリ（例: 言語 / ランタイム / DB / ホスティング）              |
| 採用技術 | 技術名とメジャーバージョン（例: `TypeScript 5.x`、`Node.js 24.x`） |
| 担当責務 | その技術を何に使うか（判定可能な粒度で1行）                        |

必要に応じて `補足` 列を追加してよい。

区分の推奨値：

- 言語 / ランタイム
- CLI フレームワーク / Web FW
- DB / MQ / Cache
- スキーマ検証 / Markdown lint / コードフォーマット
- ドキュメントサイト / 図解生成
- Git フック / コミット規約
- ソースコード管理 / CI/CD / パッケージ配信
- AI エージェント実行 / ローカル LLM

### 6.2. 補足

- 詳細ドキュメント（`tsd-<term>.md`）が存在する場合はリンクを記載する
- 補足に技術の設定手順・構成詳細を記述しない（`tsd-<term>` に委ねる）

## 7. 禁止事項

| 禁止事項                                                            | 理由                                   |
| ------------------------------------------------------------------- | -------------------------------------- |
| 同じ技術を複数の区分に重複記載する                                  | 一覧の見通しが悪くなるため             |
| 担当責務を空欄にする                                                | 合意の根拠がなくなるため               |
| 接続情報・API キー・機密 URL を記載する                             | セキュリティリスクになるため           |
| `tsd-<term>` の詳細内容（設定手順・コードブロック）を本文に展開する | インデックスの責務を逸脱するため       |
| 採用が確定していない技術を記載する                                  | 前提条件として誤認されるため           |
| `tsd-index` を複数作成する                                          | 全体インベントリの SSOT が分散するため |
