# ステークホルダー登録簿 作成ルール

Stakeholder Register Documentation Rulebook

本書は、プロジェクトに影響する、または影響を受ける関係者を、業務価値と利用者影響の観点で一貫して整理する規約です。関係者との合意対象と関与方針を明確にし、後続のコミュニケーション統制で利用できる入力にします。

## 1. 全体方針

- 対象は、プロジェクトの意思決定、実行、利用、運用、評価または影響受容に関わる個人、組織、集団である。
- 目的は、誰が何に関心を持ち、どの程度の影響を持つかを可視化し、合意漏れと利用者影響の見落としを防ぐことである。
- 本文では、関係者の識別、影響度/関心度、関与方針、情報要求、見直し条件を一組として管理する。
- 詳細な会議体、配信頻度、議事録の運用はコミュニケーション計画へ委譲し、本書ではその入力となる要求だけを扱う。
- ステークホルダーは人または組織との関係を示す。タスクの担当責務を示す Role とは区別し、対応する Role がある場合だけ対応 Role code として併記する。
- AI Agent やツールなどの実行主体は、単独のステークホルダーとしては扱わず、入力・制約・確認責任を持つ人または組織の関係者として記載する。

## 2. 位置づけと用語定義

### 2.1. 位置づけ

ステークホルダー登録簿は、プロジェクト概要で示した目的、利用者、期待効果を受けて、合意対象と関与方針を具体化する文書である。後続のコミュニケーション計画は、本書の情報要求、合意・報告の必要性、証跡要件を入力として詳細化する。

### 2.2. 用語定義

| 用語                 | 定義                                                                                 |
| -------------------- | ------------------------------------------------------------------------------------ |
| ステークホルダー     | プロジェクトの成果に影響を与える、または成果・進め方から影響を受ける個人、組織、集団 |
| 関与区分             | 意思決定者、利用者、実行担当者、外部協力者、影響受容者など、関係者の立場を表す分類   |
| 影響度               | 方針、範囲、優先順位、利用、運用、合意形成に与えられる影響の強さ                     |
| 関心度               | 成果、進捗、利用条件、変更に対して継続的に関与する必要性または意向の強さ             |
| エンゲージメント方針 | 現状から目標とする関与状態へ近づけるための情報提供、相談、合意、巻き込みの進め方     |
| 必要な合意           | 着手、変更、公開、利用、運用のために当該関係者と確認または承認する事項               |

## 3. ファイル命名・ID規則

### 3.1. 配置（推奨）

- `docs/ja/projects/<project-id>/020-project-definition/prj-stakeholder-register.md` に配置する。
- 個人連絡先、非公開の利害情報、詳細な連絡網は、アクセス制限された管理資料へ分離する。
- 関与方針の証跡は、議事録、決定記録、Issue、Pull Request など、実際に管理する文書またはツールへ残す。

### 3.2. ドキュメントID（推奨）

- 推奨: `<project-id>:prj-stakeholder-register`
- 例: `prj-0001:prj-stakeholder-register`
- ID はプロジェクト内で一意にし、関係者の ID と混同しない。

### 3.3. ファイル名（推奨）

- 推奨: `prj-stakeholder-register.md`
- 関係者 ID は `STH-<KEY>` 形式とし、同じ関係者を全ての表で同じ ID で参照する。
- 個人名ではなく、役割名、組織名、または集団名を優先して識別する。

## 4. 推奨 Frontmatter 項目

### 4.1. 設定内容

| 項目       | 説明                                                  | 必須 |
| ---------- | ----------------------------------------------------- | ---- |
| id         | `<project-id>:prj-stakeholder-register` 形式の文書 ID | ○    |
| type       | `project`                                             | ○    |
| status     | `draft` / `ready` / `deprecated`                      | ○    |
| rulebook   | `prj-stakeholder-register-rulebook`                   | ○    |
| based_on   | 関係者の根拠にした概要、憲章など                      | 任意 |
| supersedes | 置き換えた旧文書 ID                                   | 任意 |

### 4.2. 推奨ルール

- `based_on` には、本文の関係者、期待、合意対象を判断した直接根拠だけを列挙する。
- 個人名、メールアドレス、電話番号、住所、非公開の組織情報、機密性の高い利害関係は記載しない。
- 対応 Role code は、プロジェクトで採用済みであることを確認できる場合だけ記載する。該当しない外部集団には割り当てない。
- 変更時は、関係者の追加・削除・関与変更のいずれかと、見直しの証跡を残す。

## 5. 本文構成（標準テンプレ）

ステークホルダー登録簿は次の順序で構成する。

| 番号 | 見出し                 | 必須 | 内容（要点）                                                          |
| ---- | ---------------------- | ---- | --------------------------------------------------------------------- |
| 1    | 関係者一覧             | ○    | ID、関与区分、所属/組織、対応 Role code、主な責任、利用場面または備考 |
| 2    | 影響度/関心度分析      | ○    | 影響度、関心度、期待、懸念、必要な合意、評価根拠                      |
| 3    | エンゲージメント方針   | ○    | 現状、目標、対応方針、責任者、期限、証跡                              |
| 4    | コミュニケーション要件 | ○    | 情報要求、希望チャネル、合意・報告の必要性、証跡要件、計画への反映    |
| 5    | 見直し条件             | ○    | 更新トリガー、見直し内容、見直し責任者、承認者、証跡                  |

## 6. 記述ガイド

### 6.1. 関係者一覧

- 関係者を、意思決定、実行、利用、運用、外部協力、影響受容のいずれに関わるかで識別する。
- 1 行には、関係者の主な責任と、プロジェクトへの関わり方が分かる備考を記載する。
- ステークホルダー ID は全章で再利用し、同じ対象を別名で重複登録しない。

推奨表:

| ID  | 関係者 | 関与区分 | 所属/組織 | 対応 Role code | 主な責任 | 備考 |
| --- | ------ | -------- | --------- | -------------- | -------- | ---- |

### 6.2. 影響度/関心度分析

- 影響度と関心度は High / Medium / Low を使い、方針・範囲・利用・合意形成への影響を根拠として示す。
- 各関係者について、期待、懸念、必要な合意を一組で記載する。
- High の関係者は、必要な合意または関与方針を空欄にしない。

| 評価   | 影響度の基準                                       | 関心度の基準                                   |
| ------ | -------------------------------------------------- | ---------------------------------------------- |
| High   | 方針、範囲、優先順位、公開・利用可否を直接左右する | 成果または進行に継続的な確認・判断を必要とする |
| Medium | 一部の成果物、利用、運用、合意形成に影響する       | 節目または変更時に確認・調整を必要とする       |
| Low    | 影響は限定的または間接的である                     | 通知または必要時の共有で足りる                 |

### 6.3. エンゲージメント方針

- 現状と目標を対比し、差分を埋める対応方針を記載する。
- 対応方針には、目的が情報提供、相談、合意形成、承認依頼、巻き込み、リスク低減のどれか分かる行動を書く。
- 責任者、期限、証跡を記載し、未確定の場合は `_TODO_:`、`_UNDECIDED_:`、`_ASSUMPTION_:` を用いる。

推奨表:

| ID  | 関係者 | 現状 | 目標 | 対応方針 | 責任者 | 期限 | 証跡 |
| --- | ------ | ---- | ---- | -------- | ------ | ---- | ---- |

### 6.4. コミュニケーション要件

- 情報要求は、受け取る情報と、その情報で行う判断または確認が分かる粒度で書く。
- 希望チャネルは、会議体や配信頻度の詳細ではなく、対面、Issue、文書、公開ページなどの手段を示す。
- 詳細なコミュニケーション計画を作る場合は、本章をその入力とし、同じ配信計画を二重管理しない。

推奨表:

| ID  | 関係者 | 情報要求 | 希望チャネル | 合意・報告の必要性 | 証跡要件 | コミュニケーション計画への反映 |
| --- | ------ | -------- | ------------ | ------------------ | -------- | ------------------------------ |

### 6.5. 見直し条件

- 概要、範囲、体制、公開方針、主要な利用者影響が変わるときは見直す。
- 定期見直しを行う場合は、周期だけでなく、確認する内容と承認者を明記する。
- 見直しの結果は、変更差分、決定記録、Issue、Pull Request のいずれかで追跡可能にする。

推奨表:

| 更新トリガー | 見直し内容 | 見直し責任者 | 承認者 | 証跡 |
| ------------ | ---------- | ------------ | ------ | ---- |

## 7. 禁止事項

| 項目                                                                  | 理由                                                           |
| --------------------------------------------------------------------- | -------------------------------------------------------------- |
| 関係者の役割・責任・関与区分を記載しない                              | 調整先と利用者影響を判断できないため                           |
| 影響度または関心度を根拠なく評価する                                  | 対応優先度を説明できないため                                   |
| 必要な合意を空欄にする                                                | 意思決定者や合意対象を識別できないため                         |
| ステークホルダー ID をタスクの owner として使う                       | ステークホルダーと責務 Role の境界が崩れるため                 |
| 未採用の Role code を推測で記載する                                   | 組織・責務定義との不整合を招くため                             |
| 個人連絡先、非公開情報、機密性の高い利害関係を記載する                | 公開時の情報管理リスクを高めるため                             |
| 詳細な会議運営や配信計画を重複管理する                                | コミュニケーション計画との二重管理になるため                   |
| AI Agent やツールなどの実行主体を単独のステークホルダーとして記載する | 人または組織との関係として整理できず、責任境界が曖昧になるため |
