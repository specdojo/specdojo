# プロジェクト概要 作成ルール

Project Overview Documentation Rulebook

本ドキュメントは、**プロジェクト概要（Project Overview）** を統一形式で記述するためのルールです。
関係者が「何を、なぜ、どの考え方で進めるのか」を短時間で合意できる粒度に整えます。

## 1. 全体方針

- 本ルールの対象は **プロジェクト概要**（背景、必要性、基本方針、狙いの共有）です。
- 目的は「プロジェクトの立ち上げ判断と後続成果物作成の前提を揃える」ことです。
- 背景、必要性、基本的な考え方、実現したいこと、期待効果、前提条件を概要レベルで記述します。
- 詳細なスコープ、成功基準、受入条件、設計、実装、テスト設計は、本書の対象外とします。
- 正式認可、権限委譲、承認証跡などのガバナンス情報は [prj-charter-rulebook.md](prj-charter-rulebook.md) を正とし、本書では扱いません。
- 具体的な成功判定や受入条件は [prj-success-criteria-and-acceptance-criteria-rulebook.md](prj-success-criteria-and-acceptance-criteria-rulebook.md) を正とし、本書では期待効果と初期仮説に留めます。
- 曖昧表現（例:「適切に」「可能な限り」「十分に」）は避け、合意できる表現にします。
- 数値目標を置く場合は、測定方法、判定時期、初期仮説か確定目標かを明示します。

## 2. 位置づけと用語定義

### 2.1. 位置づけ（他ドキュメントとの関係）

ドキュメント間の関係は [ドキュメント構成ガイド](../guides/docs-structure-guide.md) の立ち上げフロー図を参照してください。本書（`prj-overview`）は、ステークホルダー登録簿（`prj-stakeholder-register`）とプロジェクト憲章（`prj-charter`）の起点となる構想・概要文書です。

### 2.2. 用語定義（本ルール内）

| 用語               | 定義                                                                 |
| ------------------ | -------------------------------------------------------------------- |
| **導入文**         | 対象プロジェクトまたはフレームワークを一文で説明する冒頭の定義       |
| **背景**           | 現状と問題の文脈（事実・観察・状況）                                 |
| **必要性**         | なぜ取り組む必要があるか（放置リスク、外部要因、構造的課題）         |
| **基本的な考え方** | プロジェクトを進めるうえで重視する原則、判断軸、設計思想             |
| **実現したいこと** | プロジェクトで整備・提供したい状態や成果の方向性                     |
| **期待効果**       | 実現したいことにより得られる短期・中長期の効果                       |
| **短期指標**       | 当該プロジェクト内または初期公開時点で確認する品質、コスト、納期指標 |
| **中長期指標**     | 公開後や利用拡大後に確認する波及効果の指標                           |
| **前提条件**       | 変えないもの、依存するもの、適用時に置く条件                         |
| **初期仮説**       | 実績に基づき継続的に見直す前提で置く暫定的な目標値や効果仮説         |

## 3. ファイル命名・ID規則

### 3.1. 配置（推奨）

- `docs/ja/projects/<project-id>/` のように、プロジェクト単位のフォルダ配下に配置します。
- 配下の章ディレクトリは `020-project-definition/` を用います（構成は [docs-structure-guide.md](../guides/docs-structure-guide.md) に従う）。
- 章ディレクトリには、サブディレクトリを切って、プロジェクトの背景や目的に関わる関連資料（図表、参考資料など）を置いてもよい。

### 3.2. ドキュメントID（推奨）

メタ情報（Frontmatter）を付ける場合、ID は一意である必要があります。

- 推奨: `<project-id>:prj-overview`
  - 例: `prj-0001:prj-overview`
- YAML Frontmatter では、必要に応じてコロンを含む ID をクォートします。

### 3.3. ファイル名（推奨）

ファイル名は`prj-overview.md` を推奨。日本語の場合は、`プロジェクト概要.md` 。

## 4. 推奨 Frontmatter 項目

### 4.1. 設定内容

Frontmatter は共通スキーマに従います（あわせてドキュメントのメタ情報の記述ルールも参照）。

- 参照スキーマ: [docs/specdojo/schemas/v1/deliverable-frontmatter.schema.yaml](../../../specdojo/schemas/v1/deliverable-frontmatter.schema.yaml)
- メタ情報標準: [document-metadata-standard.md](../standards/document-metadata-standard.md)

| 項目       | 説明                               | 必須 |
| ---------- | ---------------------------------- | ---- |
| id         | `<project-id>:prj-overview`        | ○    |
| type       | `project` 固定                     | ○    |
| status     | `draft` / `ready` / `deprecated`   | ○    |
| rulebook   | `prj-overview-rulebook`            | ○    |
| based_on   | 既存の経営方針・制度資料・過去 DEC | 任意 |
| supersedes | 置き換え関係                       | 任意 |

### 4.2. 推奨ルール

- `based_on` は、既存の経営方針・制度資料・過去 DEC など、根拠として直接参照するものがある場合のみ列挙します。
- ドキュメント名は Frontmatter ではなく本文 H1 に記述します（例: `# プロジェクト概要: <プロジェクト名>`）。
- H1 直下には、対象プロジェクトまたはフレームワークを一文で説明する導入文を置きます。

## 5. 本文構成（標準テンプレ）

プロジェクト概要は以下の見出し構成を基本とし、順序固定で配置します。

### 5.1. プロジェクト概要（Project Overview）

| 番号 | 見出し         | 必須 | 内容（要点）                                           |
| ---- | -------------- | ---- | ------------------------------------------------------ |
| 冒頭 | 導入文         | ○    | 対象プロジェクトまたはフレームワークの一文定義         |
| 1    | 背景           | ○    | 現状、文脈、構造的な問題                               |
| 2    | 必要性         | ○    | なぜ取り組む必要があるか、放置した場合の課題           |
| 3    | 基本的な考え方 | ○    | 進め方の原則、判断軸、工程や成果物に対する考え方       |
| 4    | 実現したいこと | ○    | 整備・提供したい成果、対象利用者、適用領域             |
| 5    | 期待効果       | ○    | 短期・中長期の効果、指標、初期仮説、測定方法           |
| 5.1  | 短期指標       | 任意 | 初期公開や当該プロジェクト内で確認する QCD 等の指標    |
| 5.2  | 中長期指標     | 任意 | 公開後の利用、継続性、参加機会、透明性等の波及効果指標 |
| 6    | 前提条件       | ○    | 依存条件、制約、変えないこと、適用上の前提             |

## 6. 記述ガイド

### 6.1. 共通

- 見出し配下は「箇条書き＋必要最小限の表」を基本とします。
- “事実/観察” と “推測/仮説” を混ぜないでください（混ざる場合はラベルで区別）。
- 読み手が「次に何を決めればよいか」が分かるように、未確定は未確定として明示します。
- 本書は概要文書であるため、後続文書で詳細化する内容は要約に留め、必要に応じて文書名を示します。

### 6.2. 導入文

- H1 直下に、対象プロジェクトまたはフレームワークを一文で説明します。
- 「何を通じて」「誰を支援し」「何を実現するか」が分かる粒度にします。
- 導入文では詳細な効果、指標、実装方式を列挙しません。

### 6.3. 背景

- 何が起きているか（現状）と、何が困りごとか（問題）を分けて書きます。
- 推奨: 事実は具体例（頻度・件数・影響）を添える。

### 6.4. 必要性

- 期限、制度/契約、競合、障害、属人化など「今やる理由」を明確にします。
- 背景で述べた可能性や課題に対し、なぜ既存のやり方では足りないかを説明します。
- 問題を列挙する場合は、後続の「基本的な考え方」や「実現したいこと」につながる粒度にします。

### 6.5. 基本的な考え方

- プロジェクトの設計思想や判断軸を 2〜5 件程度に絞って記述します。
- 原則には、名称、短い説明、プロジェクト上の意味を含めます。
- 特定の開発手法、ツール、製品に過度に寄せず、後続文書で再利用できる表現にします。

### 6.6. 実現したいこと

- 整備・提供したい成果を箇条書きで記述します。
- 対象利用者、適用領域、公開・再利用方針を必要に応じて含めます。
- 詳細な対象範囲や対象外は `prj-scope` に委譲し、本書では概要に留めます。

### 6.7. 期待効果

- 可能なら、以下のような最小の計測定義を置きます（不明なら空欄ではなく「未確定」）。
- 短期的に判定できる効果と、公開後に追跡する中長期的な効果を分けます。
- 数値目標は、初期公開時点で判定する値か、ベースライン確定後に判定する値かを明示します。
- 成功基準や受入条件として確定管理する値は `prj-success-criteria-and-acceptance-criteria` に委譲します。

### 6.8. 前提条件

- 変えない前提（例: 運用体制は当面変えない）と、依存前提（例: 他部門の協力が必要）を区別します。
- 人間の判断責任、AI Agent の役割、適用範囲、公開方針、利用時の取捨選択方針などを明示します。

### 6.9. プロジェクト憲章・成功基準との分担

- 本書では、背景・必要性・基本的な考え方・実現したいこと・期待効果を中心に記述します。
- 承認者、承認日、権限委譲、承認経路、証跡リンクは [prj-charter-rulebook.md](prj-charter-rulebook.md) 側で管理します。
- 受入条件、完了定義、GO / Not GO 判断は [prj-success-criteria-and-acceptance-criteria-rulebook.md](prj-success-criteria-and-acceptance-criteria-rulebook.md) 側で管理します。
- 同一情報を両文書に重複記載する場合は、本文を複製せず、要約 + 参照リンクで整合を保ちます。

## 7. 禁止事項

| 項目                              | 理由                                     |
| --------------------------------- | ---------------------------------------- |
| 設計詳細（DB設計、API設計等）     | プロダクト仕様/設計ドキュメントへ委譲    |
| “だいたい/適切に” 等の曖昧表現    | 合意・検証ができない                     |
| 根拠のない数値目標                | 誤った期待形成・炎上の原因になる         |
| 認可・権限委譲の詳細管理          | プロジェクト憲章との責務重複を防ぐため   |
| 受入条件や GO / Not GO 判定の詳細 | 成功基準と受入条件との責務重複を防ぐため |
| 個人名・連絡先等の公開不適切情報  | 公開リポジトリでの利用を想定するため     |
