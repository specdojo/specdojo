# RACI 作成ルール

RACI Documentation Rulebook

本ドキュメントは、プロジェクトの主要成果物および主要プロセスに対する責任分担を定義する `pm-raci.md` を作成・更新するためのルールを定める。採用ロール・実行主体・兼務割り当ての正本は組織系文書に置き、本ルールでは `pm-raci.md` に何を記載するかを定義する。

## 1. 全体方針

- `pm-raci.md` は、成果物・プロセスごとの Responsible / Accountable / Consulted / Informed を、採用済み Role code で表す。
- RACI の列には、プロジェクトの組織定義で採用した Role code のみを使用する。
- 採用済み Role code の一部だけを列に置く場合は、対象外とした理由を適用方針に明記する。
- `A` は原則として 1 成果物・1 プロセスに対して 1 Role code に集約する。複数の `A` が必要な場合は、責任範囲を行分割する。
- PM が計画化、進捗確認、課題・リスク管理に使えるよう、成果物別 RACI だけでなく主要プロセスの責任分担も記載する。
- Agent は `R`、`C` を支援してよいが、`A` は担わない。

## 2. 位置づけと用語定義

`pm-raci.md` は、プロジェクトにおける責任分担マトリクスである。Role code の採用、実行主体、兼務割り当てを再定義せず、責任分担の使い方に限定して記述する。

| 文書 | 役割 | 正本とする内容 |
| --- | --- | --- |
| `pm-organization.md` | 組織構成の方針と採用する責務語彙を定義する | RACI 列に使える Role code の範囲 |
| `pm-roles.yaml` | Role code の語彙を管理する | Schedule の `owner` や RACI に使う Role code |
| `pm-members.yaml` | 実行主体と Role code の対応を管理する | member、agent、兼務割り当て |
| `pm-raci.md` | 成果物・プロセスごとの責任分担を定義する | `R` / `A` / `C` / `I` の割り当て |
| Schedule | 実行タスクの担当と順序を管理する | タスク `owner` と進捗確認単位 |
| PJR | 課題、リスク、変更要求、決定記録を管理する | RACI から派生する管理・報告事項 |

RACI 記号の共通定義は上位標準を参照する。本文では、読み手がマトリクスを解釈するための概要表を置いてよい。

## 6. 記述ガイド

### 6.1. 目的

- 本 RACI を設ける背景と、誰が何に使うかを 2〜4 行で記述する。
- PM が計画化、順序付け、所要時間見積もり、進捗確認、課題・リスク管理に使える責任分担であることを示す。
- 採用 Role code の正本は `pm-organization.md` または同等の組織定義であることを明記する。

### 6.2. 適用方針

- RACI 列に使う Role code の範囲を明示する。
- 採用済み Role code をすべて列に置くか、使用する列だけに絞るかを決め、絞る場合は理由を書く。
- 小規模運用で実行主体が兼務する場合でも、Role code の責務境界は分けて記載する。
- `A` は 1 行につき 1 Role code に限定する。複数の最終責任が混ざる場合は、成果物またはプロセスを分割する。
- Agent、member nickname、人名は列に置かず、実行主体の割り当ては `pm-members.yaml` などの正本に委譲する。

### 6.3. RACI の定義

- RACI 記号の標準定義を参照し、プロジェクト固有の補足がある場合だけ本文に表を置く。
- `A` を「承認」「採否判断」「最終責任」のどれに使うかを、適用方針または表の説明で読み取れるようにする。
- `C` と `I` は、レビュー参加と結果共有の違いが分かる説明にする。

### 6.4. 成果物別 RACI

- 行は成果物名、成果物 ID、または管理単位として使う成果物グループとする。
- 列は採用済み Role code のみとする。
- セルは `R` / `A` / `C` / `I` の組み合わせ、または対象外を示す空欄とする。読み取りやすさを優先し、対象外が多い場合でも `A` は必ず置く。
- 各行に `A` が 1 つだけ存在することを確認する。
- Schedule の `owner` と矛盾しないよう、実作業を担う Role code に `R` を置く。

推奨フォーマット:

| 成果物 | PO | PM | BA | ARC | QE |
| --- | --- | --- | --- | --- | --- |
| `<deliverable-id>` | A | R | C | C | C |

### 6.5. プロセス別 RACI

- 行は、計画化、進捗確認、レビュー、課題・リスク登録、変更要求、公開判断など、PM が管理・報告へ転記する単位で置く。
- 成果物別 RACI と同じ列構成にする。
- 成果物作成、レビュー、変更判断を 1 行に混ぜず、最終責任者が異なる場合は行を分ける。
- 課題、リスク、変更要求、決定記録へ転記すべきプロセスには、PM の `R` または `A` を明示する。

推奨フォーマット:

| プロセス | PO | PM | BA | ARC | QE |
| --- | --- | --- | --- | --- | --- |
| 進捗確認・報告 | I | A/R | C | C | C |

### 6.6. 見直し条件

- 採用 Role code、実行主体、成果物カタログ、WBS、Schedule、運用規模の変更をトリガーに含める。
- 各トリガーには、RACI 列、全行の `A`、Schedule `owner`、課題・リスク・変更要求への影響確認を対応付ける。
- 兼務で判断や実行が滞留した場合は、Role code の追加ではなく責務境界とエスカレーション先を見直す。

推奨フォーマット:

| 更新トリガー | 見直し内容 |
| ------------ | ---------- |

### 6.7. 禁止事項

- 組織定義で採用していない Role code を RACI 列に使わない。
- member nickname、人名、agent 名を RACI 列に使わない。
- Agent に `A` を割り当てない。
- 各行の `A` を省略しない。
- 1 行に複数の `A` を置かない。
- Schedule の `owner`、課題・リスク・変更要求の管理単位と矛盾する責任分担を記載しない。

## 7. 禁止事項

| 禁止事項 | 理由 |
| -------- | ---- |
| 組織定義で未採用の Role code を列に使う | 責任分担の参照先が一致しなくなるため |
| 採用済み Role code の一部だけを列に置き、省略理由を書かない | 読み手が責務の有無と省略を区別できないため |
| member nickname・人名・agent 名を RACI 列に使う | ロールと実行主体を混在させると担当変更時に更新漏れが生じるため |
| Agent に `A` を割り当てる | 最終責任と説明責任の所在が不明確になるため |
| 各行の `A` を省略する、または複数置く | 最終判断者が不明確になるため |
| 成果物作成、レビュー、変更判断を 1 行に混ぜる | `A` の所在がプロセスごとに変わり、管理・報告へ接続しにくくなるため |
| Schedule の `owner` と矛盾する記載をする | 実作業責任と管理責任が乖離するため |
