# コミュニケーション計画 作成ルール

Communication Plan Documentation Rulebook

本ドキュメントは、プロジェクトにおける報告、連絡、会議体、エスカレーション、情報管理を定義する `pm-communication-plan.md` を作成・更新するためのルールを定める。構造・必須項目・禁止事項は本 rulebook を正とし、具体的な作り方は recipe、粒度の例は sample、雛形は template を参照する。

## 1. 全体方針

- `pm-communication-plan.md` は、進捗、課題、リスク、変更要求、判断事項の報告経路と証跡管理を定義する。
- 内容は PM が計画化、順序付け、進捗確認、再計画、課題化に使える粒度で記述する。
- 報告・連絡の責任主体は Role code で記述し、member nickname、人名、agent 名を責任主体にしない。
- 小規模運用では会議体を増やしすぎず、証跡を残すチャネルとエスカレーション条件を明確にする。
- AI Agent は草案作成、差分整理、整合確認、レビュー支援を担ってよいが、最終承認、公開可否、変更要求の採否は人間の判断に残す。
- 個人情報、個人連絡先、非公開組織情報、認証情報、秘密情報を本文に含めない。

## 2. 位置づけと用語定義

`pm-communication-plan.md` は、プロジェクト管理計画が示すレポーティング方針を実運用へ落とし込む個別計画である。組織定義や RACI を複製せず、誰が何を受け取り、何を発信し、どの条件で判断者へ上げるかを記述する。

| 文書                                   | 役割                                       | `pm-communication-plan.md` との関係        |
| -------------------------------------- | ------------------------------------------ | ------------------------------------------ |
| `pm-plan.md`                           | 管理方針、報告指標、見直し条件を定義する   | 本計画の上位方針                           |
| `pm-organization.md` / `pm-roles.yaml` | 採用 Role code と責務語彙を管理する        | 報告先、責任ロール、相談先に使う語彙の正本 |
| `pm-raci.md`                           | 成果物・プロセス別の責任分担を定義する     | 報告責任、承認責任、相談先の確認元         |
| `pm-members.yaml`                      | 実行主体と Role code の対応を管理する      | 具体的な兼務・割り当ての正本               |
| PJR / 管理台帳                         | 課題、リスク、変更要求、決定事項を管理する | 報告・連絡から派生する記録先               |

## 6. 記述ガイド

### 6.1. 概要

- 計画の目的、適用範囲、上位計画、PM と PO の責務境界を 2〜4 文で記述する。
- PM は進捗、課題、リスク、変更要求、決定事項を管理・報告に接続できる単位へ分離することを示す。
- 小規模運用で PM 責務を兼務する場合でも、責任主体は Role code で記述し、実行主体は別文書へ委譲する。
- AI Agent の支援範囲と、人間が担う最終判断を明記する。

### 6.2. ロール別コミュニケーション分類

- 採用 Role code を行として、コミュニケーション上の位置づけ、関心事項、受信情報、発信情報、頻度を記述する。
- すべての採用 Role code を必ず列挙する必要はないが、報告、確認、承認、公開判断に関わる Role code は漏らさない。
- 未採用 Role code、member nickname、人名、agent 名を行にしない。

推奨フォーマット:

| Role code | コミュニケーション上の位置づけ | 主な関心事項 | 受信情報 | 発信情報 | 頻度 / タイミング |
| --------- | ------------------------------ | ------------ | -------- | -------- | ----------------- |

### 6.3. 会議体計画

- 会議名、目的、参加ロール、頻度、成果物・証跡を表で示す。
- 定例会議を置かない場合でも、判断内容と結果をどこに残すかを明記する。
- 会議体は作業節目、成果物レビュー、課題・リスク確認、変更要求確認、公開前確認など、判断や証跡が必要な単位へ分ける。

推奨フォーマット:

| 会議名 | 目的 | 参加ロール | 頻度 | 成果物 / 証跡 |
| ------ | ---- | ---------- | ---- | ------------- |

### 6.4. 報告計画

- 報告種別では、報告種別、受信ロール、目的、期限 / タイミング、形式、責任ロールを表で示す。
- 報告に含める内容では、完了、未完了、ブロック、影響、課題・リスク・変更要求、判断事項を箇条書きにする。
- 課題、リスク、変更要求、決定事項へ転記すべき情報を本文に埋め込まない。

推奨フォーマット:

| 報告種別 | 受信ロール | 目的 | 期限 / タイミング | 形式 | 責任ロール |
| -------- | ---------- | ---- | ----------------- | ---- | ---------- |

### 6.5. 連絡チャネルとエスカレーション

- 連絡チャネルでは、チャネル、用途、使用条件、証跡を表で示す。
- エスカレーションでは、区分、一次対応、エスカレーション条件、最終判断、証跡を表で示す。
- スコープ、Schedule、品質、構成、利用者視点、公開・運用、AI Agent 出力など、判断者が異なるものは区分を分ける。

推奨フォーマット:

| チャネル | 用途 | 使用条件 | 証跡 |
| -------- | ---- | -------- | ---- |

| 区分 | 一次対応 | エスカレーション条件 | 最終判断 | 証跡 |
| ---- | -------- | -------------------- | -------- | ---- |

### 6.6. 情報管理と公開方針

- 証跡管理では、重要判断、課題、リスク、変更要求、決定事項の記録先を明記する。
- 公開情報の扱いでは、公開リポジトリや AI Agent 入力に含めてはいけない情報を具体化する。
- 生成ビューと正本が矛盾した場合は、正本を修正してから再生成する方針を書く。

### 6.7. 見直し条件

- 更新トリガー、見直し内容、責任ロールを表で示す。
- 管理計画、採用 Role code、責任分担、成果物カタログ、Schedule、外部参加、公開開始、課題滞留、リスク顕在化をトリガーに含める。

推奨フォーマット:

| 更新トリガー | 見直し内容 | 責任ロール |
| ------------ | ---------- | ---------- |

### 6.8. 関連ドキュメント

- 上位計画、管理台帳、生成ビュー、委譲先を分けて記載する。
- 存在する文書は `[[id|title]]` 形式で記載し、未作成または一般概念として扱う文書はバッククォートで仮置きする。
- 関連文書の内容を本文に丸写ししない。

### 6.9. 継続確認事項

- 未決事項、暫定判断、後続対応、判断者を表で管理する。
- `_TODO_:`, `_UNDECIDED_:`, `_ASSUMPTION_:` を使用する場合は、判断者または次アクションを添える。

推奨フォーマット:

| 論点 | 現状 | 対応方針 | 判断者 |
| ---- | ---- | -------- | ------ |

## 7. 禁止事項

| 禁止事項                                                         | 理由                                         |
| ---------------------------------------------------------------- | -------------------------------------------- |
| 報告先、責任ロール、最終判断者を曖昧にする                       | 伝達遅延と判断滞留が発生するため             |
| 会議体の目的、頻度、成果物・証跡を未定義にする                   | 会議が形骸化し、決定事項が追跡できないため   |
| エスカレーション条件を「必要に応じて」だけで済ませる             | 判断を上げるタイミングが分からないため       |
| 課題、リスク、変更要求、決定事項を本文に混在させる               | 管理台帳で追跡できなくなるため               |
| member nickname、人名、agent 名を責任主体として使う              | Role code と実行主体を混同するため           |
| Agent に最終承認、公開可否、変更要求の採否を委ねる               | 人間の判断責任を代替してしまうため           |
| 個人名、個人連絡先、非公開組織情報、認証情報、秘密情報を記載する | 公開範囲と安全性に反するため                 |
| 実装詳細、DB 定義、API 詳細を本文に含める                        | コミュニケーション計画の責務から逸脱するため |
