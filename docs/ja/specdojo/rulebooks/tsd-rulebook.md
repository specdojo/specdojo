# 特定領域 技術スタック詳細定義（TSD）作成ルール

Domain-Specific Technology Stack Definition (TSD) Documentation Rules

`tsd-<term>.md` は、特定の技術領域（例: ローカル LLM、CI/CD、DB）の **詳細設定・構成・セットアップ手順** を定義するドキュメントである。`tsd-index.md` が全体インベントリの入口であるのに対し、`tsd-<term>.md` は1つの技術領域に絞った詳細情報を提供する。

## 1. 全体方針

- 1つの `tsd-<term>.md` は1つの技術領域に絞る（複数領域を1ファイルに混在させない）
- 採用理由・設定手順・バージョン・構成詳細を記載し、設計・実装・運用の前提条件とする
- 接続情報・API キー・機密値は記載しない
- 技術名は一般的な表記に揃える（例: `Ollama`、`Node.js`）
- 実装クラス名・関数名・SQL 全文は記載しない

## 2. 位置づけ

`tsd-<term>` は `tsd-index` から参照され、特定領域の詳細を補完する：

| ドキュメント | 役割                     | 例              |
| ------------ | ------------------------ | --------------- |
| `tsd-index`  | 全体インベントリ（入口） | `tsd-index.md`  |
| `tsd-<term>` | 特定技術領域の詳細定義   | `tsd-ollama.md` |

## 3. ファイル命名・ID規則

- ファイル名: `tsd-<term>.md`（`<term>` は技術領域を表す英小文字・ハイフン）
- `id`: ファイル名から `.md` を除いた値（例: `tsd-ollama`）
- 命名例: `tsd-ollama.md` / `tsd-ci-cd.md` / `tsd-database.md`
- 配置先: `tsd-index.md` と同じディレクトリ（例: `030-architecture/020-infrastructure/`）

## 4. 推奨 Frontmatter 項目

| 項目       | 値                               | 必須 |
| ---------- | -------------------------------- | ---- |
| `id`       | `tsd-<term>`（例: `tsd-ollama`） | ○    |
| `type`     | `architecture`                   | ○    |
| `status`   | `draft` / `ready` / `deprecated` | ○    |
| `part_of`  | `tsd-index`                      | ○    |
| `rulebook` | `tsd-rulebook`                   | 推奨 |
| `based_on` | 根拠となる仕様 ID（配列）        | 任意 |

## 5. 本文構成（標準テンプレ）

`tsd-<term>` の章構成は技術領域によって柔軟に決めてよい。ただし以下の観点を網羅することを推奨する。

| 観点                   | 説明                                         | 必須 |
| ---------------------- | -------------------------------------------- | ---- |
| 概要・採用理由         | その技術を採用した背景と目的                 | ○    |
| 選定・バージョン       | 採用モデル・バージョン・用途の対応表         | 推奨 |
| 設定・セットアップ手順 | 環境構築・設定手順（コマンド・設定ファイル） | 推奨 |
| 動作確認               | 正常稼働を確認するコマンドや手順             | 任意 |
| 運用上の注意           | 制約・注意事項・トラブルシュート観点         | 任意 |

## 6. 記述ガイド

### 6.1. 概要・採用理由

- その技術領域の目的と採用背景を1〜3文で記載する
- 「何のために」「どこで使うか」を明示する

### 6.2. 選定・バージョン表

用途とモデル・バージョンの対応を表で示す。

推奨列：

| 列名              | 説明                                   |
| ----------------- | -------------------------------------- |
| 用途              | その技術を使うユースケース             |
| 採用技術 / モデル | バージョンまたはエディションを含む名称 |
| 補足              | 選定理由・制約など                     |

### 6.3. 設定・セットアップ手順

- コマンドはコードブロック（` ```bash ` または ` ```sh `）で記載する
- 設定ファイルの内容は対応する言語識別子付きコードブロックで記載する
- 出力例は ` ```text ` で記載する
- ステップが複数ある場合は順序付きリスト（`1.` `2.`）で構成する

### 6.4. 動作確認

- 確認コマンドと期待される出力例をセットで記載する
- 期待出力は ` ```text ` で記載する

## 7. 禁止事項

| 禁止事項                                       | 理由                                     |
| ---------------------------------------------- | ---------------------------------------- |
| 接続情報・API キー・機密 URL を記載する        | セキュリティリスクになるため             |
| 実装クラス名・関数名を記載する                 | 実装詳細であり、TSD の責務を逸脱するため |
| SQL 全文・スキーマ定義を記載する               | SSOT が別にあるため                      |
| `tsd-index` と同じ全区分一覧表を本文に再掲する | 二重管理になるため                       |
| 複数の無関係な技術領域を1ファイルに混在させる  | 参照・保守が困難になるため               |
| 採用が確定していない技術の手順を記載する       | 前提条件として誤認されるため             |
