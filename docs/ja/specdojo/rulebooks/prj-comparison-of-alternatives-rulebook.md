# 代替案比較 作成ルール

Comparison of Alternatives Documentation Rulebook

本ドキュメントは、複数の実行案を比較し、採択理由を説明可能な形で残すためのルールです。
意思決定の再現性と説明責任を担保します。

## 1. 全体方針

- 本ルールの対象は、代替案比較（比較対象、評価軸、採択理由）です。
- 目的は、選択肢の比較過程を可視化し、採択判断を再現可能にすることです。
- 最低 2 案以上を比較し、採択しない案の理由も記載します。
- 比較対象は、同じ意思決定を代替できる一式の戦略案として構成し、原則・施策・ツールを同じ表で混在させません。
- 前段文書からは課題、制約、必要能力、評価条件を引き継ぎ、候補集合は外部調査を含めて独立に構成します。
- 候補名を先に決めず、比較レイヤーごとに候補カテゴリ、選定基準、具体候補の順で絞ります。
- 運営・開発戦略、実装支援、タスク・状態管理など抽象度が異なる判断は、決定 ID を分けて順序を明示します。
- 標準・方法論と製品・ツールを区別し、製品を標準または唯一の代表とみなす場合は客観的根拠を示します。
- 外部ツールを扱う場合は、正本、導入時期、試行条件、移行・撤退可能性を比較します。
- PMBOK 観点では、意思決定記録、リスク比較、制約下でのトレードオフ評価を含めます。
- 評価軸は固定し、案ごとに評価基準を変えません。
- 採択案の最終判断は人間の `PO` が行うことを明記する。AI Agent は比較分析の支援を担ってよい。
- 技術的・方針的な代替案では、実現可能性、外部依存、後続成果物への影響を比較結果に含めます。
- 公開リポジトリに配置する場合は、個人名、個人連絡先、非公開の組織情報を記載しない。

## 6. 記述ガイド

### 6.1. 共通

- 比較対象案は最低 2 案以上記載します。
- 章参照は章番号ではなく章タイトルで記述します。
- 評価結果には定量情報と定性理由の両方を含めます。
- 根拠資料は、前段のスコープ、課題と解決アプローチ、前提・制約など、比較判断に直接必要な文書に限定します。
- 根拠資料との整合は、候補案を前段文書の案に限定することを意味しません。前段文書から課題と評価条件を導き、候補案は外部調査を含めて展開します。
- 複数の判断を扱う場合は `D-01` などの決定 ID を付け、戦略から支援方式へ判断する順序と、各決定で扱わない範囲を示します。
- 同じ採択理由、非採択理由、見直し条件を複数の章や表に重複記載せず、評価結果に集約します。

### 6.2. 比較対象案

- 各案は、単独で目的達成への道筋、適用範囲、主要な進め方を説明できる一式の戦略案にします。
- 相互に組み合わせ可能な原則や施策を別案として並べず、それらを組み込んだ戦略パッケージとして比較します。
- 現行案に有利な候補だけで閉じず、予測型、適応型、ハイブリッド型、最小構成、既存標準・製品など、採択しない可能性がある有力案を含めます。
- ツールやタスク管理方式は、運営・開発戦略と分離して支援方式として比較します。特定ツールが戦略そのものを規定する場合だけ、戦略案として扱います。
- 具体候補を列挙する前に、OSS・ベンダー中立型、IDE 統合型、汎用 SaaS、Agent ネイティブなど、比較目的に合う候補カテゴリと選定基準を定義します。
- 各カテゴリから、公式資料で責務、データ保管、対応環境、移行可能性を確認できる候補を選びます。市場占有率を調べていない場合は「市場を代表する」と断定しません。
- 現行・内製方式、専用ツールを採用しない案、外部の直接競合を必要に応じて含め、特定製品に有利な候補集合にしません。

推奨表（比較対象案）:

| 案ID | 比較レイヤー | 案  | 目的達成への道筋 | 主な構成要素 | 対象範囲 |
| ---- | ------------ | --- | ---------------- | ------------ | -------- |

推奨表（候補カテゴリ）:

| 決定ID | 候補カテゴリ | 候補に含める基準 |
| ------ | ------------ | ---------------- |

### 6.3. 評価軸と評価基準

- 評価軸は効果、コスト、期間、リスク、運用適合、技術実現性の 6 観点を最低ラインとします。
- 観点を省略する場合は省略理由を記載します。
- 技術実現性は、特定技術への過度な依存、既存文書体系との接続、外部依存、後続成果物への影響を見ます。
- 不確実性への適応、トレーサビリティ、人間の判断責任、AI Agent の長期作業と検証可能性が判断に影響する場合は、独立した評価軸として追加します。
- 抽象度が異なる比較レイヤーには同じ評価軸を強制せず、各レイヤー内では候補ごとに同じ軸を使用します。
- 外部ツールには、正本性、ロックイン、可逆性、成熟度、既存資産との重複を必要に応じて追加します。

推奨表（評価基準）:

| 評価軸 | 重み | 判定基準 | 備考 |
| ------ | ---- | -------- | ---- |

### 6.4. 比較結果と採択理由

- 各候補を `採択` / `一部採択` / `比較試行` / `保留` / `非採択` などに判定し、その理由と再評価条件を記載します。
- 判断過程を追跡できるよう、根拠資料へのリンクを記載します。
- 採択案の最終判断は `PO` が行い、判断者を明記する。
- 技術的実現性と影響は、決定または案ごとに表で明示します。
- 比較試行は、同じ入力、評価軸、完了条件、記録先、採用・撤退判定を定義し、試行前に特定ツールを標準扱いしません。
- 外部ツールを採択する場合は、仕様や状態の正本、同期方向、利用停止時の移行方法を明記します。

推奨表（比較結果）:

| 案ID | 効果 | コスト | 期間 | リスク | 運用適合 | 技術実現性 | 総合評価 | 判定 |
| ---- | ---- | ------ | ---- | ------ | -------- | ---------- | -------- | ---- |

推奨表（非採択または一部採択の理由）:

| 案ID | 理由 | 再評価条件 |
| ---- | ---- | ---------- |

推奨表（技術的実現性と影響）:

| 案ID | 技術的実現性 | 主な影響 |
| ---- | ------------ | -------- |

### 6.5. リスクとトレードオフ

- 案ごとのリスクとトレードオフを、採択判断に必要な粒度で記載します。
- リスクは「起きうること」と「軽減策」をセットにします。
- トレードオフは、先送りする範囲、受容する負荷、後続成果物への影響を明示します。

推奨表（リスクとトレードオフ）:

| 区分 | 内容 | 軽減策 |
| ---- | ---- | ------ |

### 6.6. 決定と見直し

- 現時点の採択方針と再評価トリガーを明記する。
- 複数レイヤーを扱う場合は、各決定の採択、試行、保留を短く要約する。
- 方針変更時は変更理由と影響範囲をプロジェクト登録簿または決定記録に残す。

## 7. 禁止事項

| 項目                                                           | 理由                                                 |
| -------------------------------------------------------------- | ---------------------------------------------------- |
| 単一案のみの比較                                               | 意思決定の妥当性が検証できないため                   |
| 同じ戦略の構成要素を別々の代替案として比較する                 | 単独で目的を達成できず、比較結果が全案採択に偏るため |
| 運営戦略と個別ツールを同じ比較レイヤーで採点する               | 抽象度が異なり、評価結果を解釈できないため           |
| 候補カテゴリと選定基準を定めず、知っている製品だけを比較する   | 候補集合が恣意的になり、有力な代替案を見落とすため   |
| 標準・方法論と製品・ツールを同じ意味で扱う                     | 成熟度と適用範囲を誤って評価するため                 |
| 根拠なく特定製品を市場の代表または標準と断定する               | 比較対象の妥当性を説明できないため                   |
| 正本と移行・撤退方法を定めず外部ツールを採択する               | 二重管理とベンダーロックインを招くため               |
| 前段文書の候補案を名称変更して再掲するだけで外部案を調査しない | 結論の追認となり、有力な代替手段を見落とすため       |
| 評価軸なしの採択結論                                           | 判断根拠が不透明になるため                           |
| 非採択理由の未記載                                             | 将来再評価時に経緯追跡できないため                   |
| 根拠資料リンクなし                                             | 説明責任を果たせないため                             |
| Agent に採択案の最終判断を委ねる                               | 人間の判断責任を代替してしまうため                   |
| 公開リポジトリに個人名・個人連絡先・非公開の組織情報を記載する | 公開範囲とプライバシーに反するため                   |
