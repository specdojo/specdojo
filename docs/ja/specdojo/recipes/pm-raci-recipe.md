# RACI 作成レシピ

RACI Writing Recipe

本ドキュメントは、RACI を「PM が計画化、進捗確認、課題・リスク管理、変更判断への接続に使える責任分担表」に仕上げるための作り方である。本書では問い、深掘り手順、レビュー観点を扱う。

## 1. このレシピの使い方

- 最初に、組織定義から RACI 列に使える Role code の範囲を確認する。
- 次に、成果物ごとに作成責任、最終判断、相談先、共有先を分ける。
- さらに、計画化、進捗確認、レビュー、課題・リスク、変更要求、公開判断など PM が管理するプロセスを行にする。
- 最後に、全行の `A` が 1 つだけで、Schedule、課題、リスク、変更要求へ転記すべき事項が読み取れるか確認する。

## 2. 作成前に集める情報

| 項目            | 集める情報                                                             |
| --------------- | ---------------------------------------------------------------------- |
| 採用 Role code  | RACI 列に使える Role code と、列から省く場合の理由                     |
| 成果物一覧      | 管理単位にする成果物 ID、文書名、成果物グループ                        |
| 成果物の責任    | 作成責任、最終判断、相談先、共有先                                     |
| Role code 根拠  | 列に使う Role code の正本と、参照範囲外ならその制約を記録する方針      |
| PM 管理プロセス | 計画化、順序付け、進捗確認、レビュー、課題・リスク、変更要求、公開判断 |
| Schedule 接続   | `owner` と RACI の `R` / `A` が矛盾しないか                            |
| PJR 接続        | 課題、リスク、変更要求、決定記録へ転記すべき責任分担                   |
| 兼務の扱い      | 実行主体が兼務しても Role code の責務境界を分ける方針                  |
| 見直し条件      | 採用ロール、実行主体、成果物、WBS、Schedule、運用規模の変更            |

## 3. 全体の作成手順

1. 組織定義を確認し、RACI 列に置く Role code を決める。
2. H1 直下と目的に、RACI の使い道を PM の管理観点で記述する。
3. 適用方針に、Role code の範囲、`A` の集約、Agent と兼務の扱いを書く。
4. RACI 記号の定義を、標準定義に沿って簡潔に示す。
5. 成果物別 RACI に、主要成果物を行として置き、各行の `A` を 1 つにする。
6. プロセス別 RACI に、PM が管理・報告へ接続するプロセスを行として置く。
7. 見直し条件に、RACI 更新のトリガーと確認する影響範囲を対応付ける。
8. 禁止事項に、未採用 Role code、個人名、Agent の `A`、複数 `A`、Schedule との矛盾を防ぐ記述を置く。

## 4. 各章の書き方

### 4.1. 目的

問い:

- この RACI は、PM がどの管理判断に使うか。
- 成果物作成、レビュー、承認、変更判断の責務境界が分かるか。
- Schedule、課題、リスク、変更要求、決定記録へ接続する単位があるか。
- Role code の実体一覧を参照できるか。できない場合、どこまでを採用済み語彙として扱い、何を _ASSUMPTION_ や result に残すか。

書き方:

- RACI を単なる一覧ではなく、計画化と進捗確認に使う責任分担表として説明する。
- 採用 Role code の正本を明記する。
- Role code や member の一覧を本文に複製しない。
- Role code の正本が参照範囲外にある場合は、推測で埋めずに参照制約と未検証範囲を明示する。

### 4.2. 適用方針

問い:

- RACI 列に置く Role code は、組織定義の採用語彙と整合しているか。
- 実行主体が兼務していても、責務境界を Role code で分けられているか。
- `A` を 1 行 1 つにできない行は、行を分割すべきではないか。

書き方:

- 使用する Role code と省略する Role code がある場合の理由を示す。
- `A` の原則、例外の扱い、Agent の扱いを箇条書きにする。
- member nickname、agent 名、個人名は実行主体の正本へ委譲する。

### 4.3. RACI の定義

問い:

- `R`、`A`、`C`、`I` の読み方に揺れがないか。
- `C` と `I` の違いを、レビュー参加と結果共有として分けられているか。
- 本文内で標準定義と異なる意味を持ち込んでいないか。

書き方:

- 必要最小限の表にする。
- プロジェクト固有の補足は、記号の意味を変えずに追記する。

### 4.4. 成果物別 RACI

問い:

- 主要成果物が、Schedule や進捗確認で管理できる粒度になっているか。
- 各行に `A` が 1 つだけあり、`R` が実作業責任として読めるか。
- レビューや合意に必要な `C`、共有すべき `I` が抜けていないか。

書き方:

- 行は成果物 ID または成果物名で統一する。
- 列は Role code のみを使う。
- 成果物グループを使う場合は、グループで管理する理由が読み取れる名称にする。

### 4.5. プロセス別 RACI

問い:

- PM が進捗、課題、リスク、変更要求、決定記録へ転記すべきプロセスがあるか。
- 成果物作成、レビュー、変更判断、公開判断を 1 行に混ぜていないか。
- `PO` の判断責任、`PM` の管理責任、`QE` のレビュー責任が読み分けられるか。

書き方:

- 「成果物作成」「レビュー」「変更要求」「公開判断」など、最終責任者が異なるプロセスは行を分ける。
- PM が管理する行には `PM` の `R` または `A` を明示する。
- 変更要求の起票・影響整理と採否判断は、責任が異なる場合は分ける。

### 4.6. 見直し条件

問い:

- どの変更が RACI の列、行、`A`、`R` を変えるか。
- Schedule の `owner`、課題・リスク・変更要求の責任分担へ影響するか。
- 兼務による滞留が見えた場合、責務境界をどう見直すか。

書き方:

- 「更新トリガー」「見直し内容」の表にする。
- 採用 Role code、実行主体、成果物、WBS、運用規模の変更を含める。
- 見直し内容には、全行の `A`、Schedule `owner`、PJR への影響確認を含める。
- Role code の正本が別文書へ委譲されている場合は、その文書が参照範囲に含まれるかも確認する。

### 4.7. 禁止事項

問い:

- 未採用 Role code、個人名、agent 名を列に置いていないか。
- Agent に `A` を割り当てていないか。
- 複数 `A` や `A` なしの行がないか。

書き方:

- 禁止事項は、実際に混入しやすい誤りを箇条書きにする。
- Schedule や PJR と矛盾する責任分担を禁止する。
- Role code の実体一覧が参照範囲外で未検証のまま、確定表現で採用済みと断定しない。

## 5. 深掘り手順

1. 成果物を「作る」「確認する」「承認する」「共有する」に分け、Role code を対応付ける。
2. `A` が複数になる行を探し、成果物またはプロセスを分割する。
3. `R` がない行を探し、実作業責任が Schedule の `owner` と矛盾しないか確認する。
4. `C` が多すぎる行を探し、実際にレビューや合意が必要な Role code に絞る。
5. 進捗確認、課題、リスク、変更要求、決定記録に転記するプロセスを追加する。
6. 実行主体の兼務と Role code の責務を混同していないか確認する。

## 6. 良い例 / 悪い例

| 観点       | 良い例                                             | 悪い例                                 |
| ---------- | -------------------------------------------------- | -------------------------------------- |
| Role code  | 組織定義で採用済みの `PM`、`QE` を列に置く         | 実行 agent の nickname を列に置く      |
| `A` の集約 | `pm-raci` は `PM` を `A/R`、`PO` を `C` にする     | `PO` と `PM` の両方を `A` にする       |
| 兼務       | 小規模でも `PO` と `PM` の責務境界を分ける         | 兼務なので全部 `PO` と書く             |
| プロセス   | 変更要求の影響整理と採否判断を別行にする           | 変更対応を 1 行にまとめる              |
| 管理接続   | 進捗確認、課題・リスク登録をプロセス別 RACI に置く | 成果物別 RACI だけで管理責任が読めない |
| 見直し条件 | 採用ロール変更時に列と全行の `A` を確認する        | 必要に応じて見直す                     |

## 7. レビュー観点

| 観点                 | 確認内容                                                         |
| -------------------- | ---------------------------------------------------------------- |
| Role code 整合       | RACI 列が組織定義の採用 Role code と整合しているか               |
| Accountable          | 各成果物・各プロセスに `A` が 1 つだけあるか                     |
| Responsible          | 各行の `R` が Schedule の `owner` や実作業責任と矛盾しないか     |
| Consulted / Informed | レビュー、合意、共有の対象が不足または過剰でないか               |
| 管理接続             | 進捗、課題、リスク、変更要求、決定記録へ接続するプロセスがあるか |
| 正本分離             | member、agent、兼務割り当てを本文に複製していないか              |
| 公開適性             | 個人情報、非公開情報、実行主体の固有名を含んでいないか           |

## 8. 仕上げチェック

- 「全体の作成手順」で示した章立てと必須表がそろっている。
- 成果物別 RACI とプロセス別 RACI の両方がある。
- RACI 列に採用済み Role code だけを使っている。
- 各行の `A` が 1 つだけで、`R` が存在する。
- PM が管理する進捗、課題、リスク、変更要求、決定記録の責務が読み取れる。
- member nickname、人名、agent 名を RACI 列に使っていない。
