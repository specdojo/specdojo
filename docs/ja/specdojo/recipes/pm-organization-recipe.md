# 組織定義 作成レシピ

Organization Definition Writing Recipe

本ドキュメントは、組織定義を「PO が構成の妥当性と最終判断の集約先を判断できる内容」に仕上げるための作り方です。構造と必須項目は [[pm-organization-rulebook|組織定義 作成ルール]] を正とし、本書では問い、深掘り手順、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、プロジェクト概要から目的、スコープ、優先順位、公開方針を抜き出す。
- 次に、その前提を支える最小の責務語彙、兼務の方針、Schedule の `owner` として使える値を決める。
- 最後に、PO の判断範囲、見直し条件、ロール定義・メンバー定義への導線を確認する。

| 種別 | 役割 | 使いどころ |
| --- | --- | --- |
| rulebook | 成果物として成立するための規約 | 構造、必須項目、禁止事項を確認する |
| recipe | 良い内容を書くための作り方 | 問い、深掘り、レビューに使う |
| sample | 完成例 | 粒度、文体、表の書き方を確認する |
| template | 記入の骨組み | 新規作成時に章とプレースホルダを用意する |

## 2. 作成前に集める情報

| 項目 | 集める情報 |
| --- | --- |
| プロジェクトの目的 | 組織構成が実現すべき価値と判断軸 |
| スコープと優先順位 | 初期に必要な活動、後回しにできる活動 |
| 公開方針 | 公開可能な情報、公開可否の最終判断者 |
| 運用規模 | 個人・小規模、複数人、外部関係者の継続参加の有無 |
| 採用する責務語彙 | Schedule の `owner` や RACI の列として使う Role code |
| 兼務の根拠 | 兼務しても判断・実行が滞らない範囲 |
| 分離が必要な兆候 | 判断の滞留、継続運用、公開・運用責任の増加 |
| 下流文書 | Role code の正本、実行主体の対応、必要な責任分担の記録先 |

## 3. 全体の作成手順

1. H1 の直下で、組織定義が支えるプロジェクト目的を一文で示す。
2. 基本方針に、運用規模、兼務方針、ロール定義とメンバー定義の正本、PO の最終判断を記載する。
3. 採用ロールと `owner` 語彙に、使用可能な Role code の正本、採用と専任配置の違い、`owner` に使えない値を記載する。
4. 関連ドキュメントに、プロジェクト概要、ロール定義、メンバー定義、必要時の RACI への導線を置く。
5. 見直し条件に、構成を変える事象、PO が確認する影響範囲、更新先を対応付ける。
6. 禁止事項に、正本の複製、責務の混同、公開不適切情報、AI Agent への最終判断の委任を記載する。
7. 具体的な Role code 一覧、member、兼務割り当てを本文に複製していないことを確認する。

## 4. 各章の書き方

### 4.1. 基本方針

問い:

- この構成は、どの目的、スコープ、優先順位を支えるか。
- 小規模運用で兼務してよい範囲と、その根拠は何か。
- PO は何を判断し、AI Agent はどこまで支援するか。

書き方:

- 実行主体の名前や具体的な割り当てではなく、構成を採用する判断を記述する。
- PO が承認、保留、差し戻しを判断する際に必要な論点を明示する。
- 未決事項は `_UNDECIDED_:` とし、判断者または判断時点を添える。

### 4.2. 採用ロールと owner 語彙

問い:

- Schedule の `owner` として使える Role code の正本はどこか。
- 採用する責務語彙と、専任 member の配置を混同していないか。
- `owner` に使ってはいけない値を明示できているか。

書き方:

- Role code の一覧そのものではなく、正本となる `pm-roles.yaml` と使い方を示す。
- 採用は責務語彙の採用であり、member の専任配置や兼務割り当ては `pm-members.yaml` に委譲すると書く。
- member nickname、agent 名、個人名、未定義 Role code を `owner` にしないことを明記する。

### 4.3. 関連ドキュメント

問い:

- 目的と公開方針はどの文書を参照するか。
- Role code、実行主体、責任分担はそれぞれどこが正本か。
- 読み手が次に詳細を確認できる導線があるか。

書き方:

- 文書名と役割を 1 行ずつ表にする。
- 存在する文書は `[[id|title]]`、未作成の文書はバッククォートで記載する。
- 関連文書の内容を本文に要約・複製しない。

### 4.4. 見直し条件

問い:

- どの事象が発生したら、兼務や判断責任を見直す必要があるか。
- PO は、目的、スコープ、優先順位、公開方針への影響をどう確認するか。
- 変更時に更新すべき正本は何か。

書き方:

- 「更新トリガー」「PO が確認する影響範囲」「更新対象」の表にする。
- 役割を増やすこと自体を目的にせず、滞留や責任の変化をトリガーにする。
- 詳細な RACI は必要な場合だけ別文書に委譲する。

### 4.5. 禁止事項

問い:

- 正本を分散させる重複記載はないか。
- Schedule の `owner` と実行主体を混同していないか。
- 公開できない情報、または AI Agent に委ねられない判断を記載していないか。

書き方:

- 禁止事項は「してはいけない記述」と「防ぐべき理由」の対応が分かるように書く。
- 個人名、連絡先、アクセス情報は書かない。
- AI Agent の役割は支援に限定し、最終承認と公開可否は人間の PO に残す。

## 5. 深掘り手順

1. プロジェクトの目的と公開方針を一文ずつ抜き出す。
2. 初期に必要な判断・作業を列挙し、Schedule の `owner` に必要な責務語彙を確認する。
3. 採用する責務語彙と、実際に担当する member の割り当てを分ける。
4. 判断または実行が滞る条件を洗い出す。
5. 各条件で、PO が確認する影響と更新する正本を対応付ける。
6. PO 以外に最終判断、公開可否、説明責任が移っていないことを確認する。

## 6. 良い例 / 悪い例

| 観点 | 良い例 | 悪い例 |
| --- | --- | --- |
| 構成根拠 | 個人・小規模運用のため兼務し、滞留時に見直す | 少人数なので柔軟に対応する |
| owner 語彙 | `pm-roles.yaml` の Role code だけを `owner` に使う | 実行 agent の nickname を `owner` にする |
| 正本への導線 | Role code は `pm-roles.yaml`、割り当ては `pm-members.yaml` と分ける | 本文にロールと担当者を全て並べる |
| PO の判断 | 公開方針との整合、未決事項、影響範囲を確認する | PO が必要に応じて判断する |
| 見直し条件 | 継続参加者の増加時に兼務と責任分担を確認する | 人が増えたら見直す |
| AI Agent | 草案・レビューを支援し、最終承認はしない | AI Agent が公開可否を決める |

## 7. レビュー観点

| 観点 | 確認内容 |
| --- | --- |
| 目的・スコープとの整合 | プロジェクトの目的、スコープ、優先順位、公開方針を支える構成か |
| 承認判断 | PO が承認、保留、差し戻しを判断する論点、未決事項、影響範囲が分かるか |
| owner 語彙 | Schedule の `owner` に使える値が `pm-roles.yaml` と整合しているか |
| 正本の分離 | Role code、実行主体、責任分担を正本へ委譲できているか |
| 公開可否 | 個人情報・非公開情報を含まず、公開判断が人間の PO に残るか |
| 下流入力 | `pm-roles.yaml` と `pm-members.yaml` を作成・更新できる導線があるか |

## 8. 仕上げチェック

- [[pm-organization-rulebook|組織定義 作成ルール]] の本文構成と禁止事項に従っている。
- [[pm-organization-sample|組織定義 sample]] と同程度の粒度で、具体的な割り当てを複製していない。
- `pm-roles.yaml` と `pm-members.yaml` への導線がある。
- Schedule の `owner` に使える値の正本が明記されている。
- PO の最終判断、公開可否、説明責任が明記されている。
- 見直し条件に、トリガー、影響範囲、更新対象がある。
