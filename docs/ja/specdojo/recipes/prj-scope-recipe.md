# プロジェクトスコープ 作成レシピ

Project Scope Writing Recipe

本ドキュメントは、プロジェクトスコープを「関係者が対象範囲と対象外を合意できる内容」に仕上げるための作り方です。
本書では問い、深掘り手順、具体化のコツ、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、プロジェクト概要や上位方針から「何を実現したいか」を短く抜き出す。
- 次に、対象業務、対象システム、対象期間、スコープ外を分けて書く。
- 最後に、境界の判断基準とスコープ変更方針を置き、迷ったときの扱いを決める。

## 2. 作成前に集める情報

最初から正確に書こうとせず、以下を短いメモで集める。

| 項目               | 集める情報                                                                             |
| ------------------ | -------------------------------------------------------------------------------------- |
| 背景と必要性       | なぜ取り組むか、放置すると何が悪化するか                                               |
| 対象利用者         | 誰が使うか、誰が影響を受けるか                                                         |
| 対象業務           | どの業務、活動、利用場面を対象にするか                                                 |
| 対象システム       | どのシステム、成果物、文書体系、基盤、外部連携を含めるか                               |
| 初期対象期間       | いつからいつまで、またはどのイベントまでを対象にするか。未確定なら _UNDECIDED_: で残す |
| 明示的な対象外     | やらないこと、理由、後続で扱う条件                                                     |
| 境界で迷う項目     | 含めるか外すか判断が必要な候補                                                         |
| 変更時の決め方     | 提案入口、影響評価、承認者、記録先                                                     |
| 下流文書への委譲先 | 成功基準、前提・制約、課題、設計、実装、テストへ委譲する内容                           |

## 3. 全体の作成手順

1. H1 で対象プロジェクト名を明示する。
2. 対象業務で、活動領域、利用場面、利用者影響を書く。
3. 対象システムで、対象となるシステム、成果物、文書体系、基盤を粒度を揃えて並べる。
4. 対象期間で、初期対象、判断イベント、継続改善との切り分けを書く。
5. スコープ外で、やらないこと、理由、後続で扱う条件を書く。
6. 境界の判断基準で、迷ったときの優先順位を定義する。
7. スコープ変更方針で、提案入口、影響評価、承認者、記録先を定義する。
8. 最後に、設計詳細や受入条件を書きすぎていないか確認する。

## 4. 各章の書き方

### 4.1. 対象業務

問い:

- どの業務、活動、利用場面を対象にするか。
- 誰が使うか、誰の負担や判断が変わるか。
- 対象にすることで、どの業務価値に近づくか。
- 何を運用・管理できる状態にしたいか。

書き方:

- 機能名だけでなく、利用者が行う作業や判断で書く。
- 「対象利用者」「利用場面」「利用者影響」を読み取れる粒度にする。
- 業務アプリケーション以外のプロジェクトでは、文書整備、公開準備、運用基盤整備などを活動領域として書く。

良い例:

- 単一店舗の店頭販売、店頭在庫、常連顧客のつけ管理を対象にし、店主代表以外が店番する場合でも同じ手順で確認できる状態を目指す。

悪い例:

- 販売管理機能を作る。

### 4.2. 対象システム

問い:

- 対象となるシステム、成果物、文書体系、基盤は何か。
- 外部連携や公開基盤はどこまで含めるか。
- 対象業務と対象システムの粒度は揃っているか。
- 後続の設計文書へ委譲すべき詳細を書いていないか。

書き方:

- 表形式で「対象」と「内容」を分けると、粒度を揃えやすい。
- 業務機能、文書体系、補助ツール、公開基盤など異なる性質の対象は、同じ粒度の大項目として扱う。
- UI、API、DB、テストケースの詳細は書かない。

### 4.3. 対象期間

問い:

- 初期対象はいつからいつまでか。
- 日付が未確定なら、どのイベントまでを対象にするか。
- 初回公開、初期リリース、運用定着、継続改善をどう分けるか。
- どのスケジュール文書を期間判断の根拠にするか。

書き方:

- 日付が未確定な場合は、イベント基準で定義する。
- 継続改善を含める場合は、初期スコープと継続改善フェーズを分ける。
- 未決事項は _UNDECIDED_: を優先し、決める文書またはタイミングを添える。実施待ちの作業を残す場合のみ _TODO_: を使い分ける。

### 4.4. スコープ外

問い:

- 今回やらないことは何か。
- 外した理由は、責務境界、コスト、効果、公開適性、期間のどれか。
- 利用者に影響がある対象外項目は、後続でいつ再検討するか。
- 対象外を書かないことで誤解される項目はないか。

書き方:

- 「対象（Out）」「理由」「補足」の表にすると、やらない理由と扱いが明確になる。
- スコープ外は弱い表現にせず、今回含めないことを明示する。
- 「将来やるかもしれない」だけで終わらせず、再検討条件を添える。

### 4.5. 境界の判断基準

問い:

- 含めるか外すかで迷ったとき、何を優先するか。
- 業務価値、利用者影響、公開適性、保守容易性のどれを重く見るか。
- 特定の技術、手法、個別要望に引っ張られたとき、どう判断するか。

書き方:

- 2〜5 件程度の優先順位付き箇条書きにする。
- 判断基準は、後続のスコープ変更や優先度判断に再利用できる表現にする。

### 4.6. スコープ変更方針

問い:

- 誰が変更を提案できるか。
- どこに記録するか。
- 何を確認してから判断するか。
- 誰が承認するか。
- 変更により影響を受ける成果物は何か。

書き方:

- 「変更入口」「影響評価」「承認者」「記録先」の表にすると、最低限の運用が揃う。
- 初期リリース前に追加する場合は、同時に外すものや延期するものを示す。
- 判断理由を DEC、課題一覧、Issue、PR、変更要求ログのいずれかに残す。

## 5. 深掘り手順

内容が浅いと感じる場合は、次の順で深掘りする。

1. **対象利用者** を具体化する。
2. **利用場面** を業務や活動の単位で書く。
3. **対象外にすべき候補** を先に列挙する。
4. **対象業務と対象システムの対応** を確認する。
5. **初期対象と継続改善** を分ける。
6. **迷ったときの判断順序** を 2〜5 件に絞る。
7. **変更時の承認者と記録先** を決める。

## 6. 良い例 / 悪い例

| 観点         | 良い例                                                                | 悪い例                       |
| ------------ | --------------------------------------------------------------------- | ---------------------------- |
| 対象業務     | 店主代表以外が店番する場合でも販売・在庫・つけ残高を確認できる状態    | 販売管理を改善する           |
| 対象システム | 販売登録、在庫管理、顧客管理、簡易レポートを同じ粒度で表にする        | 画面名、API、DBを混在させる  |
| 対象期間     | 初期リリースから2週間の記録継続確認までを対象にする                   | できるだけ早く公開する       |
| スコープ外   | EC は店頭販売の記録継続を優先するため対象外とし、次フェーズで検討する | 必要なら EC も考える         |
| 境界判断     | 日々使う業務を優先し、高度な分析は記録安定後に扱う                    | 便利そうなものは入れる       |
| 変更方針     | 提案入口、影響評価、承認者、記録先を表で示す                          | 関係者で相談して柔軟に決める |

## 7. レビュー観点

作成後は、次の観点で確認する。

| 観点             | 確認内容                                                           |
| ---------------- | ------------------------------------------------------------------ |
| 業務価値との対応 | プロジェクト概要の背景、必要性、期待効果とスコープが対応しているか |
| 利用者影響       | 対象利用者、利用場面、影響範囲が読み取れるか                       |
| 対象範囲         | 対象業務、対象システム、対象期間が判断可能な粒度か                 |
| 対象外           | やらないこと、理由、再検討条件が明示されているか                   |
| 境界判断         | 迷ったときに使える優先順位や判断原則があるか                       |
| 変更運用         | 変更入口、影響評価、承認者、記録先があるか                         |
| 責務分担         | 受入条件、設計詳細、実装詳細を書きすぎていないか                   |
| 公開適性         | 個人情報、非公開情報、機密情報が含まれていないか                   |

## 8. 仕上げチェック

- 「全体の作成手順」で示した章立てがそろっている。
- 対象業務、対象システム、対象期間、スコープ外がすべて記述されている。
- 未確定の日付やイベントは _UNDECIDED_: を優先し、決定タイミングが添えられている。
- スコープ外に理由と補足がある。
- 境界の判断基準とスコープ変更方針が、後続の判断に使える。
- 成功基準、受入条件、設計詳細、実装詳細を書きすぎていない。
