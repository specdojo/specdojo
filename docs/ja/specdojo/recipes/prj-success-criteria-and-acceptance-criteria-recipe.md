# 成功基準と受入条件 作成レシピ

Success Criteria and Acceptance Criteria Writing Recipe

本書は、業務価値を起点として、プロジェクトの成功基準と受入条件を判定可能な形にする作り方です。構造・必須項目・禁止事項は [[prj-success-criteria-and-acceptance-criteria-rulebook]] を正とし、本書では問い、深掘り、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、スコープから対象利用者、利用場面、対象外、初期対象期間を抜き出します。
- 次に、利用者にとって実現したい状態を業務価値として短く定義します。
- 業務価値ごとに成功基準を置き、その成功を確認する最小の受入条件と証跡を定義します。
- 最後に、確認者、承認者、否決時の再判定方法を明確にします。

| 種別     | 役割                       | 使いどころ                         |
| -------- | -------------------------- | ---------------------------------- |
| rulebook | 成果物として成立する規約   | 構造、必須項目、禁止事項を確認する |
| recipe   | 良い内容を書くための作り方 | 問い、深掘り、レビューに使う       |
| sample   | 完成例                     | 粒度、文体、表の書き方を確認する   |
| template | 記入用の骨組み             | 各欄を漏れなく配置する             |

## 2. 作成前に集める情報

| 項目         | 集める情報                                             |
| ------------ | ------------------------------------------------------ |
| 対象と対象外 | 今回判定する成果物・状態と、判定しない範囲             |
| 業務価値     | 対象利用者、利用場面、期待する効果または状態           |
| 成功の状態   | いつ、どの状態ならプロジェクトが成功したと判断できるか |
| 受入対象     | 利用者が確認する作業、結果、品質、公開適性             |
| 測定方法     | 観察、件数集計、文書照合、lint などの確認方法          |
| 証跡         | 確認結果を後から追跡できるファイル、記録、ログ         |
| 判断体制     | 確認者、承認者、否決時に是正を担う役割                 |
| 未確定事項   | 閾値、期限、前提のうち、意思決定が未了のもの           |

## 3. 全体の作成手順

1. 判定対象と対象外をスコープから転記せず、受入判断に必要な範囲へ要約する。
2. 利用者と利用場面に結び付く業務価値へ ID を付ける。
3. 各業務価値について、成功した状態を一文で書く。
4. 成功した状態を確認できる判定基準、測定方法、判定時期、確認者を決める。
5. 成功基準を満たすために必要な受入条件を、利用者視点・技術的受入・品質・公開適性に分ける。
6. 各受入条件に合格基準、証跡、確認者、承認者を対応付ける。
7. 判定の実施順、否決時の是正、再判定の記録方法を定義する。
8. 未確定の閾値・期限には `_UNDECIDED_:` を付け、決定時期と担当を記載する。

## 4. 各章の書き方

### 4.1. 判定対象と適用範囲

問い:

- 何の完了、公開、受入を判断するのか。
- 誰がこの判定結果を使うのか。
- 対象外にしないと、実装品質や公開後の効果まで誤って判定対象になる項目は何か。

書き方:

- 対象・対象外をともに一文以上で明示する。
- 業務価値は、利用者、利用場面、期待する状態を含む短文にする。
- 「品質を上げる」のような抽象的な価値ではなく、「利用者が同じ根拠で判断できる」のように状態を書く。

### 4.2. 成功基準

問い:

- どの業務価値が実現すれば、プロジェクトが成功したと言えるか。
- 判定時点で確認できる状態と、公開後にしか測定できない効果を分けられているか。
- 閾値がある場合、対象・単位・母数・比較対象を説明できるか。

書き方:

- 一行は一つの成功状態に限定する。
- 定性条件でも、誰が何を確認すればよいかを測定方法に書く。
- 初期公開時に測れない利用実績は、受入条件に混ぜず未解決事項または後続の測定として扱う。

### 4.3. 受入条件

問い:

- 利用者は何を行い、どの結果なら受け入れられるか。
- 技術的確認は、どの業務価値を支えるために必要か。
- 合格の証拠を第三者が後から確認できるか。

書き方:

- 条件は「対象 + 操作または確認 + 期待結果」で書く。
- 技術的受入は種別を明記し、利用者の受入条件と混在させない。
- 承認者には最終判断を担う人間の役割を置く。AI Agent を承認者にしない。

### 4.4. 判定手順と証跡

問い:

- どの条件を、誰が、どの順で確認するか。
- 否決時に、何を直し、どの条件を再確認するか。
- 判定結果と未決事項をどこに残すか。

書き方:

- 手順は確認者の順に番号付きで書く。
- 証跡は「確認結果」のような抽象名だけで終わらせず、記録内容を示す。
- 再判定時は、対象の受入条件 ID と予定を記録する。

### 4.5. 例外条件と未解決事項

問い:

- 今回の受入範囲から除くが、後で判断が必要なことは何か。
- 閾値または運用前提が未確定でも、受入を進められるか。
- 決定しない場合に誰が影響を受けるか。

書き方:

- 未確定は `_UNDECIDED_:` で明示し、決定する時期と責任者を添える。
- 例外は「受け入れる」だけで終わらせず、再判定する条件を記載する。

## 5. 深掘り手順

1. 対象利用者を、実際に判定する人と成果を利用する人に分ける。
2. 各利用場面で「できるようになること」を一文で書く。
3. その一文を確認できる観察、件数、文書照合、lint のいずれかを選ぶ。
4. 証跡に、確認日、確認者、対象条件 ID を残せるかを確認する。
5. 業務価値に直接結び付かない条件は、必要性を説明できる技術的受入へ分類する。
6. 公開後にしか判定できない指標は、初回受入の合否から分離する。

## 6. 良い例 / 悪い例

| 観点       | 良い例                                                                    | 悪い例                   |
| ---------- | ------------------------------------------------------------------------- | ------------------------ |
| 業務価値   | 店主と家族が同じ手順で在庫を確認できる                                    | システムを便利にする     |
| 成功基準   | 営業日の90%以上で販売・在庫・つけの記録が残っている                       | 日々の記録が定着している |
| 受入条件   | 商品、数量、支払方法を入力して確定すると販売記録が1件作成される           | 販売機能が正しく動く     |
| 証跡       | 確認日、確認者、条件 ID を含む受入確認記録                                | 確認したことを記録する   |
| 未解決事項 | `_UNDECIDED_:` オフライン時の記録方法を初期運用開始前に店主代表が決定する | オフライン時は後で決める |

## 7. レビュー観点

| 観点             | 確認内容                                           |
| ---------------- | -------------------------------------------------- |
| 業務価値との対応 | 全ての成功基準・受入条件に業務価値 ID がある       |
| 利用者視点       | 対象利用者、利用場面、期待結果を読み取れる         |
| 検証可能性       | 測定方法、合格基準、証跡が具体的である             |
| 責任の明確性     | 確認者と人間の承認者が明記されている               |
| 技術的受入       | 技術的条件が種別で分離され、業務価値との関係がある |
| 対象範囲         | スコープ内・対象外と矛盾しない                     |
| 公開適性         | 個人情報、非公開情報、機密情報がない               |

## 8. 仕上げチェック

- [[prj-success-criteria-and-acceptance-criteria-rulebook]] の本文構成に沿っている。
- [[prj-success-criteria-and-acceptance-criteria-sample]] と同程度の粒度である。
- 業務価値、成功基準、受入条件、証跡、確認者、承認者を対応付けた。
- 閾値には測定対象、単位、母数または比較対象を示した。
- 対象外、未決事項、否決時の再判定方法を明示した。
- 実装詳細、個別テスト手順、AI Agent による最終判断を書いていない。
