# プロジェクト憲章 作成レシピ

Project Charter Writing Recipe

本書は、プロジェクト憲章を、PO が立ち上げ認可、権限委譲、予算枠、後続判断条件を判断できる内容に仕上げるための作り方である。構造、必須項目、禁止事項は [[prj-charter-rulebook]] を正とし、本書では問い、深掘り、具体化、レビュー観点を扱う。

## 1. このレシピの使い方

- 最初に、プロジェクト概要から目的、必要性、期待効果、主要前提を抜き出す。
- 次に、ステークホルダー登録簿から、承認者、実行責任者、利用者、影響受容者、AI Agent などの関係者を確認する。
- 認可する範囲と、まだ認可しない範囲を分けて書く。
- 最後に、PM などへ委譲する権限、PO 承認が必要な事項、予算枠、GO / Not GO 判断条件、未決事項をそろえる。

| 種別 | 役割 | 使いどころ |
| --- | --- | --- |
| rulebook | 成果物として成立するための規約 | 構造、必須項目、禁止事項を確認する |
| recipe | 良い内容を書くための手順 | 問い、深掘り、具体化、レビューに使う |
| sample | 完成最小例 | 表の粒度、文体、未決事項の残し方を確認する |
| template | 記入用の骨組み | 必須章と表カラムを用意する |

## 2. 作成前に集める情報

| 項目 | 集める情報 |
| --- | --- |
| 目的と期待効果 | 何のために立ち上げるか、承認で期待する効果は何か |
| 直接根拠 | プロジェクト概要、ステークホルダー登録簿など、認可判断に使う文書 |
| 認可対象 | 立ち上げ、詳細計画、初期準備など、今回認める行為 |
| 認可しない範囲 | 本格実行、外部公開、主要投資、主要変更など、後続判断に回す事項 |
| 関係者 | PO、スポンサー、PM、利用者、影響受容者、支援者 |
| 権限委譲 | 誰に、どの範囲の実行権限または判断権限を渡すか |
| 予算枠 | 承認済み費用、追加支出の可否、外部サービス利用、未確定なら確定時期 |
| GO / Not GO 条件 | 本格実行や公開へ進む前に満たす条件と記録先 |
| 未決事項 | 承認前または後続判断前に決める論点、担当、期限 |

## 3. 全体の作成手順

1. H1 と冒頭文で、対象プロジェクト名と憲章の目的を明示する。
2. 認可対象を表にし、今回認可する範囲と認可しない範囲を分ける。
3. プロジェクト概要を根拠に、目的と期待効果を短く要約する。
4. ハイレベルスコープ、成果物、成功基準を概要レベルで書き、詳細化先を示す。
5. ステークホルダー登録簿から、認可判断に関係する初期ステークホルダーを要約する。
6. 権限委譲表を作り、委譲する事項と PO 承認が必要な事項を分ける。
7. 主要前提・制約に、公開適性、人間の判断責任、予算枠を含める。
8. 後続で作成・詳細化する文書と GO / Not GO 判断条件を整理する。
9. 承認履歴と未決事項を記入し、未確定事項を確定済みのように書いていないか確認する。

## 4. 各章の書き方

### 4.1. 本書の目的と認可対象

問い:

- この憲章は、何を立ち上げることを認めるのか。
- どの活動は、まだ認可していないのか。
- 誰が最終判断と説明責任を持つのか。

書き方:

- 「立ち上げ認可」「詳細計画策定」「初期準備」を基本の認可対象にする。
- 本格実行、外部公開、主要投資、主要変更は、後続判断で扱う場合が多い。
- 直接根拠は、本文の判断に実際に使った文書だけを列挙する。

### 4.2. プロジェクトの目的

問い:

- 放置すると何が困るのか。
- 立ち上げることで、誰のどの状態を改善するのか。
- PO はどの期待効果を見て承認、保留、差し戻しを判断するのか。

書き方:

- プロジェクト概要の詳細本文を丸写しせず、認可判断に必要な要約にする。
- 期待効果は、仕様共有、業務価値、透明性、継続性など、判断に使える観点へ分ける。
- 中長期効果は、初期承認時の確定成果ではなく、詳細化または測定対象として書く。

### 4.3. ハイレベルスコープ、成果物、成功基準

問い:

- 立ち上げ時点で対象に含める活動領域は何か。
- 明示的に対象外にするものは何か。
- 詳細をどの後続文書へ委譲するか。

書き方:

- 憲章では「候補」「主要成果物群」「成功判定の観点」に留める。
- 詳細スコープ、受入条件、成果物配置、設計詳細は後続文書へ委譲する。
- 未確定の範囲を隠さず、未決事項または詳細化先へつなげる。

### 4.4. 初期ステークホルダー

問い:

- 誰が承認し、誰が実行し、誰が利用または影響を受けるか。
- AI Agent は支援者としてどの範囲に関わるか。
- 公開文書に記載してよい識別方法になっているか。

書き方:

- ステークホルダー登録簿から、認可判断に関係する主要関係者だけを要約する。
- 個人名や連絡先ではなく、役割名、集団名、Role code を用いる。
- AI Agent を承認者または責任者にせず、人間の PO が最終判断することを明示する。

### 4.5. 権限委譲と主要前提・制約

問い:

- PM や実行責任者へ、何を任せるのか。
- PO が引き続き承認する事項は何か。
- 予算枠、公開可否、ライセンス、主要変更はどの扱いか。

書き方:

- 決裁者、実行責任者、協議先、証跡を表でそろえる。
- 予算枠が未確定なら、追加支出の可否と確定タイミングを書く。
- 公開適性、人間の判断責任、秘密情報を扱わない制約を明示する。

### 4.6. GO / Not GO 判断、承認、未決事項

問い:

- 本格実行や外部公開へ進む前に何を確認するか。
- 判断結果はどこに記録するか。
- 承認済みと言えるだけの承認者、日付、証跡があるか。

書き方:

- GO / Not GO 判断には、目的整合、公開適性、予算枠、体制、品質を含める。
- 承認履歴は、未承認なら未承認と分かる状態で残す。
- 未決事項は「論点、期限、担当、対応方針」をそろえ、放置しない。

## 5. 深掘り手順

内容が浅い場合は、次の順で具体化する。

1. プロジェクト概要の目的文を 1 文に要約し、憲章冒頭に置く。
2. 「今回認めること」と「まだ認めないこと」を左右に分けて列挙する。
3. 主要ステークホルダーについて、承認者、実行責任者、協議先、影響受容者へ分類する。
4. PM に委譲してよい判断と、PO が保持すべき判断を仕分ける。
5. 予算枠を、金額、追加支出なし、外部支出未承認、運用負担上限などの形で明文化する。
6. 本格実行や公開前に未決事項を解消する条件を GO / Not GO 判断に移す。
7. 詳細スコープ、設計、受入条件を書きすぎている箇所を後続文書へ戻す。

## 6. 良い例 / 悪い例

| 観点 | 良い例 | 悪い例 |
| --- | --- | --- |
| 認可対象 | 「立ち上げ、詳細計画策定、初期準備を認可する」と書く | 「プロジェクトを進める」とだけ書く |
| 認可しない範囲 | 「外部公開と主要支出は別途 PO が判断する」と書く | 承認範囲を広く曖昧にする |
| 権限委譲 | 決裁者、実行責任者、証跡を表で示す | 「PM に任せる」とだけ書く |
| 予算枠 | 「追加支出は未承認。上限は詳細計画で PO が確定」と書く | 費用に触れない |
| AI Agent | 「草案作成と整合確認を支援し、承認しない」と書く | AI Agent を判断者として扱う |
| 承認 | 未承認なら `_UNDECIDED_` とし、正式認可済みと書かない | 承認者未記載で承認済みとする |

## 7. レビュー観点

| 観点 | 確認内容 |
| --- | --- |
| 目的整合 | プロジェクト概要の目的、必要性、期待効果と矛盾していないか |
| 承認判断 | PO が承認、保留、差し戻しを判断できる認可対象と未決事項があるか |
| 権限範囲 | 委譲する権限と PO 承認事項が分離されているか |
| 予算枠 | 費用、追加支出、外部サービス利用、未確定時の確定方法が分かるか |
| 関係者 | ステークホルダー登録簿と整合し、個人情報や非公開情報がないか |
| 公開適性 | 公開してよい情報だけで構成され、人間の説明責任を前提にしているか |
| 責務分担 | 詳細スコープ、設計、受入条件、会議運営を書きすぎていないか |
| 証跡 | 承認、GO / Not GO、主要変更の記録先があるか |

## 8. 仕上げチェック

- [[prj-charter-rulebook]] の本文構成に沿っている。
- [[prj-charter-sample]] と同程度の粒度で、必須表が埋まっている。
- 直接根拠、認可対象、認可しない範囲、承認責任が書かれている。
- 予算枠または未確定時の扱いが書かれている。
- AI Agent を承認者にせず、人間の PO が最終判断する前提になっている。
- 正式認可済みと書く場合は、承認者、承認日、証跡リンクがある。
- 公開不適切な情報、設計詳細、受入条件の過剰記載がない。
