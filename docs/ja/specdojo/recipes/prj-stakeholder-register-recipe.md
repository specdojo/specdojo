# ステークホルダー登録簿 作成レシピ

Stakeholder Register Writing Recipe

本書は、ステークホルダー登録簿を、業務価値・利用者影響・合意対象が判断できる内容に仕上げるための作り方である。構造、必須項目、禁止事項は [[prj-stakeholder-register-rulebook]] を正とし、本書では問い、深掘り、具体化、レビュー観点を扱う。

## 1. このレシピの使い方

- 最初に、プロジェクト概要から目的、利用者、期待効果、前提条件を抜き出す。
- 次に、誰が判断し、誰が利用し、誰が協力し、誰が影響を受けるかを役割別に洗い出す。
- 各関係者について、期待、懸念、必要な合意、情報要求を同じ ID でつなげる。
- 最後に、影響度/関心度の根拠と、変更時に見直す条件を確認する。

| 種別     | 役割                           | 使いどころ                            |
| -------- | ------------------------------ | ------------------------------------- |
| rulebook | 成果物として成立するための規約 | 構造、必須項目、禁止事項を確認する    |
| recipe   | 良い内容を書くための手順       | 問い、深掘り、具体化、レビューに使う  |
| sample   | 完成最小例                     | 表の粒度、文体、ID の使い方を確認する |
| template | 記入用の骨組み                 | 必須章と表カラムを用意する            |

## 2. 作成前に集める情報

| 項目             | 集める情報                                                                                        |
| ---------------- | ------------------------------------------------------------------------------------------------- |
| 目的と期待効果   | 誰のどの課題を改善し、どの状態を目指すか                                                          |
| 利用者と利用場面 | 直接利用者、間接利用者、影響を受ける集団、利用する場面                                            |
| 意思決定         | 方針、優先順位、公開・利用可否を判断する人または組織                                              |
| 実行・協力       | 実行担当、運用担当、外部協力者と、それぞれの関わり方                                              |
| 想定される懸念   | 利用負荷、変更影響、継続性、公開適性、合意遅延の要因                                              |
| 合意対象         | 着手、範囲、変更、公開、利用開始で確認が必要な事項                                                |
| 情報要求         | 誰が、何を、どの手段で確認・判断するか                                                            |
| 更新条件         | 目的、範囲、体制、利用者影響、公開方針が変わる出来事                                              |
| 実行主体の境界   | AI Agent やツールなど人間以外の実行主体が関与する場合、入力・制約・確認の責任を持つ人の役割は誰か |

## 3. 全体の作成手順

1. プロジェクトの目的と利用者を確認し、登録対象の識別基準を決める。
2. 意思決定者、実行担当、利用者、外部協力者、影響受容者を候補として列挙する。
3. 関係者一覧に、役割名・集団名を用いて一意の `STH-<KEY>` を付ける。
4. 各関係者の期待、懸念、必要な合意を整理し、影響度と関心度に根拠を添える。
5. 現状と目標を比べ、情報提供、相談、合意、巻き込みの対応方針を決める。
6. 情報要求、希望チャネル、証跡要件を整理し、詳細な配信計画は後続のコミュニケーション計画へ委譲する。
7. 変更時と定期確認時の見直し条件、責任者、承認者を定める。
8. 個人情報、推測の Role code、設計詳細、会議運営の重複がないことを確認する。

## 4. 各章の書き方

### 4.1. 関係者一覧

問い:

- 誰が目的、範囲、公開・利用可否を判断するか。
- 誰が直接利用し、誰が間接的に影響を受けるか。
- 誰が実行、運用、協力を担い、どの責任を持つか。
- AI Agent やツールなど人間以外の実行主体が関与する場合、入力・制約・生成結果の確認責任を持つ人の役割は誰か。

書き方:

- 個人名ではなく、役割名、組織名、集団名で書く。
- ステークホルダーとタスク Role を分け、採用済みの Role だけを対応 Role code として書く。
- AI Agent やツールなどの実行主体自体は単独のステークホルダーとして記載せず、入力・制約・確認の責任を持つ人の役割として記載する。
- 同じ関係者を全章で同じ ID にする。

### 4.2. 影響度/関心度分析

問い:

- 方針、範囲、優先順位、利用可否を左右するのは誰か。
- 成果または変更を継続して確認する必要があるのは誰か。
- 期待が満たされないと、誰にどのような利用・継続上の影響が出るか。

書き方:

- `High` は、影響を与える根拠または継続確認が必要な根拠を必ず添える。
- 期待、懸念、必要な合意を対応させ、「誰と何を決めるか」を明確にする。
- 評価は人の重要度ではなく、プロジェクトへの影響と関与必要性で行う。

### 4.3. エンゲージメント方針

問い:

- 現在の関与状態と、目標とする関与状態は何か。
- 差分を埋めるために、情報提供、相談、合意、巻き込みのどれが必要か。
- 誰がいつまでに行い、どこに証跡を残すか。

書き方:

- 「確認する」だけで終わらせず、対象情報、実施者、期限、証跡を記載する。
- 外部関係者が未関与なら、現在の事実と将来の仮定を混ぜず、必要に応じて `_ASSUMPTION_:` を使う。
- 合意を求める相手と、情報共有だけでよい相手を区別する。

### 4.4. コミュニケーション要件

問い:

- 関係者は何を受け取れば、判断、利用、協力、影響確認ができるか。
- どの手段なら内容を確認でき、必要な証跡を残せるか。
- 詳細計画で追加管理すべき事項は何か。

書き方:

- 情報要求は「進捗」だけでなく、優先順位、変更理由、利用手順など判断に必要な情報に分解する。
- 会議の日時や配信頻度の詳細を重複管理しない。
- 合意・報告の必要性と、証跡要件を同じ行に置く。

### 4.5. 見直し条件

問い:

- どの変更が、関係者、期待、影響度、合意対象を変えるか。
- 誰が見直し、誰が承認し、どの証跡で追跡するか。
- 定期確認では、どの未解決の合意や情報要求を確認するか。

書き方:

- 目的、範囲、体制、公開・利用方針の変更を更新トリガーに含める。
- 周期が未定なら `_TODO_:` として決める文書またはタイミングを残す。
- 変更した表だけでなく、同じ ID を使う全章への影響を確認する。

## 5. 深掘り手順

内容が浅い場合は、次の順で具体化する。

1. 目的と期待効果から、利益を得る利用者と影響を受ける集団を洗い出す。
2. 方針、範囲、公開・利用可否の決定を列挙し、意思決定者を対応付ける。
3. 各関係者について「期待が満たされない場合の困りごと」を書き、懸念を具体化する。
4. 各懸念に対して、必要な合意または情報要求を対応付ける。
5. High の関係者を先に選び、関与方針、責任者、期限、証跡を埋める。
6. 変更イベントごとに、見直す行と承認者を確認する。

## 6. 良い例 / 悪い例

| 観点           | 良い例                                                           | 悪い例                                   |
| -------------- | ---------------------------------------------------------------- | ---------------------------------------- |
| 関係者の識別   | 「将来の利用者」を利用場面と期待を添えて記載する                 | 「ユーザー」とだけ書く                   |
| 影響度の根拠   | 「公開・利用可否を判断するため High」と書く                      | 「重要なので High」と書く                |
| 必要な合意     | 「初期公開の範囲と公開情報」と具体化する                         | 「いろいろ相談する」と書く               |
| 関与方針       | 「初期公開前に導入手順を公開し、Issue で不明点を収集する」と書く | 「利用者と連携する」と書く               |
| 情報要求       | 「変更理由と利用手順を公開文書で確認する」と書く                 | 「進捗を共有する」とだけ書く             |
| 見直し条件     | 「公開方針変更時に将来利用者の要件を見直す」と書く               | 「必要に応じて更新する」と書く           |
| 実行主体の境界 | 「AI Agent活用担当」のように責任を持つ人の役割として記載する     | 「AI Agent」を単独の関係者として記載する |

## 7. レビュー観点

| 観点                   | 確認内容                                                                    |
| ---------------------- | --------------------------------------------------------------------------- |
| 業務価値との対応       | プロジェクトの目的、利用者、期待効果から関係者を識別できているか            |
| 利用者影響             | 直接利用者、間接利用者、影響受容者の期待と懸念が分かるか                    |
| 合意対象               | High の関係者について、誰と何を合意するかがあるか                           |
| 評価根拠               | 影響度と関心度に、方針・利用・運用・合意への影響の根拠があるか              |
| 関与方針               | 現状、目標、行動、責任者、期限、証跡が対応しているか                        |
| コミュニケーション統制 | 詳細計画と重複せず、情報要求と証跡要件を入力として渡せるか                  |
| 公開適性               | 個人情報、非公開情報、未採用 Role code、根拠のない推測がないか              |
| 実行主体の境界         | AI Agent やツールなどの実行主体が、責任を持つ人の役割に対応付けられているか |

## 8. 仕上げチェック

- [[prj-stakeholder-register-rulebook]] の本文構成に沿っている。
- 各関係者が、関係者一覧から見直し条件まで同じ ID で追跡できる。
- High の関係者に、期待、懸念、必要な合意、対応方針がある。
- 利用者・影響受容者の利用場面または影響が読み取れる。
- AI Agent やツールなどの実行主体を、責任を持つ人の役割に対応付けている（単独のステークホルダーとして扱っていない）。
- 詳細な会議運営、設計詳細、個人情報を本文に含めていない。
- [[prj-stakeholder-register-sample]] と同程度に、各必須表のカラムを埋めている。
