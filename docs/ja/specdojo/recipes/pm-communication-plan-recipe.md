# コミュニケーション計画 作成レシピ

Communication Plan Writing Recipe

本ドキュメントは、コミュニケーション計画を「PM が進捗確認、課題・リスク管理、変更判断、公開前確認へ接続できる内容」に仕上げるための作り方である。本書では問い、深掘り手順、レビュー観点を扱う。

## 1. このレシピの使い方

- 最初に、上位の管理計画から報告すべき管理領域、判断者、見直し条件を確認する。
- 次に、Role code ごとの情報要求、会議体、報告種別、連絡チャネル、エスカレーション条件へ分解する。
- 最後に、進捗、課題、リスク、変更要求、決定記録へ転記すべき事項が本文に混在していないか確認する。

## 2. 作成前に集める情報

| 項目                 | 集める情報                                                                                                                                 |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| 上位管理方針         | 進捗、課題、リスク、変更要求、品質、公開判断の報告指標                                                                                     |
| Role code            | 報告先、相談先、責任ロール、最終判断者に使える責務語彙（プロジェクトの規模・複雑性に応じて、最小構成以外に必要なロールを漏れなく識別する） |
| 情報要求             | 各 Role code が受け取るべき情報、発信する情報、頻度                                                                                        |
| 会議体               | 作業節目、成果物レビュー、課題・リスク確認、変更要求確認、公開前確認の要否                                                                 |
| 報告種別             | 報告目的、受信ロール、タイミング、形式、責任ロール                                                                                         |
| 連絡チャネル         | PJR、Pull Request、Issue、実行結果、README などの使い分け                                                                                  |
| エスカレーション条件 | スコープ、Schedule、品質、構成、利用者視点、公開・運用、AI Agent 出力の判断条件                                                            |
| 公開方針             | 公開してはいけない情報、AI Agent 入力に含めない情報                                                                                        |

## 3. 全体の作成手順

1. H1 直下に、PM が何を管理・報告する計画かを一文で記述する。
2. 概要で、上位管理方針、PM と PO の責務境界、AI Agent の支援範囲、公開情報の制約を明確にする。
3. ロール別コミュニケーション分類に、Role code ごとの関心事項、受信情報、発信情報、頻度を置く。
4. 会議体計画に、会議名、目的、参加ロール、頻度、成果物・証跡を置く。
5. 報告計画に、報告種別と報告に含める内容を置く。
6. 連絡チャネルとエスカレーションに、チャネルの使い分けと判断を上げる条件を置く。
7. 情報管理と公開方針に、証跡管理と公開禁止情報を置く。
8. 見直し条件に、計画変更、Role code 変更、Schedule 変更、外部参加、公開開始などのトリガーを置く。
9. 関連ドキュメントと継続確認事項を整理し、未決事項には判断者と次アクションを添える。

## 4. 各章の書き方

### 4.1. 概要

問い:

- この計画は、PM がどの情報を報告・連絡・エスカレーションするための文書か。
- PM が整理する事項と、PO が最終判断する事項は分かれているか。
- 小規模運用や AI Agent の支援範囲を、責任境界と混同していないか。

書き方:

- 「PM は進捗、課題、リスク、変更要求、決定事項を管理・報告に接続できる単位へ分離する」のように、管理接続を明示する。
- Role code と実行主体を分け、具体的な member や agent は別文書へ委譲する。
- 公開リポジトリで扱えない情報を最初に宣言する。

### 4.2. ロール別コミュニケーション分類

問い:

- 各 Role code は、何を知る必要があり、何を発信する必要があるか。
- 受信情報と発信情報が、報告種別やエスカレーション条件とつながっているか。
- 未採用 Role code、member nickname、人名、agent 名が混入していないか。

書き方:

- 位置づけ、関心事項、受信情報、発信情報、頻度を表にする。
- すべてのロールを網羅するより、報告・確認・判断・公開に必要なロールを漏らさないことを優先する。
- 頻度は「作業節目」「変更要求起票時」「公開前」のように、発生条件で書く。

### 4.3. 会議体計画

問い:

- 会議を開く目的は、判断、確認、レビュー、共有のどれか。
- 会議を開催しない場合、判断内容と結果はどこに残るか。
- 会議体が多すぎて小規模運用の負荷になっていないか。

書き方:

- 会議名、目的、参加ロール、頻度、成果物・証跡を表にする。
- 作業計画、成果物レビュー、課題・リスク、変更要求、公開前確認のように、扱う判断単位を分ける。
- 「必要に応じて開催」だけで終わらせず、開催条件と証跡を添える。

### 4.4. 報告計画

問い:

- 進捗、課題、リスク、変更要求、品質、公開準備を誰へいつ報告するか。
- 報告内容は、後続の管理台帳や決定記録へ転記できる粒度か。
- 期限超過や依存未解決がある場合の再計画案が含まれるか。

書き方:

- 報告種別、受信ロール、目的、期限 / タイミング、形式、責任ロールを表にする。
- 報告に含める内容は、完了、未完了、ブロック、影響、判断者、次アクションに分ける。
- 課題、リスク、変更要求、決定事項を本文に混在させず、記録先へ接続する。

### 4.5. 連絡チャネルとエスカレーション

問い:

- PJR、Pull Request、Issue、実行結果、README などを何に使い分けるか。
- どの条件で PM、PO、QE、OPS などへ判断を上げるか。
- 最終判断者と証跡が、区分ごとに読めるか。

書き方:

- チャネル表には用途、使用条件、証跡を置く。
- エスカレーション表には、区分、一次対応、条件、最終判断、証跡を置く。
- 判断者が異なるスコープ、Schedule、品質、公開・運用を 1 行に混ぜない。

### 4.6. 情報管理と公開方針

問い:

- 重要判断、課題、リスク、変更要求、決定事項はどこに残るか。
- 公開できない情報や AI Agent に渡してはいけない情報は明確か。
- 生成ビューと正本が矛盾した場合の扱いがあるか。

書き方:

- 証跡管理は箇条書きで、記録先と正本優先の方針を明確にする。
- 公開情報は、個人情報、連絡先、非公開情報、認証情報、秘密情報を具体的に禁止する。
- 公開前確認の対象を、本文、生成物、差分、添付ファイルまで広げる。

## 5. 深掘り手順

1. コミュニケーション対象を「進捗」「課題」「リスク」「変更要求」「品質」「公開判断」に分ける。
2. 各対象について、受信ロール、発信ロール、報告タイミング、証跡を決める。
3. 期限超過、依存未解決、レビュー待ち、検証失敗が起きたときの報告先を決める。
4. 判断者が異なるものを分割し、PO 判断、PM 判断、QE 判断、OPS 判断を混同しない。
5. 会議を置く前に、非同期の証跡で足りるか確認する。
6. 公開不適切情報や認証情報が本文に入っていないか確認する。

## 6. 良い例 / 悪い例

| 観点             | 良い例                                                          | 悪い例                         |
| ---------------- | --------------------------------------------------------------- | ------------------------------ |
| 報告責任         | 進捗報告は PM が作業節目に PO と関連 Role code へ共有する       | 進捗は関係者へ随時共有する     |
| エスカレーション | Schedule 影響は重要タスクの遅延または依存未解決時に PM が上げる | 遅れそうなら相談する           |
| 証跡             | 変更要求は変更要求ログと決定記録へ接続する                      | チャットで合意したら完了とする |
| Role code        | 報告先を `PO`、責任ロールを `PM` と書く                         | 特定の人名や agent 名を書く    |
| 会議体           | 公開前確認の目的、参加ロール、証跡を表で示す                    | 公開前に必要なら集まる         |
| 公開情報         | 個人連絡先、認証情報、秘密情報を記載しない                      | 後で消す前提で本文に入れる     |

## 7. レビュー観点

| 観点             | 確認内容                                                                     |
| ---------------- | ---------------------------------------------------------------------------- |
| 計画化可能性     | 報告、会議体、チャネル、エスカレーションがタスク化・進捗確認に使える粒度か   |
| 報告経路         | 進捗、課題、リスク、変更要求、品質、公開準備の受信ロールと責任ロールが明確か |
| 情報要求         | Role code ごとの受信情報、発信情報、頻度が読み取れるか                       |
| 管理台帳への接続 | 課題、リスク、変更要求、決定記録へ転記すべき事項が分離されているか           |
| 判断責任         | PM の整理責任と PO の最終判断が混同されていないか                            |
| 公開適性         | 個人情報、非公開情報、認証情報、秘密情報を含んでいないか                     |

## 8. 仕上げチェック

- 「全体の作成手順」で示した章立てと必須表がそろっている。
- 各章の表と箇条書きが具体的な内容で完成している。
- 報告先、責任ロール、最終判断者が Role code で記述されている。
- 進捗、課題、リスク、変更要求、品質、公開準備の報告経路がある。
- 課題、リスク、変更要求、決定記録へ転記すべき事項が分離されている。
- member nickname、人名、agent 名を責任主体にしていない。
- 公開不適切情報、認証情報、秘密情報を含んでいない。
