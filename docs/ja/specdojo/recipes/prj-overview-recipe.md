# プロジェクト概要 作成レシピ

Project Overview Writing Recipe

本ドキュメントは、プロジェクト概要を「読み手が短時間で立ち上げ判断の前提を理解できる内容」に仕上げるための作り方です。
本書では問い、深掘り手順、具体化のコツ、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、プロジェクトの仮説を粗く書き出す。
- 次に、各章の問いに答えながら、背景、必要性、基本的な考え方、実現したいこと、期待効果、前提条件を具体化する。
- 最後に、レビュー観点で「曖昧すぎないか」「後続文書に委譲すべき詳細を書きすぎていないか」を確認する。

## 2. 作成前に集める情報

最初から正確に書こうとせず、以下を短いメモで集める。

| 項目           | 集める情報                                         |
| -------------- | -------------------------------------------------- |
| プロジェクト名 | 読み手が対象を識別できる名称                       |
| 対象領域       | どの活動、業務、サービス、システムを扱うか         |
| 現状           | いま何が起きているか                               |
| 困りごと       | 誰が、何に困っているか                             |
| 放置リスク     | 取り組まない場合に悪化すること                     |
| 実現したい状態 | 何ができるようになれば前進と言えるか               |
| 判断軸         | 進め方で重視する原則、譲れない考え方               |
| 初期効果       | 初期公開や初期リリース時点で確認できる効果         |
| 中長期効果     | 運用開始後、公開後、利用拡大後に確認したい効果     |
| 前提・制約     | 変えないこと、依存すること、まだ決まっていないこと |

## 3. 全体の作成手順

1. H1 と導入文で「何のプロジェクトか」を一文で定義する。
2. 背景で、現状と困りごとを分けて書く。
3. 必要性で、なぜ今取り組む必要があるかを書く。
4. 基本的な考え方で、判断軸を 2〜5 件に絞る。
5. 実現したいことで、成果の方向性を箇条書きにする。
6. 期待効果で、短期と中長期の効果を分け、測定方法を置く。
7. 前提条件で、適用範囲、責任分担、制約、未決事項を明示する。
8. 最後に、詳細化すべき内容を後続文書へ委譲できているか確認する。

## 4. 各章の書き方

### 4.1. 導入文

導入文は、H1 の直下に 1 文で置く。

問い:

- このプロジェクトは何を支援するものか。
- 誰が使う、または誰に影響するものか。
- 何を通じて、どの状態を実現するのか。

書き方:

- 「A は、B を通じて、C を支援する D である」の形にすると書きやすい。
- 効果、指標、実装方式を盛り込みすぎない。
- 読み手が冒頭 10 秒で対象をつかめる粒度にする。

良い例:

> 駄菓子屋きぬや 販売管理システムは、店頭販売、在庫、常連顧客のつけを一元管理し、店主一人でも日々の運用を続けやすくするための小規模店舗向け業務支援システムである。

悪い例:

> 店舗運営を適切に改善するための便利なシステムである。

### 4.2. 背景

背景では、現状と困りごとを混ぜずに書く。

問い:

- いま、どのような状況か。
- 誰が、どの作業で困っているか。
- その困りごとは、どの程度繰り返し起きているか。
- すでに見えている影響は何か。

具体化のコツ:

- まず 1〜2 段落で文脈を説明する。
- その後に、顕在化している問題を箇条書きで並べる。
- 事実と推測が混ざる場合は、推測側を「仮説」として扱う。

良い例:

- 人気商品の欠品が増えている。
- 仕入れのタイミングや数量が勘頼りで、在庫のムラが大きい。
- 店主以外が店番をした際に、販売・在庫・つけの状況をすぐに確認できない。

悪い例:

- 現状いろいろ大変で、業務改善が必要である。

### 4.3. 必要性

必要性では、背景で示した課題に対して「なぜ取り組む必要があるか」を説明する。

問い:

- 放置すると何が悪化するか。
- 今取り組まないと失われる機会は何か。
- 誰の負担、リスク、判断コストが増えるか。
- 初期対応として、何を先に安定させる必要があるか。

書き方:

- 背景の繰り返しではなく、放置リスクや着手理由を書く。
- 期限や繁忙期など、時間的な理由があれば書く。
- 「必要だから必要」ではなく、取り組まない場合の影響を示す。

### 4.4. 基本的な考え方

基本的な考え方では、プロジェクトの判断軸を短く定義する。

問い:

- 迷ったときに何を優先するか。
- 何をやりすぎないようにするか。
- どの価値を守るために、その進め方を選ぶのか。
- 後続のスコープ、設計、受入条件に効く原則は何か。

書き方:

- 原則は 2〜5 件に絞る。
- 各原則は「名称 + 一文説明」にする。
- 原則名は日本語だけでもよい。英語名を付ける場合は、読み手に意味が伝わる訳を添える。

例:

1. **まず記録する**: 高度な分析よりも、日々の記録が続くことを優先する。
2. **迷わず操作できる**: 画面数と入力項目を最小限にする。
3. **小さく始めて改善する**: 初期リリースでは単一店舗の店頭運用に絞る。

### 4.5. 実現したいこと

実現したいことでは、プロジェクトが提供したい成果の方向性を書く。

問い:

- 何ができるようになれば、プロジェクトを進める意味があるか。
- どの利用者、現場、活動が楽になるか。
- どこまでを初期対象として考えるか。
- 後続文書で詳細化すべき内容は何か。

書き方:

- 具体的な機能名だけに寄せず、実現したい状態で書く。
- 詳細な対象範囲や対象外は `prj-scope` に委譲する。
- 4〜6 件程度の箇条書きにすると読みやすい。

良い例:

- 駄菓子の販売、在庫、常連顧客のつけをシンプルな仕組みで一元管理する。
- 家族が店番をする場合でも、在庫状況とつけ残高を同じ手順で確認できるようにする。

悪い例:

- 商品マスタ画面、販売登録画面、在庫一覧画面、顧客検索画面、月次集計画面を作る。

### 4.6. 期待効果

期待効果では、短期的に確認する効果と、中長期で追跡する効果を分ける。

問い:

- 初期公開または初期リリース時点で判定できることは何か。
- 初期時点では判定せず、実績値として記録することは何か。
- 運用開始後に追跡する効果は何か。
- 数値目標は初期仮説か、確定したコミット値か。
- 測定方法は再現できるか。

短期指標の考え方:

- QCD（品質・コスト・納期）の 3 観点で、初期リリース時点の効果を確認する。
- Quality（品質）は、初期公開時点の達成度として判定しやすい。
- Cost（コスト）/ Delivery（納期）は、初回ではベースラインがない場合が多いため、まず実績値を記録する。
- 削減率や短縮率は、同等規模の実績がたまってから判定する。

中長期指標の考え方:

- 運用定着、継続性、参加機会、透明性など、時間が必要な効果を置く。
- 公開直後や運用直後は母数が小さいため、累積件数や実績記録から始める。

推奨表（観点・指標・目標・測定方法を 1 行ずつ書く。下記は短期指標の記入例）:

| 観点             | 指標                                 | 目標        | 測定方法                             |
| ---------------- | ------------------------------------ | ----------- | ------------------------------------ |
| Quality（品質）  | 初期リリースに必要な機能の達成度     | 90%以上     | 受入条件の項目に対する実現割合を確認 |
| Cost（コスト）   | 日々の記録・確認にかかる操作時間     | 1日10分以内 | 初期運用期間に所要時間を日次で計測   |
| Delivery（納期） | 最小運用が毎営業日定着するまでの期間 | 2週間以内   | 記録の有無を日次で確認               |

レビューのコツ:

- 「何を数えるか」が分からない指標は修正する。
- 「50%削減」などの相対目標には、比較対象を置く。
- 測定できない場合は「未確定」ではなく、何を記録して後で判定するかを書く。

### 4.7. 前提条件

前提条件では、プロジェクト概要を読む人が誤解しやすい境界を明示する。

問い:

- 初期対象に含めるものは何か。
- 初期対象に含めないものは何か。
- 誰が最終判断を行うか。
- AI Agent やシステムは、どこまで支援する位置づけか。
- 未決事項は何で、どの文書またはタイミングで決めるか。

書き方:

- 「対象」「対象外」「判断責任」「未決事項」を分けて考える。
- 個人情報や非公開情報は書かない。
- 未決事項は `_UNDECIDED_:` を使い、決定タイミングを添える。

## 5. 深掘り手順

内容が浅いと感じる場合は、次の順で深掘りする。

1. **誰が困っているか** を具体化する。
2. **何が繰り返し起きているか** を事実として書く。
3. **放置した場合の悪化** を書く。
4. **初期に成立させる最小状態** を決める。
5. **迷ったときの判断軸** を 2〜5 件に絞る。
6. **短期で判定する指標** と **中長期で追跡する指標** を分ける。
7. **後続文書へ委譲する詳細** を切り出す。

## 6. 良い例 / 悪い例

| 観点           | 良い例                                                                     | 悪い例                       |
| -------------- | -------------------------------------------------------------------------- | ---------------------------- |
| 導入文         | 店頭販売、在庫、つけを一元管理する小規模店舗向け業務支援システム           | 店舗運営を便利にするシステム |
| 背景           | 欠品、在庫ムラ、つけ管理漏れが起きている                                   | 現場が困っている             |
| 必要性         | 紙運用を続けると店舗継続リスクが高まる                                     | 早く作った方がよい           |
| 基本方針       | 高度な分析より日々の記録継続を優先する                                     | 使いやすく高機能にする       |
| 実現したいこと | 家族が店番する場合でも在庫とつけを同じ手順で確認できる                     | 在庫画面を作る               |
| 期待効果       | 初期リリースでは記録機能の達成度を判定し、日々の操作時間は実績値を記録する | 操作時間を大幅に短縮する     |
| 前提条件       | 初期リリースでは EC 連携、複数店舗管理、本格会計連携は対象外とする         | 必要に応じていろいろ対応する |

## 7. レビュー観点

作成後は、次の観点で確認する。

| 観点           | 確認内容                                               |
| -------------- | ------------------------------------------------------ |
| 一読性         | H1 と導入文だけで対象プロジェクトの概要が分かるか      |
| 背景の具体性   | 現状、困りごと、影響が具体的に書かれているか           |
| 必要性の明確さ | 放置リスクや今取り組む理由が背景と接続しているか       |
| 判断軸         | 基本的な考え方が後続のスコープや設計判断に使える粒度か |
| 成果の粒度     | 実現したいことが機能列挙だけになっていないか           |
| 指標の測定性   | 指標、目標、測定方法、判定時期が分かるか               |
| 責務分担       | 認可、受入条件、詳細スコープを本書に書きすぎていないか |
| 前提条件       | 対象外、判断責任、未決事項が明示されているか           |
| 公開適性       | 個人情報、非公開情報、機密情報が含まれていないか       |

## 8. 仕上げチェック

- 「全体の作成手順」で示した章立てがそろっている。
- 章ごとに「なぜその情報が必要か」が読み取れる。
- 数値目標には測定方法と判定タイミングがある。
- 初期仮説、未決事項、後続文書へ委譲する内容が混ざっていない。
- 詳細なスコープ、受入条件、認可情報、設計詳細を書きすぎていない。
