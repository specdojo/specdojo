# プロジェクト課題と解決アプローチ 作成レシピ

Project Issues and Approach Writing Recipe

本ドキュメントは、プロジェクト課題と解決アプローチを「関係者が、何が課題で、どれを、どう解く方針かを合意できる内容」に仕上げるための作り方です。
構造や必須項目は [[prj-issues-and-approach-rulebook]] を正とし、本書では問い、深掘り手順、具体化のコツ、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、プロジェクト概要・スコープ・前提制約から「何を実現したいか」と「成立条件」を短く抜き出す。
- 次に、課題一覧、原因、解決策候補、採用アプローチと理由を順に書く。
- 最後に、トレードオフ/リスクと次の検討事項を置き、未確定のまま判断を進めないようにする。

`rulebook` / `recipe` / `sample` の役割は次のように分ける。

| 種別     | 役割                           | 使いどころ                         |
| -------- | ------------------------------ | ---------------------------------- |
| rulebook | 成果物として成立するための規約 | 構造、必須項目、禁止事項を確認する |
| recipe   | 良い内容を書くための作り方     | 問い、観点、深掘り、レビューに使う |
| sample   | 完成例                         | 粒度、文体、表の書き方を確認する   |

## 2. 作成前に集める情報

最初から正確に書こうとせず、以下を短いメモで集める。

| 項目           | 集める情報                                                     |
| -------------- | -------------------------------------------------------------- |
| 背景と必要性   | なぜ取り組むか、放置すると何が悪化するか                       |
| 困っている現象 | 利用者や運用で実際に起きている問題（解決策ではなく現象）       |
| 影響           | 課題を放置した場合の損失、リスク、誰がどう困るか               |
| 原因の手がかり | 情報不足、判断基準の不在、手順のばらつき、システム制約など     |
| 解決の選択肢   | 取りうる方針の候補（最低 2 案）                                |
| 判断軸         | 効果、コスト、期間、リスク、再利用性、公開適性など重視する観点 |
| 前提・制約     | スコープ、前提・制約・依存関係から、守るべき境界と成立条件     |
| 未確定事項     | 追加調査、確認、承認が必要な事項と、その責任ロール             |

## 3. 全体の作成手順

1. H1 で対象プロジェクト名を明示し、本書の目的（何が課題で、どれを、どう解く方針かを合意する材料）と利用ロール（PO は方針承認、BA は利用者影響整理、ARC は技術的境界識別、PM は作業計画入力など）、合意対象・非合意対象を 1〜2 文で書く。
2. 課題一覧で、困っている現象、影響、優先度を書く。
3. 原因で、事実と仮説を分けて、課題ごとの発生要因を書く。
4. 解決策候補で、最低 2 案を概要と期待できる効果とともに並べる。
5. 採用アプローチと理由で、採用案・採用しない案を判断軸で説明する。
6. トレードオフ/リスクで、捨てるもの・起きうることと軽減策を書く。
7. 次の検討事項で、未確定事項と責任ロールを書く。
8. 最後に、設計詳細や曖昧表現を書きすぎていないか確認する。

## 4. 各章の書き方

### 4.1. 課題一覧

問い:

- 利用者や運用で、実際に困っている現象は何か。
- 放置すると、誰が、どの業務価値で、どう困るか。
- どの課題から先に決めるべきか（優先度）。

書き方:

- 課題は「困っている現象」で書き、解決策の欠落（例:「自動発注がない」）にしない。
- 各課題に最低限「影響」と「優先度」を添える。
- 課題ID（例: P-01）を振り、原因表や決定記録から参照できるようにする。

良い例:

- 文書の粒度・構成・記述ルールがプロジェクトごとにばらつき、再利用・レビュー・引き継ぎが難しい。

悪い例:

- 共通テンプレートがない。

### 4.2. 原因（仮説でも可）

問い:

- その課題は、なぜ起きているか。
- 確認できている事実はどれで、推測はどれか。
- 根拠となる文書や観察事実は何か。

書き方:

- 原因区分（事実 / 仮説）を明示し、混ぜない。
- 課題IDで課題一覧と対応づける。
- 根拠文書、観察事実、補足を添える。

### 4.3. 解決策候補

問い:

- 課題を解くために取りうる方針は何か。
- 各案は、どの業務価値に近づけるか。
- 案ごとに失うもの・効果は何か。

書き方:

- 最低 2 案を、案ID・概要・期待できる効果とともに並べる。
- 単一案の押し付けにしない。
- 案IDを振り、採用アプローチや代替案比較から参照できるようにする。

### 4.4. 採用アプローチと理由

問い:

- どの案を中核にし、どの案を補助的に組み合わせるか。
- 採用・非採用の判断軸は何で、今回重視する順序はどうか。
- 採用しない案は、なぜ採用しないか。

書き方:

- 採用案を明記し、判断軸で採用理由を説明する。
- 「採用対象」「判定」「理由」の表にすると、採否と根拠が読み取りやすい。
- 段階的に組み合わせる場合は、中核案と補助案を区別する。
- 判断軸の優先順位を 1 文で示す。

### 4.5. トレードオフ/リスク

問い:

- 採用の代償として、何を捨てる・先送りする・我慢するか。
- 起きうる失敗・遅延・品質低下は何か。
- それぞれの軽減策は何か。

書き方:

- 区分（`トレードオフ` / `リスク`）で揃え、課題ログやリスク登録簿へ転記しやすい粒度にする。区切りの `/` の前後にはスペースを置かない。
- リスクは「起きうること」と「軽減策」をセットで書く。

### 4.6. 次の検討事項（ToDo）

問い:

- 次の意思決定に必要な追加調査・確認・承認は何か。
- それぞれの責任ロールは誰か。

書き方:

- 「ToDo」「目的」「責任ロール」の表にする。
- ToDo の項目は、未確定事項を示す場合は `_TODO_:` または `_UNDECIDED_` ラベルを先頭に付け、後続で検索・フィルタできるようにする（例: `_TODO_`: 公開先にGitHub を採用するか決定する）。
- 責任ロールは、体制定義で採用済みのロールまたは担当単位で書く。

## 5. 深掘り手順

内容が浅いと感じる場合は、次の順で深掘りする。

1. **困っている現象** を、解決策抜きで言い換える。
2. **影響** を、誰がどの業務価値で困るかで具体化する。
3. **原因** を、事実と仮説に分けて分解する。
4. **解決策候補** を 2 案以上に広げる。
5. **判断軸の優先順位** を決める。
6. **採用・非採用の理由** を判断軸で説明する。
7. **未確定事項と責任ロール** を残す。

## 6. 良い例 / 悪い例

| 観点       | 良い例                                                            | 悪い例                   |
| ---------- | ----------------------------------------------------------------- | ------------------------ |
| 課題       | 記録ルールがばらつき、引き継ぎ時に確認に時間がかかる              | テンプレートがない       |
| 原因       | 事実: 担当ごとに記述様式が異なる / 仮説: 共通様式が浸透していない | なんとなく属人化している |
| 解決策候補 | A 案: ルール先行で整備 / B 案: サンプル先行で整備（2 案を提示）   | とにかく標準化する       |
| 採用理由   | 再利用性を最優先に A 案を中核、B 案を補助として採用               | A 案がよさそうだから採用 |
| リスク     | 参考資料の整合が崩れる → 品質管理とレビューで確認する             | 特に問題はない           |

## 7. レビュー観点

作成後は、次の観点で確認する。

| 観点                 | 確認内容                                                           |
| -------------------- | ------------------------------------------------------------------ |
| 業務価値との対応     | 課題と解決アプローチが、業務目的・利用者・期待効果と対応しているか |
| 課題の妥当性         | 課題が困っている現象で書かれ、解決策の先取りになっていないか       |
| 事実と仮説の区別     | 原因で事実と仮説が混ざっていないか                                 |
| 候補の網羅           | 解決策候補が最低 2 案あり、単一案の押し付けになっていないか        |
| 採用理由の説明可能性 | 採用・非採用が判断軸で説明され、後から再現できるか                 |
| 関係者・利用場面     | 関係者、利用場面、確認者、合意対象が読み取れるか                   |
| 責務分担             | 設計詳細・実装詳細・テスト設計を書きすぎていないか                 |
| 未確定の明示         | 未確定事項が `_TODO_` / `_UNDECIDED_` 等で明示されているか         |

## 8. 仕上げチェック

- [[prj-issues-and-approach-rulebook]] の本文構成に沿っている。
- [[prj-issues-and-approach-sample]] と同程度の粒度で書けている。
- 課題一覧、原因、解決策候補、採用アプローチと理由がすべて記述されている。
- 解決策候補が最低 2 案ある。
- 採用アプローチに判断軸と採用理由がある。
- 曖昧表現や設計詳細を書きすぎていない。
