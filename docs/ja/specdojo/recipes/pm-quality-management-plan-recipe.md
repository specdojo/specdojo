# 品質管理計画 作成レシピ

Quality Management Plan Writing Recipe

本ドキュメントは、品質管理計画を「PM が計画・進捗・課題・リスク管理へ接続でき、QE と PO が品質判断に使える内容」に仕上げるための作り方である。構造と必須項目は [[pm-quality-management-plan-rulebook|品質管理計画 作成ルール]] を正とし、本書では問い、深掘り手順、レビュー観点を扱う。

## 1. このレシピの使い方

- 最初に、上位の管理計画から品質、リスク、課題、変更、公開判断の扱いを確認する。
- 次に、品質目標、レビュー種別、メトリクス、検査基準、是正プロセスを、PM が管理台帳や Schedule へ接続できる単位へ分解する。
- 最後に、QE が判定できる基準と、PO が最終判断する条件が混在していないか確認する。

| 種別 | 役割 | 使いどころ |
| --- | --- | --- |
| rulebook | 成果物として成立するための規約 | 構造、必須項目、禁止事項を確認する |
| recipe | 良い内容を書くための作り方 | 問い、深掘り、レビューに使う |
| sample | 完成例 | 粒度、文体、表の書き方を確認する |
| template | 記入の骨組み | 新規作成時に章とプレースホルダを用意する |

## 2. 作成前に集める情報

| 項目 | 集める情報 |
| --- | --- |
| 上位管理方針 | 品質、リスク、課題、変更、公開判断の方針 |
| Role code | 品質確認、レビュー、是正、公開判断に関わる採用 Role code |
| RACI | 品質管理計画自身と成果物レビュー、公開可否判断の責任分担 |
| 品質対象 | 成果物、参考資料、生成物、公開対象、AI Agent の入力・出力 |
| レビュー種別 | 構造、内容、整合性、公開前、AI 利用などのレビュー区分 |
| 検証手段 | lint、schema、build、リンク確認、レビュー記録などの確認方法 |
| 是正経路 | 軽微修正、構造不適合、内容不整合、公開不適合、重大不適合の扱い |
| 管理台帳への接続 | 課題、リスク、変更要求、決定記録へ転記する条件 |

## 3. 全体の作成手順

1. H1 直下に、品質管理計画が PM の計画・進捗・課題・リスク管理へどう接続するかを一文で記述する。
2. 概要で、上位計画、Role code の正本、PM / QE / PO の責務境界、AI Agent の支援範囲を明確にする。
3. 品質目標に、目標値、判定方法、タイミング、責任ロールを置く。
4. レビュープロセスに、レビュー種別と手順を置き、出口条件を判定可能にする。
5. 品質メトリクスに、算出方法、閾値、計測頻度、報告先を置く。
6. 検査基準と是正に、受入基準、不適合区分、是正方法、再判定、生成物の扱いを置く。
7. 品質上の役割分担に、採用 Role code ごとの品質管理上の責務を置く。
8. 関連ドキュメントで、上位計画、正本、委譲先、管理台帳、生成ビューへの導線を整理する。
9. 見直し条件と未決事項に、更新トリガー、後続判断、判断者を残す。

## 4. 各章の書き方

### 4.1. 概要

問い:

- この計画は、PM がどの品質事項を管理するための文書か。
- QE が確認する事項と、PO が最終判断する事項は分かれているか。
- AI Agent の支援範囲が、人間の判断責任を代替していないか。

書き方:

- 「PM は品質確認を計画、進捗、課題、リスク、変更要求へ接続する」のように管理接続を明示する。
- 採用 Role code と実行主体を分け、具体的な member や agent は別文書へ委譲する。
- 公開可否、重大不適合、変更要求の採否は PO 判断に残す。

### 4.2. 品質目標

問い:

- 各目標は、いつ、誰が、何を見れば達成判定できるか。
- 目標値が定性表現だけになっていないか。
- 成果物、参考資料、生成物、公開対象、AI 利用の観点が漏れていないか。

書き方:

- 目標値は「0 件」「全対象」「公開前に確認済み」のように判定可能な条件で書く。
- 判定方法には、確認する文書、レビュー記録、コマンド、管理台帳を含める。
- 責任ロールは Role code で書き、個人名や agent 名を置かない。

### 4.3. レビュープロセス

問い:

- レビュー種別ごとに対象、実施ロール、頻度、出口条件が分かるか。
- 指摘事項をどこに記録し、いつ再判定するか。
- 重大指摘や公開不適合を PO 判断へ上げる条件があるか。

書き方:

- 構造、内容、整合性、公開前、AI 利用のように、確認観点が異なるものを分ける。
- 出口条件は「指摘クローズ」だけで終わらせず、何に対する不整合がないかを書く。
- 手順では、対象特定、根拠確認、指摘記録、是正、再判定、最終判断への接続を順に示す。

### 4.4. 品質メトリクス

問い:

- 指標ごとに算出方法、閾値、計測頻度、報告先があるか。
- 確定していないツールやコマンドを確定事項として書いていないか。
- 閾値超過時に課題、リスク、変更要求へ分離できるか。

書き方:

- lint、schema、build、リンク不整合、重大レビュー指摘残、公開不適切情報などを表にする。
- コマンドがある場合はコマンド名を明記し、ない場合は確認方法を文書名や記録先で示す。
- 報告先は `PM`, `QE`, `PO`, `ARC`, `OPS` などの Role code で書く。

### 4.5. 検査基準と是正

問い:

- 完了または公開候補として扱う条件が、構造、内容、整合、追跡、公開、AI 利用に分かれているか。
- 不適合の区分ごとに、是正方法と再判定者が分かるか。
- 生成物と正本が矛盾した場合、どちらを直すかが明確か。

書き方:

- 検査基準は、観点と受入基準の表にする。
- 是正プロセスは、軽微、構造不適合、内容不整合、公開不適合、重大のように扱いを分ける。
- 重大な不整合は本文に残さず、PJR、課題、リスク、変更要求、決定記録へ接続する。

### 4.6. 品質上の役割分担

問い:

- 採用 Role code ごとの品質管理上の責務が読めるか。
- Role code の定義全文を複製していないか。
- 未採用 Role code、member nickname、人名、agent 名が混入していないか。

書き方:

- `PM` は管理接続、`QE` は品質確認、`PO` は最終判断のように使い分けを書く。
- `ARC`, `BA`, `OPS` などは、品質に関係する確認範囲だけを書く。
- 実行主体の割り当てや兼務は、組織・メンバー定義へ委譲する。

## 5. 深掘り手順

1. 品質対象を「成果物」「参考資料」「生成物」「公開対象」「AI Agent 入出力」に分ける。
2. 各対象について、確認観点、判定方法、責任 Role code、記録先を決める。
3. 不適合が発生したときに、軽微修正、課題、リスク、変更要求、決定記録のどれへ分離するかを決める。
4. メトリクスの閾値を超えたときの報告先と再判定者を決める。
5. Role code、RACI、Schedule `owner`、実行主体の割り当てが混ざっていないか確認する。
6. 公開不適切情報、認証情報、秘密情報が本文やサンプルに入っていないか確認する。

## 6. 良い例 / 悪い例

| 観点 | 良い例 | 悪い例 |
| --- | --- | --- |
| 品質目標 | Markdown lint の失敗件数を 0 件にする | 文書品質を高める |
| 判定方法 | rulebook、sample、lint、レビュー記録で確認する | 担当者が確認する |
| レビュー出口条件 | rulebook と成果物カタログに対する不整合がない | 問題がなければ完了 |
| 是正 | 構造不適合は rulebook に合わせて修正し、QE と ARC が再確認する | 指摘があれば直す |
| Role code | 重大不適合の最終判断は `PO`、品質確認は `QE` と書く | 品質担当者が判断する |
| 公開適性 | 個人情報、認証情報、秘密情報を記載しない | 公開前に消す前提で本文に入れる |

## 7. レビュー観点

| 観点 | 確認内容 |
| --- | --- |
| 計画化可能性 | 品質確認、レビュー、検証、是正がタスク化・進捗確認に使える粒度か |
| 全体計画との整合 | `pm-plan` の品質・リスク・課題管理方針と矛盾していないか |
| 判定可能性 | 品質目標、出口条件、メトリクス、検査基準が判定可能か |
| 管理台帳への接続 | 不適合、未決事項、リスク、変更要求、決定事項へ転記すべき事項が分離されているか |
| 判断責任 | PM の整理責任、QE の確認責任、PO の最終判断が混同されていないか |
| 下流入力 | PO が承認し、QE が検証・レビュー手順を判定できる入力になっているか |
| 公開適性 | 個人情報、非公開情報、認証情報、秘密情報を含んでいないか |

## 8. 仕上げチェック

- [[pm-quality-management-plan-rulebook|品質管理計画 作成ルール]] の本文構成と禁止事項に従っている。
- [[pm-quality-management-plan-sample|品質管理計画 sample]] と同程度の粒度で、表と箇条書きが完成している。
- `pm-plan.md` の品質・リスク・課題管理方針と矛盾していない。
- 品質目標、レビュー出口条件、品質メトリクス、検査基準が判定可能である。
- PM、QE、PO の責務境界が Role code で記述されている。
- 課題、リスク、変更要求、決定記録へ転記すべき事項が分離されている。
- member nickname、人名、agent 名を責任主体にしていない。
- 公開不適切情報、認証情報、秘密情報を含んでいない。
