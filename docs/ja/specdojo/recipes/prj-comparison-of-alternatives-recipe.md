# 代替案比較 作成レシピ

Comparison of Alternatives Writing Recipe

本ドキュメントは、代替案比較を「採択案、非採択案、技術的影響、見直し条件を説明できる内容」に仕上げるための作り方です。
構造や必須項目は [[prj-comparison-of-alternatives-rulebook]] を正とし、本書では問い、深掘り手順、具体化のコツ、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、スコープ、課題と解決アプローチ、前提・制約から比較に使う決定事項だけを抜き出す。
- 次に、代替案を最低 2 案に広げ、同じ評価軸で比較する。
- 最後に、採択案、非採択案、技術的影響、リスク、再評価条件を残し、判断の再現性を確保する。

`rulebook` / `recipe` / `sample` の役割は次のように分ける。

| 種別     | 役割                           | 使いどころ                         |
| -------- | ------------------------------ | ---------------------------------- |
| rulebook | 成果物として成立するための規約 | 構造、必須項目、禁止事項を確認する |
| recipe   | 良い内容を書くための作り方     | 問い、観点、深掘り、レビューに使う |
| sample   | 完成例                         | 粒度、文体、表の書き方を確認する   |

## 2. 作成前に集める情報

最初から比較表を書かず、以下を短いメモで集める。

| 項目           | 集める情報                                                   |
| -------------- | ------------------------------------------------------------ |
| 比較目的       | 何を決めるための比較か、どの判断を支援するか                 |
| 根拠資料       | スコープ、課題と解決アプローチ、前提・制約、関連する決定事項 |
| 代替案         | 同じ課題を解ける候補案。採択しない可能性がある案も含める     |
| 評価軸         | 効果、コスト、期間、リスク、運用適合、技術実現性             |
| 技術的成立条件 | 既存資産、外部依存、制約、後続成果物への影響                 |
| 採択判断者     | 最終判断を行う人間の責任ロール。通常は `PO`                  |
| 再評価条件     | 方針を見直すトリガー、次に確認する指標、記録先               |

## 3. 全体の作成手順

1. H1 で比較対象の成果物名またはプロジェクト名を明示する。
2. 比較目的と前提条件で、何を決めるための比較かを短く書く。
3. 比較対象案で、案 ID、概要、対象範囲を揃えて並べる。
4. 評価軸と評価基準で、案ごとに変えない共通の軸を定義する。
5. 比較結果と採択理由で、採択案、非採択案、一部採択案を判断軸で説明する。
6. 技術的実現性と影響を、後続成果物や運用への影響まで含めて書く。
7. リスクとトレードオフで、採択によって受容する不利条件と軽減策を書く。
8. 決定と見直しで、最終判断者、再評価条件、記録先を明示する。

## 4. 各章の書き方

### 4.1. 比較目的と前提条件

問い:

- この比較で、どの判断を支援するのか。
- 比較対象に含める範囲と、含めない範囲は何か。
- 根拠資料から引き継ぐ制約、用語、既定方針は何か。

書き方:

- 比較目的は 1〜2 文で書き、設計詳細や実装方式の説明に寄せすぎない。
- 根拠資料は、直接判断に使う文書に絞る。
- 未確定事項は `_TODO_` / `_UNDECIDED_` / `_ASSUMPTION_` で明示する。

### 4.2. 比較対象案

問い:

- 同じ目的を達成しうる案が最低 2 案あるか。
- 各案の対象範囲は同じ粒度で説明できているか。
- 課題と解決アプローチの案 ID と対応させられるか。

書き方:

- 案 ID は `ALT-01` のように安定した識別子にする。
- 案名は手段だけでなく、優先する方針が分かる名前にする。
- 対象範囲には、成果物、活動、機能、運用範囲を混在させすぎない。

### 4.3. 評価軸と評価基準

問い:

- 評価軸は案ごとに変わっていないか。
- 業務価値、意思決定、技術制約、リスクの観点が揃っているか。
- `High` / `Middle` / `Low` の判定理由を説明できるか。

書き方:

- 効果、コスト、期間、リスク、運用適合、技術実現性を最低ラインにする。
- 重みを使う場合は、重みの理由を備考に書く。
- 低いほどよい軸は、備考で判定方向を明示する。

### 4.4. 比較結果と採択理由

問い:

- 採択案は、どの評価軸で優位なのか。
- 非採択案や一部採択案の理由は、再評価時に使える粒度か。
- 技術的実現性と後続成果物への影響が読み取れるか。

書き方:

- 比較結果表の直後に、採択案と組み合わせ方を 1 文で書く。
- 採択理由は 3〜5 点に絞り、評価軸に対応させる。
- 技術的実現性と影響は、採択理由とは別表にして埋もれないようにする。

### 4.5. リスクとトレードオフ

問い:

- 採択によって先送りする範囲、受容する負荷、失う選択肢は何か。
- 起きうる失敗と軽減策は対になっているか。
- PM や後続作業がリスク登録簿へ転記できる粒度か。

書き方:

- 区分は `リスク` / `トレードオフ` など少数に揃える。
- 軽減策は「確認する」だけで終えず、何を残すか、誰が判断するかまで書く。
- 技術詳細に踏み込む場合は、後続の設計成果物へ委譲する。

### 4.6. 決定と見直し

問い:

- 現時点の採択方針は一文で言えるか。
- 採択案の最終判断者が人間の責任ロールとして明記されているか。
- どの条件で再評価し、どこに判断理由を残すか。

書き方:

- `PO` が最終判断し、AI Agent は比較分析の支援に留めることを明記する。
- 再評価条件は、時間、指標、利用者フィードバック、依存変更など具体的なトリガーにする。
- 方針変更時の記録先を、決定記録、変更要求ログ、プロジェクト登録簿などから選ぶ。

## 5. 深掘り手順

内容が浅いと感じる場合は、次の順で深掘りする。

1. 比較目的を「何を決めるためか」に言い換える。
2. 案を増やすのではなく、同じ課題を解く代表案に絞る。
3. 評価軸を業務価値、費用、期間、リスク、運用、技術に分類する。
4. `High` / `Middle` / `Low` の根拠を 1 行ずつ補う。
5. 採択案の不利条件をトレードオフとして書く。
6. 非採択案が再浮上する条件を明示する。
7. 最終判断者と記録先を確認する。

## 6. 良い例 / 悪い例

| 観点       | 良い例                                                           | 悪い例                                     |
| ---------- | ---------------------------------------------------------------- | ------------------------------------------ |
| 比較目的   | 初期公開までに再利用可能な文書体系を成立させる方針を選ぶ         | どれがよいか考える                         |
| 代替案     | ALT-01 ルール先行、ALT-02 サンプル先行、ALT-03 ツール先行        | ちゃんと整備する                           |
| 評価軸     | 効果、コスト、期間、リスク、運用適合、技術実現性を同じ基準で評価 | 案ごとに都合のよい観点だけを書く           |
| 採択理由   | 再利用性と AI 利用可能性を重視し、ALT-01 を中核にする            | ALT-01 がよさそうだから                    |
| 技術影響   | 既存の文書配置と lint で始められ、後続 template の基準になる     | 実現性や影響の根拠を書かずに結論だけ述べる |
| 見直し条件 | 初回利用者の導入で迷いが多い場合、サンプル優先の比重を上げる     | 必要なら見直す                             |

## 7. レビュー観点

作成後は、次の観点で確認する。

| 観点                 | 確認内容                                                     |
| -------------------- | ------------------------------------------------------------ |
| 比較目的の明確性     | 何を決めるための比較かが最初に分かるか                       |
| 根拠資料との整合     | スコープ、課題と解決アプローチ、前提・制約と矛盾していないか |
| 候補案の妥当性       | 最低 2 案あり、同じ目的を解ける案として並んでいるか          |
| 評価軸の一貫性       | 案ごとに評価軸や判定基準を変えていないか                     |
| 採択理由の説明可能性 | 採択、一部採択、非採択の理由が評価軸で説明されているか       |
| 技術的実現性         | 外部依存、既存資産、後続成果物への影響が読み取れるか         |
| 人間の判断責任       | 採択案の最終判断者が `PO` など人間の責任ロールになっているか |
| 再評価可能性         | 見直し条件と記録先が明記されているか                         |

## 8. 仕上げチェック

- [[prj-comparison-of-alternatives-rulebook]] の本文構成に沿っている。
- [[prj-comparison-of-alternatives-sample]] と同程度の粒度で書けている。
- 比較対象案が最低 2 案ある。
- 評価軸と評価基準が案ごとに固定されている。
- 採択案、非採択案、一部採択案の理由が記載されている。
- 技術的実現性と後続成果物への影響が比較されている。
- リスクとトレードオフに軽減策がある。
- 採択案の最終判断者と再評価条件が明記されている。
