# 代替案比較 作成レシピ

Comparison of Alternatives Writing Recipe

本ドキュメントは、代替案比較を「採択案、非採択案、技術的影響、見直し条件を説明できる内容」に仕上げるための作り方です。
本書では問い、深掘り手順、具体化のコツ、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、スコープ、課題と解決アプローチ、前提・制約から、解くべき課題、必要能力、制約、評価条件を抜き出す。
- 次に、既存標準・開発方式・運営方式・ツールを調査し、同じ意思決定を代替できる戦略案を最低 2 案に広げる。
- 運営・開発戦略とタスク・状態管理方式など抽象度が異なる判断は、比較レイヤーを分ける。
- 最後に、採択案、非採択案、技術的影響、リスク、再評価条件を残し、判断の再現性を確保する。

## 2. 作成前に集める情報

最初から比較表を書かず、以下を短いメモで集める。

| 項目           | 集める情報                                                   |
| -------------- | ------------------------------------------------------------ |
| 比較目的       | 何を決めるための比較か、どの判断を支援するか                 |
| 根拠資料       | スコープ、課題と解決アプローチ、前提・制約、関連する決定事項 |
| 必要能力       | 課題を解くために必要な能力。特定の解決手段とは分けて集める   |
| 外部候補       | 既存標準、開発方式、運営方式、製品、ツールと適用事例         |
| 代替案         | 同じ意思決定を代替できる一式の戦略案                         |
| 比較レイヤー   | 運営・開発戦略、実行プロセス、タスク・状態管理方式など       |
| 評価軸         | 効果、コスト、期間、リスク、運用適合、技術実現性             |
| 技術的成立条件 | 既存資産、外部依存、制約、後続成果物への影響                 |
| 採択判断者     | 最終判断を行う人間の責任ロール。通常は `PO`                  |
| 再評価条件     | 方針を見直すトリガー、次に確認する指標、記録先               |

## 3. 全体の作成手順

1. H1 で比較対象の成果物名またはプロジェクト名を明示する。
2. 比較目的と前提条件で、何を決めるための比較か、どの比較レイヤーを扱うかを短く書く。
3. 前段文書から課題・制約・必要能力を抽出し、既存標準や有力な外部方式を追加調査する。
4. 比較対象案で、単独でも目的達成への道筋を説明できる戦略案を、案 ID、概要、主要構成、対象範囲を揃えて並べる。
5. 抽象度の異なる支援方式は、戦略案と分離した表で並べる。
6. 評価軸と評価基準で、案ごとに変えない共通の軸を定義する。
7. 比較結果と採択理由で、採択案、非採択案、一部採択案を判断軸で説明する。
8. 技術的実現性と影響を、後続成果物や運用への影響まで含めて書く。
9. リスクとトレードオフで、採択によって受容する不利条件と軽減策を書く。
10. 決定と見直しで、最終判断者、再評価条件、記録先を明示する。

## 4. 各章の書き方

### 4.1. 比較目的と前提条件

問い:

- この比較で、どの判断を支援するのか。
- 運営・開発戦略、実行プロセス、支援方式のどの比較レイヤーを扱うのか。
- 比較対象に含める範囲と、含めない範囲は何か。
- 根拠資料から引き継ぐ課題、必要能力、制約、評価条件は何か。
- 根拠資料が示す対象外範囲や、初期段階で優先する最小成立範囲は、前提条件に引き継いでいるか。

書き方:

- 比較目的は 1〜2 文で書き、設計詳細や実装方式の説明に寄せすぎない。
- 根拠資料は、直接判断に使う文書に絞る。
- 未確定事項は _TODO_ / _UNDECIDED_ / _ASSUMPTION_ で明示する。
- 対象外とする範囲や、全体を一度に完成させきれない場合の扱いがあれば、前提条件で明示する。

### 4.2. 比較対象案

問い:

- 同じ目的を達成しうる案が最低 2 案あるか。
- 各案の対象範囲は同じ粒度で説明できているか。
- 各案は単独でも目的達成への道筋を説明できるか。
- 現行案と根拠の異なる、採択しない可能性がある有力な外部案を含めたか。
- 原則、施策、ツールを別々の戦略案として混在させていないか。

書き方:

- 案 ID は `ALT-01` のように安定した識別子にする。
- 案名は手段だけでなく、優先する方針が分かる名前にする。
- 対象範囲には、成果物、活動、機能、運用範囲を混在させすぎない。
- 前段文書の案 ID を再掲する場合も、それだけで候補を閉じず、外部調査で得た候補を追加する。
- 相互に組み合わせる原則や施策は、一式の戦略案へまとめる。
- タスク・状態管理ツールなど抽象度の低い方式は、戦略案とは別の支援方式表で比較する。

### 4.3. 評価軸と評価基準

問い:

- 評価軸は案ごとに変わっていないか。
- 業務価値、意思決定、技術制約、リスクの観点が揃っているか。
- 不確実性への適応、トレーサビリティ、人間の判断責任、AI Agent の長期作業と検証可能性を評価できるか。
- `High` / `Middle` / `Low` の判定理由を説明できるか。
- 評価軸は、根拠資料の課題 ID や業務価値のどの項目に対応するか説明できるか。

書き方:

- 効果、コスト、期間、リスク、運用適合、技術実現性を最低ラインにし、比較目的に応じて適応性、追跡性、AI 適合、成熟度を追加する。
- 重みを使う場合は、重みの理由を備考に書く。
- 低いほどよい軸は、備考で判定方向を明示する。
- 評価軸の説明には、根拠資料の課題 ID や業務価値との対応を添え、評価が業務価値から離れないようにする。

### 4.4. 比較結果と採択理由

問い:

- 採択案は、どの評価軸で優位なのか。
- 非採択案や一部採択案の理由は、再評価時に使える粒度か。
- 技術的実現性と後続成果物への影響が読み取れるか。

書き方:

- 比較結果表の直後に、採択案と組み合わせ方を 1 文で書く。
- 採択理由は 3〜5 点に絞り、評価軸に対応させる。
- 技術的実現性と影響は、採択理由とは別表にして埋もれないようにする。

### 4.5. リスクとトレードオフ

問い:

- 採択によって先送りする範囲、受容する負荷、失う選択肢は何か。
- 起きうる失敗と軽減策は対になっているか。
- PM や後続作業がリスク登録簿へ転記できる粒度か。

書き方:

- 区分は `リスク` / `トレードオフ` など少数に揃える。
- 軽減策は「確認する」だけで終えず、何を残すか、誰が判断するかまで書く。
- 技術詳細に踏み込む場合は、後続の設計成果物へ委譲する。
- 軽減策や記録先の具体名称（計画書、登録簿等）は、根拠資料で存在が確認できる仕組みに限定する。根拠資料に存在しない計画書名・台帳名を新設せず、確認できない場合は「後続の品質確認」「成果物カタログと後続作業」のように確認済みの仕組みに言い換える。

### 4.6. 決定と見直し

問い:

- 現時点の採択方針は一文で言えるか。
- 採択案の最終判断者が人間の責任ロールとして明記されているか。
- どの条件で再評価し、どこに判断理由を残すか。

書き方:

- `PO` が最終判断し、AI Agent は比較分析の支援に留めることを明記する。
- 再評価条件は、時間、指標、利用者フィードバック、依存変更など具体的なトリガーにする。
- 方針変更時の記録先を、決定記録、変更要求ログ、プロジェクト登録簿などから選ぶ。

## 5. 深掘り手順

内容が浅いと感じる場合は、次の順で深掘りする。

1. 比較目的を「何を決めるためか」に言い換える。
2. 前段文書から、課題、必要能力、制約、評価条件を抽出する。
3. 現行案から独立して、既存標準、開発方式、運営方式、製品、ツールを調査する。
4. 候補を比較レイヤー別に分け、同じ意思決定を代替できる代表案に絞る。
5. 各案を単独で目的達成への道筋を説明できる戦略パッケージにする。
6. 評価軸を業務価値、費用、期間、リスク、運用、技術に分類する。
7. 評価軸を根拠資料の課題 ID や業務価値に対応付ける。
8. `High` / `Middle` / `Low` の根拠を 1 行ずつ補う。
9. 採択案の不利条件をトレードオフとして書く。
10. 非採択案が再浮上する条件と、最終判断者、記録先を明示する。

## 6. 良い例 / 悪い例

| 観点         | 良い例                                                           | 悪い例                                                         |
| ------------ | ---------------------------------------------------------------- | -------------------------------------------------------------- |
| 比較目的     | 初期公開までに再利用可能な文書体系を成立させる方針を選ぶ         | どれがよいか考える                                             |
| 代替案       | 予測型、アジャイル型、仕様駆動ハイブリッド型を戦略一式で比較     | Why、仕様、ツールを別案として並べ、すべて採択する              |
| 比較レイヤー | 運営戦略とタスク管理方式を分けて比較                             | アジャイルと個別タスク管理ツールを同じ表で採点                 |
| 評価軸       | 効果、コスト、期間、リスク、運用適合、技術実現性を同じ基準で評価 | 案ごとに都合のよい観点だけを書く                               |
| 採択理由     | 再利用性と AI 利用可能性を重視し、ALT-01 を中核にする            | ALT-01 がよさそうだから                                        |
| 技術影響     | 既存の文書配置と lint で始められ、後続 template の基準になる     | 実現性や影響の根拠を書かずに結論だけ述べる                     |
| 見直し条件   | 初回利用者の導入で迷いが多い場合、サンプル優先の比重を上げる     | 必要なら見直す                                                 |
| 軽減策の根拠 | 後続の品質確認で対応する（根拠資料にある仕組みを指す）           | 品質管理計画で対応する（根拠資料に存在しない計画名を断定する） |

## 7. レビュー観点

作成後は、次の観点で確認する。

| 観点                 | 確認内容                                                                                             |
| -------------------- | ---------------------------------------------------------------------------------------------------- |
| 比較目的の明確性     | 何を決めるための比較かが最初に分かるか                                                               |
| 根拠資料との整合     | スコープ、課題と解決アプローチ、前提・制約と矛盾していないか                                         |
| 候補案の妥当性       | 最低 2 案あり、同じ目的を解ける案として並んでいるか                                                  |
| 外部候補の網羅       | 前段文書の案の再掲だけでなく、既存標準・方式・製品を調査しているか                                   |
| 比較レイヤーの統一   | 戦略、原則、施策、ツールが同じ表に混在していないか                                                   |
| 評価軸の一貫性       | 案ごとに評価軸や判定基準を変えていないか。根拠資料の課題 ID・業務価値に対応しているか                |
| 採択理由の説明可能性 | 採択、一部採択、非採択の理由が評価軸で説明されているか                                               |
| 技術的実現性         | 外部依存、既存資産、後続成果物への影響が読み取れるか                                                 |
| 軽減策の実在性       | 軽減策や記録先が根拠資料で確認できる仕組みに基づいているか。未確認の計画書名・台帳名を作っていないか |
| 人間の判断責任       | 採択案の最終判断者が `PO` など人間の責任ロールになっているか                                         |
| 再評価可能性         | 見直し条件と記録先が明記されているか                                                                 |

## 8. 仕上げチェック

- 「全体の作成手順」で示した章立てと必須表がそろっている。
- 比較対象案が最低 2 案ある。
- 各案が単独で目的達成への道筋を説明でき、同じ比較レイヤーにある。
- 前段文書の候補案を名称変更して再掲するだけでなく、有力な外部案を調査している。
- 運営・開発戦略とタスク・状態管理方式を必要に応じて分離している。
- 評価軸と評価基準が案ごとに固定されている。
- 採択案、非採択案、一部採択案の理由が記載されている。
- 評価軸が根拠資料の課題 ID や業務価値に対応している。
- 技術的実現性と後続成果物への影響が比較されている。
- リスクとトレードオフに軽減策がある。
- 軽減策・記録先が根拠資料で確認できる仕組みに基づいている。
- 採択案の最終判断者と再評価条件が明記されている。
