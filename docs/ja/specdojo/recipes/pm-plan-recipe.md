# プロジェクト管理計画 作成レシピ

Project Management Plan Writing Recipe

本ドキュメントは、プロジェクト管理計画を「PM が計画化、進捗確認、課題・リスク管理、変更管理に使える内容」に仕上げるための作り方です。構造と必須項目は [[pm-plan-rulebook|プロジェクト管理計画 作成ルール]] を正とし、本書では問い、深掘り手順、レビュー観点を扱います。

## 1. このレシピの使い方

- 最初に、組織定義とロール定義から、採用 Role code、PM と PO の責務境界、実行主体の割り当て先を確認する。
- 次に、スコープ、スケジュール、コスト、品質、リスク、課題、変更管理を、PM が運用できる管理単位へ分解する。
- 最後に、進捗、課題、リスク、変更要求、決定記録へ転記すべき事項が本文に混在していないか確認する。

| 種別     | 役割                           | 使いどころ                               |
| -------- | ------------------------------ | ---------------------------------------- |
| rulebook | 成果物として成立するための規約 | 構造、必須項目、禁止事項を確認する       |
| recipe   | 良い内容を書くための作り方     | 問い、深掘り、レビューに使う             |
| sample   | 完成例                         | 粒度、文体、表の書き方を確認する         |
| template | 記入の骨組み                   | 新規作成時に章とプレースホルダを用意する |

- Frontmatter は template 自身の frontmatter をコピーせず、template の `frontmatter_template` フィールドの内容を生成物の Frontmatter として使う。`_PROJECT_ID_` を対象プロジェクト ID に置換し、`_TODO_` などの記入プレースホルダは本文と同様に埋める。詳細は [document-metadata-standard.md](../standards/document-metadata-standard.md) の `生成物 Frontmatter 雛形` を参照する。

## 2. 作成前に集める情報

| 項目                 | 集める情報                                                                                                                                                       |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 組織方針             | 採用 Role code、最終判断の集約先、AI Agent の支援範囲                                                                                                            |
| ロール定義           | `PM`、`PO`、`ARC`、`QE`、`OPS` などの project_note と使い分け                                                                                                    |
| ロール定義の文脈整合 | 参照したロール定義（`pm-roles.yaml` または代替の sample）の project_note が本プロジェクトの実態と一致するか。一致しない場合は Role code 語彙のみを正本として扱う |
| 責任分担             | `pm-plan` 自身と主要管理プロセスの RACI                                                                                                                          |
| スコープ管理単位     | 成果物、対象外、未決事項、変更要求として扱う条件                                                                                                                 |
| スケジュール管理単位 | 依存関係、順序、所要時間、期限超過、ブロック理由                                                                                                                 |
| コスト相当の制約     | 作業負荷、外部依存、運用負荷、再作業負荷                                                                                                                         |
| 品質基準             | 構造、一貫性、計画化可能性、公開適性                                                                                                                             |
| 管理台帳への接続     | 課題、リスク、変更要求、決定記録へ転記する単位                                                                                                                   |

## 3. 全体の作成手順

1. H1 直下に、PM が何を管理する計画かを一文で記述する。
2. 概要で、PM と PO の責務境界、Role code の正本、AI Agent の支援範囲を明確にする。
3. 基本方針の表に、管理領域、管理単位、PM の運用方針、最終判断、逸脱時対応を記入する。
4. スコープ、スケジュール、コストの各方針を、判断条件と記録先が分かる粒度へ具体化する。
5. 品質、リスク、課題、変更管理を、台帳へ転記できる単位で分離する。
6. 体制と意思決定に、Role code の使い分け、意思決定表、`pm-plan` 自身の RACI を置く。
7. レポーティングと見直しに、指標、報告タイミング、閾値超過時対応、GO / Not GO 判断材料を置く。
8. 関連ドキュメントで、正本、委譲先、管理台帳への導線を整理する。
9. 継続確認事項に、未決、暫定判断、後続対応、判断者を残す。

## 4. 各章の書き方

### 4.1. 概要

問い:

- この計画は、PM が何を管理するための文書か。
- PM が起案・整理する事項と、PO が最終判断する事項は分かれているか。
- Role code の正本と実行主体の割り当て先はどこか。
- 参照したロール定義の project_note は、本プロジェクトの実際の業務文脈と一致しているか。一致しない場合、その旨と語彙のみを正本とする扱いを明記しているか。

書き方:

- 「PM は計画・進捗・課題・リスク・変更要求を管理する」「PO は目的・優先順位・公開方針・変更採否を判断する」のように、責務を分けて書く。
- 小規模運用で PM 責務を兼務する場合でも、Schedule `owner` や RACI では Role code を使うと明記する。
- AI Agent の役割は支援に限定し、最終承認、公開可否に加えて説明責任も人間の `PO` に残すと明記する。
- ロール定義の project_note に本プロジェクトと異なる業務文脈（他プロジェクト向けの sample 由来の記述など）が残る場合は、Role code 語彙だけを正本として扱い、文脈の不一致を継続確認事項に分離する。

### 4.2. 管理方針（スコープ / スケジュール / コスト）

問い:

- 何を管理単位にすれば、作業順序、所要時間、進捗状態を判断できるか。
- 逸脱時に、課題、リスク、変更要求、決定記録のどれへ分離するか。
- 金額予算がない場合、何をコスト相当の制約として見るか。

書き方:

- 管理領域ごとに、PM が日々見る単位と、PO などへ判断を上げる条件を分ける。
- スケジュール管理には、Role code を `owner` に使うこと、member nickname や agent 名を使わないことを入れる。
- コスト管理は、作業負荷、外部依存、運用負荷、再作業負荷のように、判断可能な制約へ置き換える。

### 4.3. 品質・リスク・課題管理方針

問い:

- 品質の最低基準は、後続作業に使える粒度で判定できるか。
- 将来起きる可能性のある不確実性と、すでに起きた障害を分けているか。
- 変更要求として扱う条件と、通常更新で扱う条件は分かれているか。

書き方:

- 品質基準は、構造、一貫性、計画化可能性、公開適性など、PM が管理できる単位で表にする。
- リスクは「発生する可能性があるもの」、課題は「すでに発生したもの」として分ける。
- 変更管理では、目的、スコープ、主要成果物、Role code、責任分担、公開方針、Schedule への影響を判断条件にする。

### 4.4. 体制と意思決定

問い:

- 採用 Role code の一覧を複製せず、管理計画での使い分けだけを示せているか。
- 起案・整理、最終判断、相談先、証跡が分かれているか。
- `pm-plan` 自身の RACI は、RACI の正本と矛盾していないか。

書き方:

- Role code の project_note を丸写しせず、管理計画に関係する責務へ言い換える。
- project_note の業務文脈が本プロジェクトと異なる場合（汎用 sample をロール定義の代替として参照した場合など）は、本書でも Role code 語彙のみを正本として採用ロール表に反映し、文脈差異は本文に混在させず継続確認事項へ分離する。
- 意思決定表では、PM が判断するものと、PO へ上げるものを分ける。
- RACI は詳細を `pm-raci.md` に委譲し、本書では `pm-plan` 行だけを必要最小限で示す。

### 4.5. レポーティングと見直し

問い:

- 進捗、課題、リスク、変更要求、品質のどの指標を報告するか。
- どの状態になったら見直しや再計画が必要か。
- GO / Not GO 判断時に、PO が確認すべき材料はそろっているか。

書き方:

- 報告は「誰に、いつ、何を、閾値超過時にどうするか」を表にする。
- 見直し条件は、組織・ロール変更、スコープ変更、Schedule 変更、課題滞留、公開開始などのトリガーへ分ける。
- GO / Not GO は、PM が判断材料をそろえ、PO が最終判断する流れで書く。

## 5. 深掘り手順

1. 管理対象を「成果物」「タスク」「判断」「記録」に分ける。
2. 各管理対象について、PM が確認する指標と、PO へ上げる条件を分ける。
3. 依存未解決、入力不足、検証失敗、レビュー待ちが発生したときの記録先を決める。
4. リスク、課題、変更要求、決定記録のどれに該当するかを分類する。
5. Role code、Schedule `owner`、RACI、実行主体の割り当てが混ざっていないか確認する。
6. 公開不適切情報や認証情報が本文に入っていないか確認する。
7. 参照したロール定義（sample 代替を含む）の project_note が本プロジェクトの文脈と一致するかを確認し、不一致なら語彙のみを正本として扱う旨を継続確認事項に残す。

## 6. 良い例 / 悪い例

| 観点             | 良い例                                                                                          | 悪い例                                                             |
| ---------------- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| PM 責務          | PM は計画化、進捗確認、課題・リスク管理を担う                                                   | PO が全体をいい感じに管理する                                      |
| 判断境界         | PM が影響を整理し、PO がスコープ変更を採否判断する                                              | 必要に応じて関係者で判断する                                       |
| owner 語彙       | Schedule の `owner` は `pm-roles.yaml` の Role code に限定する                                  | 実行 agent の nickname を `owner` にする                           |
| リスクと課題     | 将来の不確実性はリスク、発生済みの阻害要因は課題として分ける                                    | 問題点をまとめて箇条書きにする                                     |
| 変更管理         | 目的、主要成果物、責任分担への影響は変更要求にする                                              | 大きな変更も通常更新として扱う                                     |
| 公開適性         | 認証情報、個人情報、非公開情報を記載しない                                                      | 公開前に消す前提で本文に入れる                                     |
| ロール定義の文脈 | project_note の業務文脈が本プロジェクトと異なる場合、語彙のみを正本として継続確認事項に分離する | project_note の業務文脈をそのまま責務記述に転記する                |
| AI Agent の範囲  | 最終承認、公開可否、説明責任は人間の `PO` に残すと明記する                                      | 整合確認や機械的更新と一緒に最終判断も AI Agent に委ねられると書く |

## 7. レビュー観点

| 観点             | 確認内容                                                                                                                         |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| 計画化可能性     | 内容がタスク化、順序付け、所要時間見積もり、進捗確認に使える粒度か                                                               |
| 依存関係         | depends_on、Role code、RACI、Schedule `owner` と矛盾していないか                                                                 |
| 管理台帳への接続 | 課題、リスク、変更要求、決定記録へ転記すべき事項が分離されているか                                                               |
| 判断責任         | PM の運用判断と PO の最終判断が混同されていないか                                                                                |
| 下流入力         | コミュニケーション計画、品質管理計画、Schedule、PJR へ渡せる入力があるか                                                         |
| 公開可否         | 個人情報、非公開情報、認証情報、秘密情報を含んでいないか                                                                         |
| ロール定義の文脈 | 参照したロール定義の project_note が本プロジェクトと異なる場合、語彙のみを正本とする扱いと継続確認事項への分離が明記されているか |

## 8. 仕上げチェック

- [[pm-plan-rulebook|プロジェクト管理計画 作成ルール]] の本文構成と禁止事項に従っている。
- [[pm-plan-sample|プロジェクト管理計画 sample]] と同程度の粒度で、表と箇条書きが完成している。
- `pm-organization.md` と `pm-roles.yaml` の採用 Role code と矛盾していない。
- 参照したロール定義の project_note が本プロジェクトの業務文脈と一致しない場合、語彙のみを正本とする扱いと継続確認事項への分離が記述されている。
- PM が管理する事項と、PO が最終判断する事項が分かれている。
- 課題、リスク、変更要求、決定記録へ転記すべき事項が分離されている。
- Schedule `owner`、RACI、実行主体を混同していない。
- 公開不適切情報、認証情報、秘密情報を含んでいない。
