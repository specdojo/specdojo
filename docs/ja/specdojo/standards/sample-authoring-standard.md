# Sample 記述標準

Sample Authoring Standard

`docs/ja/specdojo/samples/` 配下の各 `*-sample.(md|yaml|json)` が従うべき構成・記述ルール・禁止事項・運用ルールを定義します。Frontmatter（メタ情報）の規約は [document-metadata-standard.md](document-metadata-standard.md) を正本とし、本書では扱いません。

## 1. 適用範囲

- 対象: `docs/ja/specdojo/samples/` 配下のすべての `*-sample.(md|yaml|json)`
- 目的: sample の構成・記述品質を統一し、粒度・文体・表の書き方を確認できる完成最小例を提供する
- 一次根拠: 対応する `docs/ja/specdojo/rulebooks/<prefix>-rulebook.md`
- Frontmatter 規約の正本: [document-metadata-standard.md](document-metadata-standard.md)
- ファイル名・ID 規則: [docs-structure-guide.md](../guides/docs-structure-guide.md)

## 2. サンプル群の目的

sample は個別ドキュメントの書式例であると同時に、同じプロジェクトを題材として要求・設計・実装・テスト・運用の成果物がどうつながるかを示す一連の完成例です。

- rulebook の必須要素を満たす最小例を提示し、抽象的な規約を具体的な記述へ変換する。
- 複数の sample で背景、用語、登場人物、制約、数値目標を共有し、成果物間のトレーサビリティを読み取れるようにする。
- 小規模で理解しやすい題材を使い、ドキュメント種別ごとの役割と粒度の違いを学べるようにする。
- サンプルのために過度な機能や登場人物を追加せず、対象ドキュメントの説明に必要な範囲へ記述を限定する。

sample は実在プロジェクトの記録ではありません。記載値は学習・例示のための仮定を含みますが、同じ仮定を複数文書で使用する場合は値と意味を統一します。

## 3. 共通サンプル文脈

### 3.1. 物語上の背景

主人公は CoderDojo に通う小学6年生の算野創太です。祖母の算野きぬが営む「駄菓子屋きぬや」で、加齢に伴う物忘れにより販売記録や在庫確認などの問題が増えてきたため、創太は祖母と店の運営を支援するシステムを作ることを決意します。

この共通シナリオは、駄菓子屋の問題解決と、創太が周囲の助言を受けながらシステム開発を学ぶ過程を扱います。業務成果物の sample では物語表現を前面に出さず、必要な背景・関係者・判断根拠だけを業務文書として記述します。

### 3.2. プロジェクト概要

プロジェクト名は「駄菓子屋きぬや 販売管理システム構築」とします。地域の子どもたちに長年親しまれている駄菓子屋きぬやでは、販売記録、在庫管理、常連客のつけ管理をノートと店主の記憶に頼っています。

主な問題は次のとおりです。

- 人気商品の欠品が増えている。
- 仕入れのタイミングと数量が勘に依存し、在庫の過不足が生じている。
- 常連客のつけ管理が曖昧で、記録漏れや残高確認の手間が発生している。
- 店主以外が店番をすると、販売、在庫、つけの状況を同じ手順で確認できない。

### 3.3. 目的と必要性

- 駄菓子の販売、在庫、顧客情報、つけ残高をシンプルに一元管理する。
- 店主一人でも無理なく使える画面と操作フローを提供する。
- 家族が店番する場合も同じ手順で販売と在庫確認を行えるようにする。
- 店主の経験と記憶だけに依存せず、店舗運営を継続できる状態を作る。
- 初期リリースでは日々の記録を安定させ、後続の仕入れ改善や分析の基礎データを蓄積する。

### 3.4. 期待効果

| 観点       | 初期目標                        | 測定の考え方                             |
| ---------- | ------------------------------- | ---------------------------------------- |
| 欠品削減   | 人気商品の欠品回数を月間50%削減 | 対象商品を定め、欠品発生を日次で記録する |
| 在庫適正化 | 在庫の無駄を年間20%削減         | 廃棄、値引き、長期滞留を月次で集計する   |
| つけ管理   | つけ残高を1分以内で確認可能     | 顧客検索から残高確認までの時間を計測する |
| 継続運用   | 営業日の90%以上で記録を継続     | 販売、在庫、つけの記録有無を日次確認する |

数値はサンプル群で使用する初期仮説です。成果物の目的に応じて測定期間や判定条件を具体化しますが、別の値へ変更する場合は関連 sample も同時に整合させます。

### 3.5. 前提条件

- 当面は単一店舗で運用し、複数店舗展開は対象外とする。
- 店頭で使用する端末は1台のタブレットを前提とする。
- 店舗の運用体制を大きく変えず、現行の販売手順に近い操作を優先する。
- 初期リリースは店頭販売、在庫、つけ管理を対象とし、EC、本格会計、仕入れ先とのシステム連携は対象外とする。
- 店主が最終的な業務判断を行い、システムは記録と確認を支援する。
- リリース日程、費用上限、オフライン時の運用など未確定の事項は、対象成果物で _UNDECIDED_: または _ASSUMPTION_: として明示する。

- 本文、表の表示名、ステークホルダー名には「店主代表」「開発担当」「利用者代表」などの役割名を使用する。
- Member や実行主体の安定識別子が必要な項目には、表に定義した役割ベースの `nickname` を使用する。
- Schedule の `owner`、RACI、承認責任など責務を表す項目には、`PO` / `BA` / `ARC` / `DEV` / `QE` などの Role code を使用する。
- 人物名を nickname に変形した値、イニシャル、愛称は使用しない。nickname は人物が交代しても意味が変わらない役割ベースの名前とする。
- 役割名、nickname、Role code のどれを使うかは、[people-and-organization-definition-standard.md](people-and-organization-definition-standard.md) の Role / Member / Task owner の区分に従う。

## 4. 出力フォーマットと命名

- ファイル名は `<prefix>-sample.md` / `<prefix>-sample.yaml` / `<prefix>-sample.json` とし、対応 rulebook の `target_format` に合わせる。
- `target_format` が未記載の場合は markdown を対象とみなす。
- Markdown の場合、H1 はファイル内で 1 つだけとし、`id` / `type` / `status` / `rulebook` を含む Frontmatter を先頭に置く。
- YAML / JSON の場合、対応 rulebook が定める先頭メタ項目・ルートキー・必須キーを満たす。

## 5. 構成の原則

- 章構成・必須表は対応 rulebook の `本文構成（標準テンプレ）` に従い、sample 独自の章立てを作らない。
- Markdown の見出しは `##` から開始し、章番号は 1 始まりの連番、末尾に `.` を付ける。
- 必須章をすべて満たしつつ、最小の記述量で完成例として成立させる。
- 任意章は、完成例として理解を助ける場合に限り含める。

## 6. 記述ガイド

- sample は対応 rulebook を一次根拠とし、必須要件を満たす最小記述例に再構成する。
- rulebook の本文を丸写しせず、完成した記述例として書き下す。
- 用語は参照元の rulebook と整合させ、命名ゆれを持ち込まない。
- 業務文脈は `共通サンプル文脈` に統一する。前提が不足する場合は同文脈で合理的に仮定し、複数文書へ影響する仮定は本書へ反映する。
- 対象人物は人物名で記載せず、`共通登場人物` に定義した役割名、役割ベースの `nickname`、または Role code で記述する。
- 曖昧語を避け、粒度・文体・表の書き方が手本になる判定可能な記述にする。
- リンクはファイルがある場合に記載し、ない場合はバッククォートで仮置きする。

## 7. 内容充実化（薄いドキュメント防止）

- 各必須章に、対応 rulebook が要求する具体項目をすべて埋める。
- 「適切に」「十分に」などの抽象語だけで終わらせず、判断可能な記述にする。
- 表が要求される章では、カラムを埋めた実例を 1 行以上示す。
- 数値・指標を扱う章では、測定方法や判定の前提が読み取れるようにする。
- ただし、実装依存の詳細（SQL 全文、具体クラス名、詳細 API 設計）には踏み込まない。

## 8. 禁止事項

- 章番号なし見出し（例: `## 基本情報`）を使用しない（Markdown の場合）。
- 章番号末尾の `.` を省略しない。
- 共通の業務文脈以外の業種題材を混在させない。
- sample の本文、表、ID、nickname に共通登場人物の人物名、人物名由来の識別子、愛称を使用しない。
- rulebook の本文をそのまま複製しない。
- デッドリンクを記載しない。
- sample 本文に実装詳細（SQL 全文、具体クラス名、詳細 API 設計）を書かない。
- 対応 rulebook の禁止事項に抵触する記述を含めない。

## 9. 運用ルール

- 対応 rulebook の本文構成が変わった場合は、sample の章構成と必須表を追従させる。
- `共通サンプル文脈` が更新された場合は、関連する既存 sample、`.github/instructions/sample.instructions.md`、`upsert-sample` Skill との整合を確認する。
- 表は必要に応じて整形スクリプト（`Format Markdown Table` タスク）で揃える。
